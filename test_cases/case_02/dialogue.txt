사용자: Express로 이미지 업로드 받는 API 만들어야 하는데 어디서부터 시작해야 해? req.body로 받으면 되나?

AI: 안타깝게도 req.body로는 파일이 안 잡혀요. 이미지나 파일은 보통 multipart/form-data 형식으로 오는데, Express 기본 파서(express.json, express.urlencoded)는 이걸 처리 못 합니다. 그래서 보통 multer라는 미들웨어를 추가로 씁니다. busboy나 formidable도 있지만 Express 생태계에선 multer가 사실상 표준이에요.

사용자: 음 근데 왜 일반 미들웨어로는 안 되는 거야? multer가 내부에서 정확히 뭘 해주는 건데?

AI: multipart/form-data는 텍스트랑 바이너리가 boundary 문자열로 구분돼서 한 요청에 섞여 들어오는 형식이에요. JSON처럼 단순히 문자열을 파싱하는 게 아니라, 스트림을 읽으면서 각 파트의 헤더를 읽고 본문을 잘라내는 작업이 필요합니다. multer는 이걸 대신 해주면서 텍스트 필드는 req.body에, 파일은 req.file(또는 req.files)에 꽂아주는 역할을 해요. 즉 파싱 + Express 친화적 형태로 정리, 이 두 가지가 핵심입니다.

사용자: 오케이 그럼 실제로 어떻게 쓰는 건지 코드로 좀 보여줘

AI: 가장 기본적인 형태는 이렇습니다.
```js
const express = require('express');
const multer = require('multer');

const upload = multer({ dest: 'uploads/' });
const app = express();

app.post('/upload', upload.single('image'), (req, res) => {
  console.log(req.file);   // 업로드된 파일 정보
  console.log(req.body);   // 같이 온 텍스트 필드
  res.json({ ok: true });
});
```
upload.single('image')에서 'image'는 클라이언트가 보낸 form 필드 이름이에요. 여러 개면 .array('images', 5)나 .fields([...])를 씁니다.

사용자: 잠깐, 그럼 파일은 어디에 저장되는 거야? 메모리야 디스크야?

AI: 위 예시처럼 dest 옵션만 주면 디스크의 uploads/ 폴더에 임시 파일명으로 저장됩니다. 근데 사실 저장 방식은 multer 본체가 아니라 "스토리지"라는 별도 어댑터가 결정해요. 기본 제공되는 게 diskStorage(디스크에 저장)랑 memoryStorage(버퍼로 메모리에 보관) 두 가지고, dest는 내부적으로 diskStorage를 쓰는 단축 문법입니다.

사용자: 스토리지 어댑터라는 게 정확히 어떻게 동작하는 건데? 왜 굳이 분리돼 있어?

AI: 스토리지는 _handleFile(파일을 받아서 어딘가에 쓰기)과 _removeFile(에러 시 정리) 두 메서드를 가진 객체예요. multer는 멀티파트를 파싱하다가 파일 파트를 만나면 그 스트림을 스토리지에 넘기고, 스토리지가 "이걸 디스크에 쓸지, 메모리에 담을지, S3에 올릴지"를 결정합니다. 이렇게 분리해둔 덕분에 multer-s3, multer-gridfs-storage 같은 서드파티 스토리지로 갈아끼우기만 하면 저장 위치를 통째로 바꿀 수 있어요. 파싱과 저장의 책임을 분리한 구조죠.

사용자: 그렇구나. 디스크 스토리지로 파일명까지 직접 정하는 코드 좀 보여줄래?

AI: diskStorage를 직접 만들면 폴더랑 파일명을 함수로 제어할 수 있습니다.
```js
const path = require('path');
const storage = multer.diskStorage({
  destination: (req, file, cb) => {
    cb(null, 'uploads/');
  },
  filename: (req, file, cb) => {
    const unique = Date.now() + '-' + Math.round(Math.random() * 1e9);
    cb(null, unique + path.extname(file.originalname));
  }
});

const upload = multer({ storage });
```
원본 파일명을 그대로 쓰면 한글 깨짐이나 충돌이 생기기 쉬워서 보통 이렇게 고유 이름을 만들어 줍니다.

사용자: 근데 이미지만 받고 싶은데 아무 파일이나 다 받으면 곤란하잖아. 어떻게 제한해?

AI: multer 옵션에 fileFilter를 넘기면 됩니다. 파일의 mimetype을 보고 받을지 거를지 결정해요.
```js
const upload = multer({
  storage,
  fileFilter: (req, file, cb) => {
    if (file.mimetype.startsWith('image/')) cb(null, true);
    else cb(new Error('이미지만 업로드 가능합니다'), false);
  }
});
```
다만 mimetype은 클라이언트가 보낸 값이라 위조될 수 있어요. 엄격하게 하려면 업로드 후 파일의 매직 넘버(file-type 같은 라이브러리)로 한 번 더 검증하는 게 좋습니다.

사용자: 크기 제한은? 너무 큰 파일 올라오면 서버 죽을 것 같은데

AI: 같은 옵션 객체에 limits를 추가하면 됩니다. fileSize는 바이트 단위예요.
```js
const upload = multer({
  storage,
  fileFilter,
  limits: { fileSize: 5 * 1024 * 1024 }  // 5MB
});
```
초과하면 multer가 MulterError('LIMIT_FILE_SIZE')를 던지니까 에러 핸들러에서 잡아서 413 같은 응답을 주면 됩니다. 중요한 건 이 검사는 스트림을 읽으면서 동작해서, 전체를 다 받은 다음에 거르는 게 아니라 한도 초과 시점에 끊어버린다는 점이에요.

사용자: 정리하면 multer는 multipart 요청을 파싱해서 req.file에 꽂아주는 역할이고, 실제로 어디에 저장할지는 스토리지 어댑터가 결정하는 거네. 그래서 나중에 로컬 디스크에서 S3로 옮기고 싶으면 스토리지만 multer-s3로 갈아끼우면 라우터 코드는 안 건드려도 되는 구조구나.

AI: 정확합니다. 파싱(multer)과 저장(스토리지)이 분리돼 있다는 게 핵심이라서, 라우터에선 upload.single('image')만 그대로 두고 스토리지 인스턴스만 교체하면 됩니다. 거기에 fileFilter로 MIME을 거르고 limits로 크기를 막는 두 겹의 방어선까지 두면 기본적인 업로드 API는 안정적으로 굴러가요.
