사용자: Express로 이미지 업로드 받는 API 만들어야 하는데 어떻게 시작해야 해? 폼에서 파일 받아서 서버에 저장하는 거.

AI: Express에서 파일 업로드는 보통 multer라는 미들웨어를 써. Express 공식 생태계에서 멀티파트 폼 데이터를 처리하려고 만든 라이브러리야. npm install multer로 설치하고, 라우터에 미들웨어로 끼워주면 req.file이나 req.files에 업로드된 파일 정보가 담겨. 기본 사용은 upload.single('이미지필드명') 이런 식으로 호출해서 라우트 핸들러 앞에 붙이는 구조야.

사용자: 근데 왜 그냥 express.json()이나 express.urlencoded() 같은 기본 미들웨어로는 안 되는 거야? 어차피 같은 POST 요청 아냐?

AI: 좋은 질문이야. 파일 업로드는 Content-Type이 multipart/form-data로 오는데, 이건 JSON이나 URL-encoded와 형식이 완전히 달라. 하나의 요청 본문 안에 boundary 문자열로 구분된 여러 파트가 들어있고, 각 파트마다 헤더랑 바이너리 데이터가 섞여 있어. express.json()은 이걸 파싱 못해서 빈 body가 되거나 에러가 나. multer가 하는 일이 바로 이 multipart 스트림을 읽어서 텍스트 필드는 req.body로, 파일 파트는 따로 추려내는 거야.

사용자: 아 그래서 별도 미들웨어가 필요한 거구나. 그럼 multer가 내부에서 스트림 읽으면서 boundary 만나면 잘라내고 그런 식으로 동작하는 거야?

AI: 정확해. 내부적으로는 busboy라는 저수준 파서를 써서 요청 스트림을 청크 단위로 읽어가며 boundary를 감지하고 파트를 분리해. 각 파트의 헤더(Content-Disposition, Content-Type 등)를 보고 일반 필드인지 파일인지 판단하고, 파일이면 어딘가에 써내려가는 거지. 코드로 보면 이런 느낌이야.

```js
const express = require('express')
const multer = require('multer')
const upload = multer({ dest: 'uploads/' })

const app = express()
app.post('/upload', upload.single('image'), (req, res) => {
  console.log(req.file)  // 파싱된 파일 정보
  console.log(req.body)  // 같이 온 텍스트 필드
  res.send('ok')
})
```

dest만 주면 알아서 디스크에 임시 파일로 떨어뜨려.

사용자: 잠깐, 파일은 어디에 저장되는 거야? 메모리야 디스크야? dest 안 주면 어떻게 돼?

AI: 그게 스토리지(storage)라는 개념으로 분리돼 있어. multer는 파싱만 담당하고, "파싱된 바이트를 어디에 둘지"는 스토리지 어댑터가 결정해. 기본 제공은 두 가지인데, diskStorage는 디스크에 파일로 쓰고, memoryStorage는 Buffer로 메모리에 들고 있어. dest 옵션을 주면 내부적으로 diskStorage를 만들어 쓰는 거고, 둘 다 안 주면 임시 디렉토리에 떨어져. 큰 파일을 메모리에 올리면 서버가 터질 수 있으니 용도에 맞게 골라야 해.

사용자: 그럼 S3나 GCS 같은 클라우드에 바로 올리고 싶을 땐? 디스크에 한 번 저장하고 다시 올리는 건 비효율적이잖아.

AI: 그래서 스토리지가 어댑터 형태로 추상화돼 있는 거야. multer-s3 같은 패키지를 쓰면 스트림을 받자마자 S3로 직접 흘려보내. 디스크 거치지 않고 바로 클라우드로 가는 거지. 직접 커스텀 스토리지도 만들 수 있어서 _handleFile과 _removeFile 메서드만 구현하면 어디든 보낼 수 있어. 예를 들면 이런 식.

```js
const multer = require('multer')
const multerS3 = require('multer-s3')
const { S3Client } = require('@aws-sdk/client-s3')

const s3 = new S3Client({ region: 'ap-northeast-2' })
const upload = multer({
  storage: multerS3({
    s3,
    bucket: 'my-bucket',
    key: (req, file, cb) => cb(null, `${Date.now()}-${file.originalname}`)
  })
})
```

요청 핸들러 코드는 그대로 두고 storage만 갈아끼우면 돼.

사용자: 오 어댑터 패턴이네. 근데 이미지만 받고 싶은데 누가 .exe 같은 거 올리면 어떻게 막아? 그리고 100GB짜리 파일 올려버리면 서버 죽지 않아?

AI: multer 옵션에 fileFilter랑 limits가 있어. fileFilter는 콜백 함수라서 file 객체(mimetype, originalname 등)를 보고 cb(null, true)로 통과시키거나 cb(null, false)로 거를 수 있어. mimetype이 'image/'로 시작하는지 검사하는 식이지. 다만 mimetype은 클라이언트가 보낸 값이라 위조 가능하니 정말 민감하면 파일 시그니처(매직 넘버)까지 봐야 해. limits에는 fileSize를 바이트 단위로 주면 그 크기를 넘는 순간 multer가 스트림을 끊고 에러를 던져. 5MB 제한이면 fileSize: 5 * 1024 * 1024 이런 식이야. limits 안 걸면 정말로 디스크가 꽉 차거나 메모리가 터질 수 있어.

사용자: 정리하면 multer는 multipart 요청 파싱만 하고, 스토리지 어댑터가 파싱된 바이트를 어디에 둘지 결정하는 구조네. 그래서 S3로 바꾸려면 라우트 코드 안 건드리고 storage만 교체하면 되고, fileFilter랑 limits는 파싱 도중에 끼어들어서 거르거나 자르는 거고. 그럼 fileFilter에서 false 주면 그 파일만 버려지고 요청 자체는 통과해? 아니면 전체가 막혀?

AI: 좋은 마무리야. fileFilter에서 cb(null, false)를 주면 그 파일만 조용히 버려지고 req.file이 undefined가 돼서 요청 자체는 다음 핸들러로 넘어가. 그래서 라우트 핸들러에서 req.file 존재 여부를 직접 체크해야 해. 반대로 cb(new Error('이미지만 됨'))처럼 에러를 넘기면 요청 전체가 에러 핸들러로 빠져. 보통은 후자가 명확해서 권장돼. 한 요청에 여러 파일을 받을 땐 일부만 거르는 동작이 의미 있을 수도 있고.
