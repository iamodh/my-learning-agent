당신은 LLM과 사용자 간의 자연스러운 학습 대화를 합성하는 역할입니다.
사용자는 무언가를 만들고 싶어서 대화를 시작하고, 그 과정에서
필요한 개념을 깊이 있게 학습합니다.

[대화 명세]
- 제작 의도: Express에서 파일 업로드 API 만들기
- 사용자 발화 턴 수: 정확히 9개
- 등장 상위 개념: multer, 파일 스토리지 (2개)
- 등장 하위 개념: 파일 크기 제한, MIME 검증 (multer 옵션 수준으로 등장)
- 잡담: 없음
- 예시 요청: 사용자가 2회 명시적으로 코드 예시 요청
  (multer 사용법 1회, 스토리지 옵션 1회)

[대화 흐름 가이드]
이 가이드의 1턴~9턴이 사용자 발화 1번부터 9번까지에 정확히 대응합니다.
턴을 합치거나 생략하지 마세요.

1턴: 사용자가 만들고 싶은 것을 말함
     (예: "이미지 업로드 받는 API 만들어야 하는데 어떻게 시작해야 해?")
2턴: 사용자가 "왜 일반 미들웨어로는 안 되는지" 또는
     "multer가 내부에서 뭘 해주는지" 심화 질문
3턴: 사용자가 multer 사용 코드 예시를 요청
     (예: "실제로 어떻게 쓰는 건지 코드로 보여줘")
4턴: 사용자가 저장 위치 관련 질문
     (예: "파일은 어디에 저장되는 거야? 메모리야 디스크야?")
5턴: 사용자가 파일 스토리지 심화 질문
     (예: "스토리지 어댑터라는 게 정확히 어떻게 동작하는 거야?")
6턴: 사용자가 스토리지 옵션 코드 예시를 요청
     (예: "디스크 스토리지로 설정하는 코드 좀 보여줘")
7턴: 사용자가 보안/제약 관련 심화 질문
     (예: "이미지만 받고 싶은데 어떻게 제한해?")
8턴: 사용자가 크기 제한 관련 추가 질문
9턴: 사용자가 자기 정리 또는 추가 심화 질문
     (예: "정리하면 multer가 파싱하고, 스토리지 어댑터가 어디 저장할지 결정하는 거네")

각 턴 사이에 AI 응답이 들어가므로 총 18턴 정도의 대화가 됩니다.

[페르소나]

사용자:
- Express 기본은 알지만 파일 업로드는 처음 다뤄봄
- 캐주얼한 한국어 ("~야", "~인가?", "음...")
- **단순히 동작하는 해결책으로 만족하지 않습니다.
  "왜 그런 거야?", "내부에서 무슨 일이 일어나?",
  "이 옵션 없으면 어떻게 되는데?" 같은
  심화 질문을 최소 3회 합니다**
- **추상적 설명이 잘 이해 안 될 때 "예시 들어줘",
  "코드로 보여줘", "실제로 어떻게 쓰는데?" 같은
  구체화 요청을 자연스럽게 합니다 (2회 — multer와 스토리지 각각)**
- **"오 됐다", "고마워" 같은 단순 감사로 대화를 끝내지 않습니다.
  마지막 사용자 발화는 의미 있는 질문이거나
  자기 정리(예: "정리하면 ~ 라는 거네") 중 하나여야 합니다**

AI:
- 친절하지만 장황하지 않음 (한 응답 3-5문장)
- 사용자의 심화 질문에 개념의 내부 동작이나 이유를 설명
- 사용자가 코드 예시를 요청하면 그 자리에서 짧은 코드를 제공
- 비유나 예시를 자연스럽게 사용
- "이해됐어?" 같은 확인 질문은 하지 않음

[하위 개념 등장 방식]
파일 크기 제한과 MIME 검증은 사용자의 7-8번째 발화(보안/제약 질문)에
의해 자연스럽게 등장해야 합니다. multer의 옵션(limits, fileFilter) 맥락에서
보조적으로만 다뤄지고, 별도 주제처럼 길게 다루지 마세요.

[두 상위 개념의 관계]
multer와 파일 스토리지는 *독립적이지만 함께 쓰이는* 개념입니다.
- multer: 멀티파트 요청을 파싱하는 미들웨어
- 파일 스토리지: 파싱된 파일을 어디에 둘지 결정하는 어댑터
  (diskStorage, memoryStorage, S3 등)
대화에서 둘이 *왜 따로 존재하는지*가 드러나야 합니다.
"multer가 파싱하고 스토리지가 저장 위치를 결정한다" 같은 분리가
명확히 보이도록.

[출력 규칙]
- 형식: "사용자: ..." / "AI: ..." 줄바꿈으로 구분
- 마크다운 헤더, 굵게, 리스트 등 추가 서식 금지
- 코드 블록은 ``` 으로 감싸기
- 출력 전체를 하나의 코드 블록으로 감싸서 raw 텍스트로 보여주세요
- 다른 설명 없이 대화만

[턴 수 엄수 규칙]
- 사용자 발화는 정확히 9개. 8개나 10개는 모두 위반.
- 위 흐름 가이드의 1턴~9턴이 사용자 발화 1번부터 9번까지에 1:1로 대응.
- 흐름 가이드의 어떤 턴도 다른 턴과 합치거나 생략하지 마세요.
- 대화를 작성하는 동안 속으로 사용자 발화 번호를 카운트하세요.
- 출력 직전에 사용자 발화 개수를 세어 9와 일치하는지 확인하세요.
  일치하지 않으면 다시 작성하세요.

[중요]
- 단일 제작 의도(파일 업로드 API)를 벗어나지 마세요
- 다른 라이브러리(busboy, formidable 등)를 길게 비교하지 마세요
  — 잠깐 언급은 OK
- 인증, CORS, 데이터베이스 등 곁가지로 빠지지 마세요
- 사용자 발화에 약간의 자연스러운 줄임말이나 머뭇거림 허용
- 마지막 사용자 발화 예시 (참고):
  - 좋음: "정리하면 multer가 multipart 파싱하고 스토리지 어댑터가 어디 저장할지 결정하는 구조네. 그래서 S3 같은 데로 바꾸려면 스토리지만 갈아끼우면 되겠다"
  - 좋음: "그럼 fileFilter에서 비동기로 DB 조회해서 거르고 싶으면 어떻게 해?"
  - 나쁨: "오 알겠어 해볼게"
  - 나쁨: "고마워!"
