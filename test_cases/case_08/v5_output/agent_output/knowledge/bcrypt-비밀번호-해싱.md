# bcrypt 비밀번호 해싱

bcrypt는 비밀번호를 단방향 해시로 변환해 저장하기 위해 설계된 알고리즘이다. 평문 비밀번호를 DB에 그대로 두면 유출 시 모든 사용자의 비밀번호가 노출되고, 사람들이 여러 사이트에 같은 비밀번호를 쓰기 때문에 피해가 연쇄된다. 그래서 가입 시점에 비밀번호를 해시로 바꿔 저장하고, 로그인 시점에는 입력값을 같은 방식으로 해싱해 비교한다.

## 일반 해시 함수와의 차이

- **자동 salt**: bcrypt는 매번 임의의 salt를 만들어 비밀번호에 붙인 뒤 해싱한다. 같은 비밀번호라도 매번 다른 해시값이 나오므로 미리 계산해둔 해시 테이블(레인보우 테이블) 공격이 무력화된다. salt는 결과 문자열에 함께 인코딩되므로 따로 저장할 필요가 없다.
- **의도적으로 느림**: work factor를 통해 해시 계산을 수천~수만 번 반복시켜, 무차별 대입 공격에 걸리는 시간을 크게 늘린다.

SHA-256 같은 일반 해시 함수는 빠르게 동작하도록 설계돼 있어 비밀번호 저장 용도로는 부적합하다.

## 사용 방법

가입 시 `hash`로 저장하고, 로그인 시 `compare`로 검증한다.

```js
const bcrypt = require('bcrypt');
const saltRounds = 12;

// 가입: 해시로 변환해 저장
const hashed = await bcrypt.hash(password, saltRounds);

// 로그인: 입력값과 저장된 해시 비교
const valid = await bcrypt.compare(password, user.password);
```

`compare`는 내부적으로 저장된 해시에서 salt를 꺼내 입력값에 다시 적용해 비교하므로, salt를 별도로 다룰 필요가 없다.

## saltRounds (work factor) 선택

saltRounds는 work factor에 해당한다. 1 올릴 때마다 해시 계산 시간이 2배가 된다. 너무 낮으면 무차별 공격에 약하고, 너무 높으면 로그인 응답이 느려져 사용자 경험이 나빠진다.

- 현실적인 권장값은 10~12 사이.
- 일반적인 서버에서 10은 약 100ms, 12는 약 250~400ms.
- 보안을 중시하면 12, 트래픽이 많고 응답 속도가 중요하면 10에서 시작.
- 자기 서버에서 직접 시간을 재보고 200~300ms 안에 들어오는 값을 고르는 게 좋다.
