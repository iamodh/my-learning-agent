당신은 LLM과 사용자 간의 자연스러운 학습 대화를 합성하는 역할입니다.
사용자는 큰 목표를 가지고 대화를 시작한 뒤, 그 목표를 이루기 위해
필요한 기술을 *단계별로* 학습하며 빌드업하는 흐름을 따라갑니다.

[대화 명세]
- 제작 의도: Express 회원가입 시스템 구현
- 사용자 발화 턴 수: 정확히 12개
- 등장 상위 개념: bcrypt, JWT (2개)
- 등장 하위 개념: salt rounds (bcrypt 섹션), 만료 처리 (JWT 섹션)
- 잡담: 없음
- 예시 요청: 2회 (bcrypt 사용법 1회, JWT 발급 1회)
- 인과 흐름: 회원가입 → 비밀번호 안전 저장 필요 → bcrypt → 로그인 유지 필요 → JWT

[대화 흐름 가이드]
아래는 *사용자 발화 12개*의 흐름입니다. 각 항목은 하나의 사용자 발화에
1:1로 대응합니다. 각 사용자 발화 직후에 AI 응답이 따라오며, AI 응답은
별도 턴으로 세지 않습니다.

사용자 발화 1: 큰 목표 제시
     (예: "Express 서버에 회원가입 기능 만들고 싶은데, 뭐부터 시작해야 해?")
     → AI는 전체 흐름 간단히 요약 (가입 → 비밀번호 저장 → 로그인 → 세션 유지)

사용자 발화 2: 첫 번째 단계 — 비밀번호 저장 고민
     (예: "이메일이랑 비밀번호 받아서 DB에 저장하면 되는 건가? 근데 비밀번호 그냥 저장하면 안 된다는 건 알아")
     → AI는 평문 저장의 위험성과 해싱의 필요성 설명, bcrypt 소개

사용자 발화 3: bcrypt 본질 심화 질문
     (예: "bcrypt가 정확히 어떻게 동작하는 거야? 그냥 해시 함수랑 뭐가 달라?")
     → AI는 salt 추가와 의도적 느림(work factor) 설명

사용자 발화 4: bcrypt 사용 코드 예시 요청
     (예: "실제로 어떻게 쓰는지 코드로 보여줄래?")
     → AI는 bcrypt.hash와 bcrypt.compare 사용 예시 제공

사용자 발화 5: salt rounds 관련 심화 질문
     (예: "근데 그 saltRounds 숫자는 어떻게 정해? 10이 좋아 12가 좋아?")
     → AI는 rounds가 보안과 성능의 트레이드오프임을 설명 (현실적 권장값 포함)

사용자 발화 6: 두 번째 단계 — 로그인 후 상태 유지 고민
     (예: "그럼 이제 비밀번호 저장은 됐고, 로그인 성공하면 그 상태를 어떻게 유지해? 매번 다시 비밀번호 입력하게 할 순 없잖아")
     → AI는 세션 vs 토큰 방식 짧게 비교 후 JWT 도입 제안

사용자 발화 7: JWT 본질 심화 질문
     (예: "JWT는 그럼 어떻게 동작하는 거야? 서버에 저장 안 하고도 인증이 된다는 게 신기한데")
     → AI는 서명된 토큰 구조와 stateless 인증 원리 설명

사용자 발화 8: JWT 발급 코드 예시 요청
     (예: "JWT 만드는 코드 좀 보여줘. jsonwebtoken 라이브러리 쓰면 되는 거지?")
     → AI는 jwt.sign과 jwt.verify 사용 예시 제공

사용자 발화 9: 만료 처리 관련 심화 질문
     (예: "토큰 만료는 어떻게 정하는 게 좋아? 너무 길면 보안 위험 같고, 너무 짧으면 사용자 불편할 거 같은데")
     → AI는 만료 시간 권장 패턴 (액세스 짧게, 리프레시 길게) 설명

사용자 발화 10: 두 단계 연결 질문
     (예: "그럼 로그인할 때 흐름이 어떻게 돼? bcrypt로 비교하고 통과하면 JWT 발급하는 거지?")
     → AI는 전체 로그인 플로우 요약 (이메일 조회 → bcrypt.compare → jwt.sign → 클라이언트 전달)

사용자 발화 11: 보안 추가 질문
     (예: "JWT는 클라이언트 어디에 저장해? 쿠키야 로컬스토리지야?")
     → AI는 저장소별 보안 트레이드오프 설명 (XSS, CSRF 관점)

사용자 발화 12: 자기 정리 — 전체 빌드업 정리
      (예: "정리하면 회원가입은 bcrypt로 비밀번호 안전하게 저장하고, 로그인 후엔 JWT로 상태 유지하는 구조네. 인증 흐름이 이제 그림 그려진다")

각 사용자 발화 사이에 AI 응답이 들어가므로 총 24턴 정도의 대화가 됩니다.

[페르소나]

사용자:
- Express 기본은 알지만 인증 구현은 처음
- 캐주얼한 한국어 ("~야", "~인가?", "음...")
- **이 케이스의 특징은 *큰 목표에서 출발해 단계별로 필요한 기술을 발견하며 학습*하는 빌드업입니다**
- 각 단계는 *이전 단계의 자연스러운 결과*로 등장
  (예: "비밀번호 저장 했으니 → 이제 로그인 후 상태 유지를?")
- **단순히 동작하는 해결책으로 만족하지 않습니다.
  심화 질문을 최소 4회 합니다 (각 단계마다 본질을 파헤침)**
- **추상적 설명이 잘 이해 안 될 때 "코드로 보여줘" 같은
  구체화 요청을 합니다 (2회 — bcrypt와 JWT 각각)**
- **"오 됐다", "고마워" 같은 단순 감사로 대화를 끝내지 않습니다.
  마지막 사용자 발화는 *전체 빌드업을 정리*하는 형태입니다**

AI:
- 친절하지만 장황하지 않음 (한 응답 3-5문장)
- 각 단계의 *왜 이게 필요한지*를 명확히 설명 (인과 연결)
- 사용자가 다음 단계로 진행할 신호를 주면 자연스럽게 이어감
- 사용자가 코드 예시를 요청하면 그 자리에서 짧은 코드를 제공
- "이해됐어?" 같은 확인 질문은 하지 않음

[인과 흐름의 자연스러움]
이 케이스의 핵심은 *각 단계가 이전 단계에서 자연스럽게 파생*되어야 한다는 점입니다:

- 발화 1: 회원가입 시스템 → 비밀번호 저장 문제 등장
- 발화 2-5: 비밀번호 저장 → bcrypt 학습
- 발화 6: 비밀번호 저장 완료 → 로그인 상태 유지 문제 등장
- 발화 7-9: 로그인 유지 → JWT 학습
- 발화 10-11: 두 기술의 연결 및 운영 고려
- 발화 12: 전체 빌드업 자기 정리

각 *전환점* (발화 6 같은)에 *명확한 인과*가 있어야 합니다.
"이제 다른 거 물어볼게" 같은 임의 전환이 아니라,
*"이걸 해결했으니 이제 다음 문제로"* 같은 흐름.

[두 개념의 관계]
bcrypt와 JWT는 *순차적으로* 등장하는 *독립 기술*입니다:
- bcrypt: 비밀번호 *저장 시* 사용 (가입 시점)
- JWT: 로그인 *이후 상태 유지*에 사용 (로그인 성공 후)
대화에서 *각 기술이 어느 시점에 왜 필요한지*가 명확히 드러나야 하며,
발화 10에서 둘이 *어떻게 연결*되는지가 한 번 정리되어야 합니다.

[하위 개념 등장 방식]
*salt rounds*는 bcrypt 학습 중(발화 5)에 자연스럽게 등장합니다.
*만료 처리*는 JWT 학습 중(발화 9)에 자연스럽게 등장합니다.
별도 주제처럼 길게 다루지 말고, *각 상위 기술의 운용 디테일*로 다뤄지세요.

[출력 규칙]
- 형식: "사용자: ..." / "AI: ..." 줄바꿈으로 구분
- 마크다운 헤더, 굵게, 리스트 등 추가 서식 금지
- 코드 블록은 ``` 으로 감싸기
- 출력 전체를 하나의 코드 블록으로 감싸서 raw 텍스트로 보여주세요
- 다른 설명 없이 대화만

[턴 수 엄수 규칙]
- *사용자 발화*는 정확히 12개. 11개나 13개는 모두 위반.
- AI 응답은 각 사용자 발화 사이에 들어가지만 카운트에 포함되지 않습니다.
- 위 흐름 가이드의 *사용자 발화 1~12*가 실제 사용자 줄 1번부터 12번까지에
  1:1로 대응합니다.
- 흐름 가이드의 어떤 사용자 발화도 다른 것과 합치거나 생략하지 마세요.
- 인과 흐름(발화 1 → 2-5 bcrypt → 6 → 7-9 JWT → 10-12 통합)이 명확히
  드러나야 합니다.
- 대화를 작성하는 동안 속으로 "사용자: " 줄의 개수를 카운트하세요.
- 출력 직전에 "사용자: " 로 시작하는 줄의 개수를 세어 12와 일치하는지
  확인하세요. 일치하지 않으면 다시 작성하세요.
