# bcrypt

bcrypt는 비밀번호 저장 전용으로 설계된 단방향 해시 알고리즘이다. 평문 비밀번호를 그대로 DB에 저장하면 유출 시 사용자가 다른 사이트에서 쓰는 같은 비밀번호까지 연쇄적으로 노출되기 때문에, 비밀번호는 단방향으로 해시해 저장하고 로그인 시 입력값을 다시 해시해 비교한다.

## 일반 해시 함수와의 차이

bcrypt가 SHA-256 같은 일반 해시와 다른 두 가지 핵심:

- **자동 salt**: 매번 임의의 salt를 생성해 비밀번호에 붙인 뒤 해싱한다. 같은 비밀번호도 매번 다른 해시값이 나오므로, 미리 계산해둔 해시 테이블(레인보우 테이블) 공격이 무력화된다. salt와 해시는 결과 문자열에 함께 인코딩되므로 따로 저장할 필요가 없다.
- **의도적인 느림**: work factor 개념으로 해시 계산을 수천~수만 번 반복시킨다. 공격자가 무차별 대입을 시도해도 시간이 매우 오래 걸리도록 만든다.

## 사용 방법

핵심 함수는 두 개다. 가입 시 `hash`로 저장하고, 로그인 시 `compare`로 검증한다.

```js
const bcrypt = require('bcrypt');
const saltRounds = 12;

// 가입: 평문을 해시해 저장
const hashed = await bcrypt.hash(password, saltRounds);

// 로그인: 입력값과 저장된 해시 비교
const valid = await bcrypt.compare(password, user.password);
```

`compare`는 내부적으로 저장된 해시에서 salt를 꺼내 입력값에 다시 적용해 비교한다. salt를 별도로 다룰 필요가 없다.

## saltRounds (work factor)

`saltRounds`는 work factor다. 1 올릴 때마다 해시 계산 시간이 2배가 된다. 보안과 성능의 트레이드오프:

- 너무 낮으면 무차별 공격에 약하다.
- 너무 높으면 로그인이 느려져 사용자 경험이 나빠진다.

현실적인 권장값은 10~12 사이다. 일반적인 서버 기준 10은 약 100ms, 12는 약 250~400ms 정도 걸린다. 보안을 중시하면 12, 트래픽이 많고 응답 속도가 중요하면 10으로 시작하고, 자기 서버에서 직접 시간을 재 200~300ms 안에 들어오는 값을 고르는 것이 좋다.
