# Express 회원가입·인증 시스템

## 만들려고 한 것

Express 서버에 회원가입 기능과, 로그인 이후 사용자 상태를 유지하는 인증 구조를 구축한다.

## 전체 구조

회원가입 시스템은 네 단계로 나뉜다.

1. **가입**: 이메일/비밀번호를 받아 비밀번호를 안전하게 해싱해 저장
2. **로그인 검증**: 입력 비밀번호를 저장된 해시와 비교
3. **상태 유지**: 로그인 성공 후 토큰 발급
4. **요청 인증**: 이후 요청마다 토큰으로 사용자 인증

## 로그인 전체 플로우

```
클라이언트가 이메일/비밀번호 전송
  → 서버가 이메일로 DB에서 사용자 조회
  → bcrypt.compare로 입력 비밀번호와 저장된 해시 비교
  → 일치하면 jwt.sign으로 토큰 발급
  → 토큰을 클라이언트에 응답
  → 클라이언트는 이후 요청마다 Authorization 헤더에 토큰 첨부
  → 서버는 verify 미들웨어로 검증하고 req.user에 사용자 정보 주입
```

시점별 핵심 함수:

- 가입 시점: `bcrypt.hash`
- 로그인 시점: `bcrypt.compare` + `jwt.sign`
- 이후 요청: `jwt.verify`

## 사용된 지식과 사용 이유

### [bcrypt 비밀번호 해싱](../knowledge/bcrypt-비밀번호-해싱.md)

- **어디에 쓰였나**: 가입 단계의 비밀번호 저장, 로그인 단계의 비밀번호 검증.
- **왜 썼나**: 평문 저장은 DB 유출 시 모든 사용자의 비밀번호가 노출되고 비밀번호 재사용으로 피해가 번지기 때문이다. bcrypt는 임의 salt로 레인보우 테이블 공격을 막고, work factor로 무차별 대입을 느리게 만들어 비밀번호 저장에 적합하다. `saltRounds`로 해싱 강도를 조율한다.

### [JWT 기반 stateless 인증](../knowledge/jwt-stateless-인증.md)

- **어디에 쓰였나**: 로그인 성공 후 상태 유지, 이후 모든 요청의 인증 미들웨어.
- **왜 썼나**: 매 요청마다 비밀번호를 다시 받을 수 없으므로 로그인 상태를 유지할 수단이 필요하다. JWT는 서명 검증만으로 인증되어 서버에 토큰을 저장할 필요가 없고(stateless), SPA·모바일에서 확장하기 좋다. 만료 시간과 저장 위치(httpOnly 쿠키)로 토큰 보안을 조율한다.

## 두 기술의 책임 분리

- **bcrypt**: 가입 시점 — 비밀번호를 단방향으로 안전하게 보관
- **JWT**: 로그인 이후 — 서명된 토큰으로 stateless하게 사용자 상태를 증명

시점도 책임도 분리돼 있어, 이 골격 위에 리프레시 토큰이나 소셜 로그인을 얹어도 구조가 흔들리지 않는다.
