사용자: Express 서버에 회원가입 기능 만들고 싶은데, 뭐부터 시작해야 해? 막막하네

AI: 큰 그림부터 잡으면 좋아요. 회원가입 시스템은 보통 네 단계로 나뉩니다. 첫째, 이메일/비밀번호를 받아서 저장(이때 비밀번호를 안전하게 해싱해야 함), 둘째, 로그인 요청 시 비밀번호 검증, 셋째, 로그인 성공 후 사용자 상태를 유지(보통 토큰 발급), 넷째, 이후 요청마다 그 토큰으로 인증. 각 단계에서 쓰이는 기술이 따로 있는데, 일단 가입 단계부터 차근차근 가봅시다.

사용자: 이메일이랑 비밀번호 받아서 DB에 저장하면 되는 건가? 근데 비밀번호 그냥 저장하면 안 된다는 건 알아

AI: 맞아요, 평문 저장은 절대 금지입니다. DB가 유출되면 모든 사용자의 비밀번호가 그대로 노출되고, 사람들이 여러 사이트에 같은 비밀번호를 쓰기 때문에 피해가 연쇄적으로 번져요. 그래서 비밀번호는 단방향 해시로 변환해 저장하고, 로그인 시엔 입력값을 다시 해싱해서 비교합니다. Node에서는 보통 bcrypt 라이브러리를 써요. 일반 해시 함수(SHA-256 같은)와 달리 비밀번호 저장 전용으로 설계된 알고리즘이라 더 안전합니다.

사용자: bcrypt가 정확히 어떻게 동작하는 거야? 그냥 해시 함수랑 뭐가 달라?

AI: 두 가지 핵심 차이가 있어요. 첫째, bcrypt는 매번 임의의 salt를 생성해서 비밀번호에 붙인 뒤 해싱합니다. 그래서 같은 비밀번호도 매번 다른 해시값이 나오고, 미리 계산해둔 해시 테이블(레인보우 테이블) 공격이 무력화돼요. 둘째, bcrypt는 의도적으로 느립니다. work factor라는 개념을 통해 해시 계산을 수천~수만 번 반복시켜서, 공격자가 무차별 대입을 시도해도 시간이 엄청 오래 걸리게 만들어요. salt와 해시는 결과 문자열에 함께 인코딩되니까 따로 저장할 필요도 없습니다.

사용자: 오 그렇구나. 실제로 어떻게 쓰는지 코드로 보여줄래?

AI: 핵심은 두 함수예요. 가입 시 hash로 저장하고, 로그인 시 compare로 검증합니다.

```js
const bcrypt = require('bcrypt');
const saltRounds = 12;

// 회원가입
app.post('/signup', async (req, res) => {
  const { email, password } = req.body;
  const hashed = await bcrypt.hash(password, saltRounds);
  await User.create({ email, password: hashed });
  res.json({ ok: true });
});

// 로그인 검증
app.post('/login', async (req, res) => {
  const { email, password } = req.body;
  const user = await User.findOne({ where: { email } });
  const valid = await bcrypt.compare(password, user.password);
  if (!valid) return res.status(401).json({ error: 'invalid' });
  res.json({ ok: true });
});
```

compare는 내부적으로 저장된 해시에서 salt를 꺼내 입력값에 다시 적용해 비교하니까, salt를 별도로 다룰 필요가 없습니다.

사용자: 근데 그 saltRounds 숫자는 어떻게 정해? 10이 좋아 12가 좋아?

AI: saltRounds는 work factor예요. 1 올릴 때마다 해시 계산 시간이 2배가 됩니다. 보안과 성능의 트레이드오프인데, 너무 낮으면 무차별 공격에 약하고 너무 높으면 로그인이 느려져서 사용자 경험이 나빠지죠. 현실적인 권장값은 10~12 사이입니다. 일반적인 서버 기준 10은 약 100ms, 12는 약 250~400ms 정도 걸려요. 보안을 중시하면 12, 트래픽이 많고 응답 속도가 중요하면 10으로 시작하고, 자기 서버에서 직접 시간 재보고 200~300ms 안에 들어오는 값을 고르는 게 좋습니다.

사용자: 그럼 이제 비밀번호 저장은 됐고, 로그인 성공하면 그 상태를 어떻게 유지해? 매번 다시 비밀번호 입력하게 할 순 없잖아

AI: 좋은 흐름이에요. 크게 두 방식이 있습니다. 하나는 세션 방식 — 서버가 세션 ID를 발급하고 메모리/Redis에 사용자 정보를 저장한 뒤, 클라이언트는 쿠키로 그 ID를 들고 다닙니다. 다른 하나는 토큰 방식 — 서버가 서명된 토큰을 발급하고 클라이언트가 그걸 들고 다니면서 매 요청마다 첨부합니다. 서버는 토큰의 서명만 검증하면 되니 별도 저장소가 필요 없어요. 요즘 SPA나 모바일 앱에서 많이 쓰는 게 JWT(JSON Web Token) 방식인데, stateless하고 확장하기 좋아서 일단 이걸로 가봅시다.

사용자: JWT는 그럼 어떻게 동작하는 거야? 서버에 저장 안 하고도 인증이 된다는 게 신기한데

AI: JWT는 세 부분으로 구성된 문자열이에요: header.payload.signature. header에는 알고리즘 정보, payload에는 사용자 ID 같은 정보, signature는 서버만 아는 비밀키로 header+payload를 서명한 값입니다. 서버는 토큰을 발급할 때 서명을 만들고, 받을 때 다시 같은 비밀키로 서명을 계산해서 일치하는지만 확인합니다. 일치하면 payload를 신뢰할 수 있어요 — 누가 중간에 payload를 바꿨다면 서명이 안 맞을 테니까요. 그래서 서버에 토큰 자체를 저장할 필요가 없고, 이게 stateless 인증의 핵심입니다.

사용자: JWT 만드는 코드 좀 보여줘. jsonwebtoken 라이브러리 쓰면 되는 거지?

AI: 맞아요. sign으로 발급하고 verify로 검증합니다.

```js
const jwt = require('jsonwebtoken');
const SECRET = process.env.JWT_SECRET;

// 발급
const token = jwt.sign(
  { userId: user.id, email: user.email },
  SECRET,
  { expiresIn: '1h' }
);
res.json({ token });

// 검증 미들웨어
function authenticate(req, res, next) {
  const token = req.headers.authorization?.split(' ')[1];
  try {
    const payload = jwt.verify(token, SECRET);
    req.user = payload;
    next();
  } catch (e) {
    res.status(401).json({ error: 'invalid token' });
  }
}
```

payload에는 식별자 정도만 넣고 민감 정보(비밀번호 같은 거)는 절대 넣지 마세요. 서명은 위변조만 막을 뿐 내용 자체는 누구나 디코딩해서 볼 수 있거든요.

사용자: 토큰 만료는 어떻게 정하는 게 좋아? 너무 길면 보안 위험 같고, 너무 짧으면 사용자 불편할 거 같은데

AI: 정확한 고민이에요. 일반적인 패턴은 액세스 토큰과 리프레시 토큰을 분리하는 겁니다. 액세스 토큰은 짧게 — 15분에서 1시간 — 잡아서 탈취돼도 피해를 최소화하고, 리프레시 토큰은 길게 — 7일에서 30일 — 잡아서 사용자가 자주 로그인 안 해도 되게 합니다. 액세스가 만료되면 클라이언트가 리프레시 토큰으로 새 액세스를 받아오는 식이에요. 리프레시 토큰은 보통 DB에 저장해서 강제 로그아웃(폐기)이 가능하게 만듭니다. 간단한 앱이면 그냥 액세스 토큰 하나만 1~2시간으로 잡고 시작해도 괜찮아요.

사용자: 그럼 로그인할 때 흐름이 어떻게 돼? bcrypt로 비교하고 통과하면 JWT 발급하는 거지?

AI: 정확해요. 전체 로그인 플로우는 이렇게 됩니다. 클라이언트가 이메일/비밀번호 전송 → 서버가 이메일로 DB에서 사용자 조회 → bcrypt.compare로 입력 비밀번호와 저장된 해시 비교 → 일치하면 jwt.sign으로 토큰 발급 → 토큰을 클라이언트에 응답으로 전달 → 클라이언트는 이후 요청마다 Authorization 헤더에 토큰을 실어 보냄 → 서버는 verify 미들웨어로 검증하고 req.user에 사용자 정보를 꽂아줌. 가입 시점엔 bcrypt.hash, 로그인 시점엔 bcrypt.compare + jwt.sign, 그 이후 요청들엔 jwt.verify가 핵심이에요.

사용자: JWT는 클라이언트 어디에 저장해? 쿠키야 로컬스토리지야?

AI: 둘 다 트레이드오프가 있어요. 로컬스토리지는 다루기 쉽지만 JavaScript로 접근 가능해서 XSS 공격에 토큰이 그대로 털릴 수 있습니다. 쿠키는 httpOnly 플래그를 주면 JavaScript에서 접근 못 하니까 XSS에는 안전한데, 대신 브라우저가 자동으로 쿠키를 첨부하기 때문에 CSRF 공격에 노출돼요. 그래서 권장되는 방식은 httpOnly + Secure + SameSite=Strict(또는 Lax) 쿠키에 저장하는 거예요. 이러면 XSS도 막고 SameSite로 CSRF도 상당 부분 차단됩니다. 추가로 중요한 작업엔 CSRF 토큰을 별도로 두면 더 단단해지고요.

사용자: 정리하면 회원가입은 bcrypt로 비밀번호 안전하게 저장하고, 로그인 후엔 JWT로 상태 유지하는 구조네. 인증 흐름이 이제 그림 그려진다

AI: 잘 정리하셨어요. 가입 시점엔 bcrypt가 비밀번호를 단방향으로 안전하게 보관하는 역할, 로그인 이후엔 JWT가 서명된 토큰으로 stateless하게 사용자 상태를 증명하는 역할 — 두 기술이 시점도 책임도 분리돼 있다는 게 핵심입니다. 여기에 saltRounds로 해싱 강도를, 만료 시간과 저장 위치로 토큰 보안을 조율하면 됩니다. 이 골격 위에 나중에 리프레시 토큰이나 소셜 로그인을 얹어도 구조가 흔들리지 않아요.
