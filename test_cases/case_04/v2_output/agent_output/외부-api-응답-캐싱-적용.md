# 외부 API 응답 Redis 캐싱 적용

## 만들려는 것

Express 기반 서비스에서 외부 API를 호출하는 부분에 Redis 캐싱을 도입한다. 응답이 거의 바뀌지 않는데도 매 요청마다 외부 API를 때려서 비용이 늘고 응답이 느려지는 문제를 해결하는 것이 목적이다.

## 구성 방침

- 클라이언트: `ioredis`(또는 `node-redis`)를 Express에 붙여 로컬 Redis에 연결.
- 흐름: 요청 시 Redis를 먼저 조회 → 히트면 즉시 반환, 미스면 외부 API 호출 후 결과를 Redis에 저장하고 반환.
- 만료: `SET ... EX <초>`로 TTL을 부여해 자동 만료. TTL은 데이터 변경 주기에 맞춰 정한다.
- 키 네이밍: `리소스:id:속성` 형태의 콜론 계층(예: `product:123`)을 따른다.
- 무효화: 데이터가 바뀌는 시점에는 TTL을 기다리지 않고 `redis.del(key)`로 명시적 삭제. 덮어쓰기보다 race condition 위험이 낮아 안전하다.
- 스탬피드 대비: 트래픽이 큰 키는 캐시 미스 시 락(`SETNX`)으로 한 요청만 외부 API를 호출하도록 한다.

## 참조한 지식과 이유

| 사용한 지식 | 참조 이유 |
|---|---|
| Redis 캐싱 - Redis가 캐시로 적합한 이유 | 외부 API 응답 캐싱에 왜 Redis를 선택했는지의 근거(메모리 기반 속도, 다중 인스턴스 공유). |
| Redis 캐싱 - 기본 적용 패턴 | Express 라우트 핸들러 안에서 조회→미스 시 외부 호출→저장 흐름을 그대로 구현하는 코드 골격. |
| Redis 캐싱 - TTL과 만료 동작 | 캐시가 영원히 살아 있지 않도록 `EX`로 TTL을 부여하고, 데이터 변경 주기에 맞춰 값을 정하기 위해. |
| Redis 캐싱 - 키 네이밍 컨벤션 | 키 충돌과 가독성 문제를 막기 위해 `리소스:id:속성` 계층 규칙을 채택. |
| Redis 캐싱 - 캐시 무효화 | 데이터가 바뀔 때 TTL만 의존하지 않고 `del`로 무효화하는 방식을 선택. 덮어쓰기 대신 삭제를 고른 이유는 race condition 회피. |
| Redis 캐싱 - 캐시 스탬피드 | 인기 키 만료 시 외부 API 폭주를 막기 위한 락 기반 대응을 구성에 포함. |
