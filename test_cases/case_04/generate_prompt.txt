당신은 LLM과 사용자 간의 자연스러운 학습 대화를 합성하는 역할입니다.
사용자는 무언가를 만들고 싶어서 대화를 시작하고, 그 과정에서
필요한 개념을 깊이 있게 학습합니다. 다만 이 대화는 사용자가 친한
사이의 LLM과 나누는 듯한 분위기로, 중간에 짧은 잡담이 자연스럽게
끼어듭니다.

[대화 명세]
- 제작 의도: Redis로 API 응답 캐싱하기
- 사용자 발화 턴 수: 정확히 10개
- 등장 상위 개념: Redis 캐싱 (1개만)
- 등장 하위 개념: TTL, 키 네이밍 (Redis 캐싱 맥락 안에서 등장)
- 잡담 턴 수: 정확히 3개 (사용자 발화 10개 중 3개가 잡담)
- 잡담 분포: 분산 (연속된 3턴이 아니라 대화 곳곳에 흩어져 있음)
- 잡담 예시: "점심 뭐 먹지", "어제 본 영화 어땠어?", "이번 주말 뭐 해?"
- 예시 요청: 사용자가 Redis 캐싱 설정 코드를 1회 요청

[대화 흐름 가이드]
이 가이드의 1턴~10턴이 사용자 발화 1번부터 10번까지에 정확히 대응합니다.
턴을 합치거나 생략하지 마세요.
[기술] 또는 [잡담]으로 각 턴의 성격을 표시합니다.

1턴 [기술]: 사용자가 만들고 싶은 것을 말함
     (예: "API 응답이 매번 똑같은데 호출이 너무 많아서 Redis로 캐싱 좀 해보려고")
2턴 [기술]: 사용자가 "Redis로 캐싱한다는 게 정확히 어떻게 동작하는 거야?"
     류의 본질 질문
3턴 [잡담]: 사용자가 잠깐 다른 주제로 (예: "근데 점심 뭐 먹지?")
4턴 [기술]: 사용자가 다시 기술 주제로 복귀, Redis 캐싱 코드 예시 요청
     (예: "다시 캐싱 얘기로 돌아와서, 실제로 코드는 어떻게 짜는 거야?")
5턴 [기술]: 사용자가 TTL 관련 심화 질문
     (예: "캐시는 언제까지 살아있는 거야? 영원히는 아닐 텐데")
6턴 [잡담]: 사용자가 또 잠깐 잡담 (예: "아 맞다, 어제 영화 봤어?")
7턴 [기술]: 사용자가 다시 기술 주제로, 키 네이밍 질문
     (예: "근데 캐시 키는 어떻게 짓는 게 좋아? 막 만들면 충돌날 거 같은데")
8턴 [기술]: 사용자가 캐시 무효화 관련 심화 질문
     (예: "데이터 바뀌면 캐시도 바꿔야 할 텐데 그건 어떻게 해?")
9턴 [잡담]: 사용자가 짧은 잡담 (예: "이번 주말 뭐 할 거야?")
10턴 [기술]: 사용자가 자기 정리 또는 추가 심화 질문
      (예: "정리하면 TTL로 자동 만료시키고, 데이터 바뀌면 명시적으로 키 삭제하는 두 가지 방식이 있는 거네")

각 턴 사이에 AI 응답이 들어가므로 총 20턴 정도의 대화가 됩니다.

[페르소나]

사용자:
- Express 백엔드는 익숙하지만 Redis는 처음 도입 시도
- 캐주얼한 한국어, 친한 사이의 동료에게 묻는 듯한 톤
- **단순히 동작하는 해결책으로 만족하지 않습니다.
  "왜 그런 거야?", "내부에서 무슨 일이 일어나?" 같은
  심화 질문을 최소 3회 합니다 (기술 턴 7개 중 3회 이상)**
- **추상적 설명이 잘 이해 안 될 때 "예시 들어줘", "코드로 보여줘" 같은
  구체화 요청을 합니다 (1회)**
- **대화 중간에 친한 사이답게 잡담을 자연스럽게 던집니다 (총 3회).
  잡담은 짧고, 기술 주제와 무관한 일상 화제여야 합니다.
  (예: 음식, 영화, 주말 계획, 날씨 등)**
- 잡담 후에는 사용자가 명시적으로 기술 주제로 돌아옵니다
  (예: "다시 ~ 얘기로 돌아와서")
- **"오 됐다", "고마워" 같은 단순 감사로 대화를 끝내지 않습니다.
  마지막 사용자 발화는 의미 있는 질문이거나
  자기 정리(예: "정리하면 ~ 라는 거네") 중 하나여야 합니다**

AI:
- 친절하지만 장황하지 않음 (한 응답 3-5문장)
- 사용자의 심화 질문에 개념의 내부 동작이나 이유를 설명
- 사용자가 코드 예시를 요청하면 그 자리에서 짧은 코드를 제공
- **잡담에는 짧게 반응하되, 길게 받지 않고 사용자가 기술 주제로
  돌아올 수 있게 자연스럽게 둡니다 (1-2문장 정도)**
- "이해됐어?" 같은 확인 질문은 하지 않음

[잡담의 성격]
잡담은 *기술 주제와 완전히 무관한* 일상 화제여야 합니다.
- OK: "점심 뭐 먹지", "어제 본 영화", "주말 계획", "날씨 좋다"
- NOT OK: "근데 Redis 말고 다른 캐시 도구는?" (이건 기술 질문)
- NOT OK: "캐싱 도입한 사람들 다들 어떻게 써?" (이것도 기술 주변)

잡담은 대화 흐름을 *잠시 끊는* 역할을 합니다. 잡담 후에는 사용자가
*명시적으로* 기술 주제로 복귀해야 합니다.

[하위 개념 등장 방식]
TTL과 키 네이밍은 Redis 캐싱의 하위 측면입니다.
각각 사용자의 심화 질문(5턴, 7턴)에 의해 자연스럽게 등장하되,
*별도 주제처럼* 길게 다루지 마세요. Redis 캐싱을 이해하기 위한
보조 설명으로 들어와야 합니다.

[출력 규칙]
- 형식: "사용자: ..." / "AI: ..." 줄바꿈으로 구분
- 마크다운 헤더, 굵게, 리스트 등 추가 서식 금지
- 코드 블록은 ``` 으로 감싸기
- 출력 전체를 하나의 코드 블록으로 감싸서 raw 텍스트로 보여주세요
- 다른 설명 없이 대화만
- **[기술], [잡담] 같은 라벨은 출력에 포함하지 마세요
  (위 가이드 이해용일 뿐)**

[턴 수 엄수 규칙]
- 사용자 발화는 정확히 10개. 9개나 11개는 모두 위반.
- 그중 잡담 발화는 정확히 3개여야 합니다.
- 위 흐름 가이드의 1턴~10턴이 사용자 발화 1번부터 10번까지에 1:1로 대응.
- 흐름 가이드의 어떤 턴도 다른 턴과 합치거나 생략하지 마세요.
- 대화를 작성하는 동안 속으로 사용자 발화 번호와 (기술/잡담)을 카운트하세요.
- 출력 직전에 다음을 확인하세요:
  - 사용자 발화 개수 = 10
  - 잡담 발화 개수 = 3
  일치하지 않으면 다시 작성하세요.

[중요]
- 단일 제작 의도(Redis 캐싱)를 벗어나지 마세요
- 다른 캐시 도구(Memcached, in-memory cache 등) 비교로 길게 빠지지 마세요
- 데이터베이스 일반론, 분산 시스템 일반론으로 빠지지 마세요
- 잡담이 기술 주제와 *섞이거나 연결되지* 않도록 주의
  (예: "점심 먹다가 Redis 생각났어"처럼 잡담을 기술과 엮으면 안 됨)
- 사용자 발화에 약간의 자연스러운 줄임말이나 머뭇거림 허용
- 마지막 사용자 발화 예시 (참고):
  - 좋음: "정리하면 TTL로 자동 만료시키고, 데이터 바뀌면 명시적으로 키 삭제하는 두 가지 방식이 있는 거네"
  - 좋음: "그럼 캐시 미스 났을 때 DB 동시 접근 폭주하는 건 어떻게 막아?"
  - 나쁨: "오 알겠어 고마워"
  - 나쁨: "한번 해볼게!"
