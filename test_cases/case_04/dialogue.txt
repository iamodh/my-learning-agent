사용자: 우리 서비스에서 외부 API 호출하는 데가 있는데, 응답이 거의 안 바뀌는데도 매번 때려서 비용도 늘고 느려. Redis로 캐싱 좀 해보려고 하는데 어디서부터 시작해야 돼?

AI: 좋은 선택이야. 외부 API 응답처럼 자주 안 바뀌는 데이터는 캐싱 효과가 크거든. 기본 흐름은 이래 — 요청 들어오면 먼저 Redis에서 키로 조회해보고, 있으면 그대로 반환(캐시 히트), 없으면 외부 API 호출하고 결과를 Redis에 저장한 뒤 반환(캐시 미스). 일단 로컬에 Redis 띄우고 Express에서 ioredis나 node-redis 클라이언트 붙이는 것부터 하면 돼.

사용자: 잠깐, 근데 Redis로 캐싱한다는 게 정확히 내부에서 어떻게 동작하는 거야? 그냥 어딘가에 저장하는 거랑 뭐가 달라?

AI: Redis는 데이터를 디스크가 아니라 메모리(RAM)에 키-값 형태로 들고 있어서, 디스크 조회나 외부 API 호출보다 훨씬 빨라. 보통 응답이 마이크로초 단위라 캐시로 쓰기 딱 좋은 거지. 그리고 단일 스레드로 명령을 순차 처리해서 동시성 문제도 단순한 편이야. 그냥 메모리 변수랑 다른 점은 별도 프로세스라 여러 서버 인스턴스가 같은 캐시를 공유할 수 있다는 거야.

사용자: 아 그렇구나. 근데 점심 뭐 먹지 진짜… 벌써 11시 반이야.

AI: ㅋㅋ 그 시간대 위험하지. 배고픈 상태로 코드 짜면 버그 늘어나니까 일단 먹고 와.

사용자: 밥 시키고 다시 캐싱 얘기로 돌아와서, 실제로 코드는 어떻게 짜는 거야? Express 미들웨어 같은 걸로 감싸나?

AI: 가장 흔한 패턴은 라우트 핸들러 안에서 직접 처리하거나 미들웨어로 빼는 거야. 짧게 보여줄게.

```javascript
import Redis from 'ioredis';
const redis = new Redis();

app.get('/api/products/:id', async (req, res) => {
  const key = `product:${req.params.id}`;
  const cached = await redis.get(key);
  if (cached) return res.json(JSON.parse(cached));

  const data = await fetchFromExternalAPI(req.params.id);
  await redis.set(key, JSON.stringify(data), 'EX', 300);
  res.json(data);
});
```

`EX 300`이 TTL 300초라는 뜻이야. 5분 동안만 캐시 유지하고 그 뒤엔 자동 삭제돼.

사용자: 오 그 TTL 얘기 좀 더 해줘. 캐시가 영원히 살아있으면 안 될 것 같긴 한데, TTL은 어떻게 정하는 게 맞아? 그리고 만료되면 Redis 내부에서 정확히 무슨 일이 일어나?

AI: TTL은 데이터가 얼마나 자주 바뀌는지에 맞춰서 정해. 거의 안 바뀌는 메타데이터는 1시간, 자주 바뀌는 건 30초~5분 정도가 흔해. 내부적으론 Redis가 키마다 만료 시간을 기록해두고, 두 가지 방식으로 정리해 — 클라이언트가 그 키에 접근할 때 만료됐는지 체크해서 삭제(lazy), 그리고 백그라운드에서 주기적으로 랜덤 샘플링해서 만료된 키들을 청소(active expiration). 그래서 TTL 지난 키가 잠깐 메모리에 남아있을 순 있지만 조회하면 무조건 없는 걸로 처리돼.

사용자: 아 맞다, 어제 영화 봤어? 나 듄 2편 드디어 봤는데 진짜 길더라.

AI: 듄은 좀 각오하고 봐야 하는 영화지 ㅋㅋ 사막 풍경은 끝내주는데 러닝타임이 부담스러워.

사용자: 맞아 엉덩이 아팠어. 근데 다시 캐시 얘기로 — 키 네이밍은 어떻게 하는 게 좋아? 막 `data1`, `userdata` 이런 식으로 지으면 나중에 충돌나거나 관리 안 될 거 같은데.

AI: 좋은 감이야. 보통은 콜론(`:`)으로 계층 구분하는 컨벤션을 써. 예를 들어 `product:123`, `user:42:profile`, `search:keyword:laptop:page:2` 이런 식으로. 리소스 종류, 식별자, 변형(파라미터)을 순서대로 넣는 거지. 이게 좋은 이유는 나중에 `KEYS product:*`나 `SCAN`으로 패턴 매칭해서 일괄 처리할 수 있고, 다른 개발자가 봐도 뭔지 바로 알 수 있어서야. 그리고 환경별로 prefix 붙이는 것도 흔해 — `prod:product:123`처럼.

사용자: 그럼 데이터가 바뀌면 캐시도 같이 바꿔야 할 텐데, TTL만 믿고 기다리긴 좀 그렇잖아. 그건 어떻게 처리해? 그리고 왜 그렇게 해야 되는 거야?

AI: 두 가지 전략이 있어. 첫째는 write-through 비슷한 거 — DB 업데이트 직후에 `redis.del(key)`로 해당 키를 지우는 거야. 다음 조회 때 자연스럽게 캐시 미스 나서 새 데이터로 다시 채워져. 둘째는 업데이트하면서 캐시도 새 값으로 덮어쓰는 거. 보통 첫 번째(삭제)가 더 안전해, 왜냐면 덮어쓰는 도중에 동시에 다른 요청이 옛날 값을 또 쓸 수 있거든 — race condition이 생겨. 삭제는 "다음에 누가 읽을 때 최신으로 다시 받아와" 라는 단순한 의도라 꼬일 일이 적어.

사용자: 아 그래서 그냥 지우는 게 안전한 거구나. 근데 이번 주말 뭐 해? 나는 그냥 집에서 뒹굴 듯.

AI: 나도 비슷해 ㅋㅋ 가끔은 아무것도 안 하는 게 제일 잘 쉬는 거더라.

사용자: 정리하면 Redis는 메모리에 키-값으로 저장해서 빠른 거고, 만료는 TTL로 자동 처리하되 데이터가 바뀌는 시점엔 명시적으로 `del`로 무효화하는 두 축으로 가는 거네. 키는 `리소스:id:속성` 식으로 콜론 계층 잡아서 충돌이랑 가독성 둘 다 잡고. 그럼 만약에 인기 키 하나가 만료된 순간에 요청 수백 개가 동시에 와서 다 캐시 미스 나면 외부 API로 폭주하는 거 아냐? 그건 어떻게 막아?

AI: 정확히 짚었어 — 그게 캐시 스탬피드(cache stampede) 또는 thundering herd라고 부르는 문제야. 대표적인 대응은 세 가지야. 첫째, 락(lock)을 써서 캐시 미스 시 한 요청만 외부 API를 호출하게 하고 나머지는 그 결과를 기다리게 하는 것(예: Redis의 SETNX로 짧은 락). 둘째, TTL이 끝나기 직전에 백그라운드에서 미리 갱신하는 방식(probabilistic early expiration). 셋째, 만료된 직후 잠깐 옛날 값을 그대로 쓰면서(stale-while-revalidate) 뒤에서 새로 채우는 방식. 트래픽이 큰 키일수록 첫 번째 락 방식이 가장 확실해.
