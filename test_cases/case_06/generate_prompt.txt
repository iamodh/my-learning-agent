당신은 LLM과 사용자 간의 자연스러운 학습 대화를 합성하는 역할입니다.
이 케이스는 사용자가 코드를 직접 붙여넣고 디버깅을 요청하는 형태의
실전 대화입니다.

[대화 명세]
- 제작 의도: GraphQL 리졸버 작성 중 비동기 에러 해결
- 사용자 발화 턴 수: 정확히 8개
- 등장 상위 개념: GraphQL 리졸버 (1개)
- 등장 하위 개념: 비동기 리졸버 패턴, 에러 처리
- 잡담: 없음
- 예시 요청: 없음 (사용자가 *자신의 코드*를 보여주고, AI가 *수정 코드*로 답함)
- 코드 제공 방향: 사용자가 자기 코드와 에러 메시지를 붙여넣음

[대화 흐름 가이드]
아래는 *사용자 발화 8개*의 흐름입니다. 각 항목은 하나의 사용자 발화에
1:1로 대응합니다. 각 사용자 발화 직후에 AI 응답이 따라오며, AI 응답은
별도 턴으로 세지 않습니다.

사용자 발화 1: 코드를 붙여넣고 에러 상황 설명
     (예: "GraphQL 리졸버 짰는데 자꾸 'Cannot return null for non-nullable field' 에러 떠.
     내 코드는 이래: [resolver 코드]")
     → AI는 원인 진단하고 비동기 처리 누락 지적, 수정 코드 제공

사용자 발화 2: 수정 후 또 다른 문제 보고
     (예: "이제 그 에러는 없는데 데이터가 undefined로 와. 호출하는 곳 코드는 이래: [호출 코드]")
     → AI는 Promise 반환 패턴 또는 async/await 사용 설명, 코드 예시

사용자 발화 3: 비동기 패턴 추가 심화 질문
     (예: "근데 async 함수 안에서 await 안 쓰면 어떻게 되는 거야?")
     → AI는 await 누락 시 동작 설명, 코드로 비교

사용자 발화 4: 에러 처리 첫 질문
     (예: "근데 DB 호출 실패하면 어떻게 처리해? 그냥 throw하면 클라이언트에 에러 다 노출되는 거 아니야?")
     → AI는 에러 마스킹 또는 formatError 옵션 설명, 코드 예시

사용자 발화 5: 에러 처리 추가 질문
     (예: "그럼 특정 에러만 클라이언트에 보여주고 싶으면 어떻게 구분해?")
     → AI는 커스텀 에러 클래스 또는 코드별 분기 패턴 설명, 코드 예시

사용자 발화 6: 추가 코드 공유하며 검증 요청
     (예: "이렇게 정리해봤는데 맞아? [최종 리졸버 코드]")
     → AI는 코드 리뷰, 개선점 제안

사용자 발화 7: 운영 관점 질문
     (예: "이거 운영 환경에서 로깅은 어떻게 해야 해? 에러 추적이 필요한데")
     → AI는 로깅 패턴 또는 에러 모니터링 도구 연결 설명

사용자 발화 8: 자기 정리 또는 추가 심화 질문
     (예: "그럼 모든 리졸버를 async로 통일하고, 에러는 커스텀 클래스로 래핑한 뒤 formatError에서 마스킹하는 게 안전한 패턴이겠네")

각 사용자 발화 사이에 AI 응답이 들어가므로 총 16턴 정도의 대화가 됩니다.
**이 케이스는 코드 블록이 대화의 대부분을 차지합니다
(사용자 코드, AI 수정 코드, 최종 코드 등 총 6-8개 코드 블록).**

[페르소나]

사용자:
- GraphQL은 기본은 알지만 비동기 처리에서 막힘
- 실무에서 마주친 실제 에러를 해결하려는 *디버깅 모드*
- 캐주얼한 한국어, 코드 중심의 짧고 실용적인 질문
- **추상적 개념 질문보다는 *내 코드 어디가 문제야*류의 구체적 디버깅 질문 위주**
- **각 사용자 발화마다 자기 코드 또는 에러 메시지를 코드 블록으로 붙여넣습니다**
- 심화 질문은 *디버깅 흐름 안에서* 자연스럽게 등장
  (예: "왜 이게 안 되는 거야?", "어디서 막힌 거야?")
- **마지막 사용자 발화는 *자기 정리* 또는 *패턴 일반화 질문*이어야 합니다
  (단순 감사 X)**

AI:
- 친절하지만 장황하지 않음
- **응답에 *수정 코드*를 코드 블록으로 제공**
- 원인 진단을 짧게 텍스트로 설명한 뒤 코드로 보여줌
- 코드 외 설명은 2-3문장으로 짧게
- "이해됐어?" 같은 확인 질문은 하지 않음

[코드 블록 사용 방식]
이 케이스는 코드가 대화의 *주된 매체*입니다:
- 사용자: 자기 코드, 에러 메시지를 코드 블록으로 붙여넣음
- AI: 수정 코드 또는 패턴 코드를 제공

각 코드 블록은 짧고 (5-15줄 정도), 문맥상 *필요한 부분만* 보여줍니다.
긴 전체 파일을 통째로 붙이지 마세요.

[하위 개념 등장 방식]
*비동기 리졸버 패턴*은 사용자 발화 1-3의 흐름에서 자연스럽게 등장합니다.
*에러 처리*는 사용자 발화 4-5에서 등장합니다.
별도 주제처럼 길게 다루지 말고, *디버깅 과정에서 마주친 문제로* 다뤄지세요.

[추상적 정의가 부족한 점에 대해]
이 케이스의 의도된 특징입니다:
- 사용자는 *디버깅 모드*라서 "GraphQL 리졸버가 뭐야?" 같은 정의 질문을 하지 않습니다
- AI도 정의를 길게 설명하지 않고 *실용적 해결*에 집중합니다
- 대화에는 *추상적 개념 정의 문장*이 거의 등장하지 않습니다

이는 *본인 에이전트가 대화에 없는 정의를 합성해서 지식 문서 첫 문단에
배치할 수 있는지*를 평가하기 위한 설계입니다. 합성 단계에서는 정의 부족을
'문제'로 보지 마세요 — 의도된 입력입니다.

[출력 규칙]
- 형식: "사용자: ..." / "AI: ..." 줄바꿈으로 구분
- 마크다운 헤더, 굵게, 리스트 등 추가 서식 금지
- 코드 블록은 ``` 로 감싸기 (필수)
- 출력 전체를 하나의 코드 블록으로 감싸서 raw 텍스트로 보여주세요
- 다른 설명 없이 대화만

[턴 수 엄수 규칙]
- *사용자 발화*는 정확히 8개. 7개나 9개는 모두 위반.
- AI 응답은 각 사용자 발화 사이에 들어가지만 카운트에 포함되지 않습니다.
- 위 흐름 가이드의 *사용자 발화 1~8*이 실제 사용자 줄 1번부터 8번까지에
  1:1로 대응합니다.
- 흐름 가이드의 어떤 사용자 발화도 다른 것과 합치거나 생략하지 마세요.
- 대화를 작성하는 동안 속으로 "사용자: " 줄의 개수를 카운트하세요.
- 출력 직전에 "사용자: " 로 시작하는 줄의 개수를 세어 8과 일치하는지
  확인하세요. 일치하지 않으면 다시 작성하세요.
