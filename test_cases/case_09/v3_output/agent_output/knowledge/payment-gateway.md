# PG사(Payment Gateway)의 역할

PG사는 우리 서비스와 카드사 사이에 끼어서 결제를 중개해 주는 사업자다. 기술적으로는 카드사와 직접 연동하는 것도 가능하지만, 사실상 거의 모든 서비스가 PG사를 끼는 이유는 두 가지다.

## 1. 카드사 연동 추상화

비자, 마스터, 국내 카드사 수십 곳과의 연결을 단일 API로 묶어준다. 직접 하려면 카드사마다 따로 계약하고 연동해야 한다.

## 2. PCI DSS 부담 이전

카드 번호가 PG사 서버로만 흐르고 우리 서버에는 닿지 않게 해준다. 직접 카드 번호를 받으면 PCI DSS(카드 정보 보안 규제)를 우리가 통과해야 하는데, 그 비용과 책임이 매우 크다. PG사를 끼면 이 부담의 대부분이 PG사로 넘어간다.

## 선택지

국내에서는 토스페이먼츠, 아임포트(포트원), KG이니시스가 많이 쓰이고, 글로벌은 Stripe가 표준이다. 국내 사이드 프로젝트에서는 토스페이먼츠 문서가 가장 깔끔하다는 점에서 본 학습에서는 토스페이먼츠를 채택했다.

## 공통 흐름

PG사마다 세부는 다르지만 큰 그림은 비슷하다.

1. 클라이언트가 PG사 SDK로 결제창을 띄운다.
2. 사용자가 카드 정보를 입력한다(이 정보는 PG사로만 흐른다).
3. PG사가 결제 결과(성공/실패 + 결제 키)를 우리 서버로 알려준다.
4. 우리 서버가 그 키로 PG사에 최종 confirm을 호출해 결제를 확정한다.
