# 세션 기반 로그인 상태 유지

OAuth로 사용자를 식별한 다음에는, 매 요청마다 외부 인증 제공자에게 다시 묻지 않기 위해 서버가 로그인 상태를 직접 유지해야 한다. 이때 쓰는 메커니즘이 세션이다.

## 동작 방식

1. 콜백에서 사용자 정보를 받은 직후 서버가 세션 ID를 발급한다.
2. 세션 ID를 HttpOnly 쿠키로 브라우저에 심는다.
3. 서버는 세션 저장소(Redis나 DB)에 `{ sessionId → userId }` 매핑을 보관한다.
4. 이후 요청이 들어올 때마다 쿠키의 세션 ID로 사용자를 식별한다.

```js
req.session.userId = profile.email; // express-session 기준
```

## JWT 대신 세션을 선택한 이유

토큰(JWT) 방식도 같은 목적을 달성할 수 있으나, 세션 방식은 만료·강제 로그아웃 컨트롤이 더 직관적이라는 점에서 본 케이스에 더 적합하다고 판단해 세션으로 결정했다.

## 만료 정책

세션 만료는 너무 짧으면 잦은 로그아웃으로 UX가 나빠지고, 너무 길면 보안 위험이 커진다. 보통 두 전략을 섞어 쓴다.

- **Sliding expiration**: 요청이 올 때마다 만료 시각을 갱신한다. 활동 중인 사용자는 끊기지 않는다.
- **Absolute expiration**: 활동 여부와 무관하게 "최초 로그인 후 N시간"이라는 상한을 둔다.

일반 서비스라면 sliding 30분 + absolute 14일 정도가 무난하다. 결제 같은 민감 액션 직전에는 더 짧은 재인증을 따로 요구하기도 한다.

Redis 같은 저장소를 쓰면 키에 TTL을 걸어두면 자동으로 만료되어 구현이 깔끔하다.
