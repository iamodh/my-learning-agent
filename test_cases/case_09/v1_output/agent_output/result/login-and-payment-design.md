# 사이드 프로젝트: 로그인 + 결제 기능 설계

## 만들려고 한 것
사이드 프로젝트에 **로그인과 결제 기능을 함께** 붙이는 것. 따로 설계하지 않고 같이 본 이유는, 두 기능 모두 "사용자가 누구인지 서버가 계속 알고 있어야 한다"는 공통점이 있어 **상태 유지(세션) 메커니즘이 공통 토대**로 깔리기 때문이다.

## 전체 그림
- **로그인**: 외부 인증(OAuth)으로 사용자를 식별 → 세션 발급 → 쿠키로 들고 다님.
- **결제**: 세션 검증 → 주문 소유자·금액 대조 → PG사에 confirm.
- **연결고리**: 세션 관리가 두 흐름의 접점이며, 만료 정책을 어떻게 잡느냐가 양쪽 UX·보안에 모두 영향을 준다.

## 사용된 지식과 사용 이유

### 1. [OAuth 기반 외부 인증](../knowledge/auth/oauth.md)
- **왜 사용**: 이메일/비밀번호를 직접 관리하는 부담을 피하려고 구글/깃허브 OAuth로 갔다. 비밀번호가 우리 서버를 거치지 않아 관리 책임이 줄어든다.

### 2. [세션 관리와 만료 정책](../knowledge/auth/session-management.md)
- **왜 사용**: OAuth로 사용자를 한 번 식별한 뒤 매 요청마다 구글에 다시 묻지 않으려면 로그인 상태를 유지해야 한다. JWT 대신 세션을 택한 이유는 만료·강제 로그아웃 제어가 직관적이고, 결제 같은 민감 액션과 잘 맞기 때문이다.

### 3. [PG사 결제 연동](../knowledge/payment/pg-payment-flow.md)
- **왜 사용**: 카드사와 직접 연결하면 PCI DSS 규제 부담이 커진다. PG사를 끼면 카드 번호가 우리 서버에 닿지 않고 보안 책임이 이전돼서, 사이드 프로젝트 규모에 현실적이다. "결제창은 클라이언트, 최종 승인은 서버" 패턴으로 금액 위변조를 막는다.

### 4. [결제 요청의 인증 가드](../knowledge/payment/payment-auth-guard.md)
- **왜 사용**: 결제 confirm도 로그인 사용자가 호출하므로, 세션 만료 시 어중간하게 진행하면 이중 결제 위험이 있다. 인증 미들웨어로 401 처리하고 주문 소유자를 세션과 대조해, 로그인 흐름과 결제 흐름을 안전하게 잇는다.

## 인과 정리
OAuth(식별) → 세션(상태 유지) → 결제 가드(세션을 결제에 적용) → PG 결제(실제 금전 흐름). 세션 관리가 인증과 결제 사이의 인과적 연결 지점이다.
