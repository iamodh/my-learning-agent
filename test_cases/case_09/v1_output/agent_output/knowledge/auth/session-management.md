# 세션 관리와 만료 정책

## 핵심 요약
인증으로 사용자를 한 번 식별한 뒤, **매 요청마다 다시 외부 제공자에게 묻지 않고 로그인 상태를 유지**하는 메커니즘이 세션 관리다. 만료 정책을 어떻게 잡느냐가 UX와 보안을 동시에 좌우한다.

## 세션 유지 흐름
1. 인증 콜백에서 사용자 정보를 받자마자 서버가 세션 ID를 발급한다.
2. 세션 ID를 **HttpOnly 쿠키**로 브라우저에 심는다.
3. 서버는 세션 저장소(Redis나 DB)에 `{ sessionId → userId }`를 보관한다.
4. 이후 요청마다 쿠키의 세션 ID로 사용자를 식별한다.

```js
req.session.userId = profile.email; // express-session 기준
```

## 세션 vs JWT
토큰(JWT) 방식도 있지만, 세션 방식이 **만료·강제 로그아웃 컨트롤이 더 직관적**이다. 강제 로그아웃·만료 제어가 중요한 케이스(예: 결제 동반)에서는 세션 방식이 유리하다.

## 만료 정책 (두 전략 혼용)
- **Sliding expiration**: 요청이 올 때마다 만료 시각을 갱신 → 활동 중인 사용자는 안 끊긴다.
- **Absolute expiration**: 활동 여부와 무관하게 "최초 로그인 후 N시간"으로 상한을 둔다.

권장 기본값: **sliding 30분 + absolute 14일**. 결제 같은 민감 액션 직전에는 더 짧은 재인증을 별도로 요구하기도 한다.

Redis 같은 저장소를 쓰면 키에 TTL을 걸어두면 자동 만료되어 구현이 깔끔하다.
