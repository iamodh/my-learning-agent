# 결제 처리

결제 처리는 PG사(Payment Gateway)를 통해 클라이언트가 띄운 결제창의 결과를 서버에서 최종 승인(confirm)하는 흐름이다. 국내 사이드 프로젝트라면 토스페이먼츠가 문서가 깔끔해 추천된다.

## PG사가 하는 일

기술적으로 카드사와 직접 연결도 가능하지만 거의 안 한다. PG사는 두 가지를 해준다.

1. 비자/마스터/국내 카드사 수십 곳과의 연결을 단일 API로 추상화한다. 직접 하려면 카드사마다 따로 계약·연동해야 한다.
2. 카드 번호가 PG사 서버로만 흐르고 우리 서버에는 닿지 않게 해준다. 이로써 **PCI DSS**(카드 정보 보안 규제) 부담이 거의 다 PG사로 넘어간다. 직접 카드 번호를 받으면 우리가 그 인증을 통과해야 해서 비용·책임이 매우 커진다.

이 때문에 PG사를 끼는 것이 사실상 표준이다. (국내: 토스페이먼츠, 아임포트/포트원, KG이니시스 / 글로벌: Stripe)

## 토스페이먼츠 confirm 흐름

토스페이먼츠는 "결제창은 클라이언트, 최종 승인은 서버" 패턴이다. 클라이언트에서 SDK로 결제창을 띄우면 성공 시 `paymentKey`, `orderId`, `amount`를 들고 success URL로 돌아온다. 서버는 그것을 받아 confirm API를 한 번 더 호출해야 결제가 확정된다.

```js
app.post('/payments/confirm', async (req, res) => {
  const { paymentKey, orderId, amount } = req.body;

  // 1) 우리 DB에 저장된 주문 금액과 클라이언트가 보낸 amount 일치 확인 (위변조 방지)
  const order = await db.orders.findById(orderId);
  if (order.amount !== amount) return res.status(400).send('amount mismatch');

  // 2) 토스에 최종 승인 요청
  const secret = Buffer.from(`${process.env.TOSS_SECRET_KEY}:`).toString('base64');
  const tossRes = await fetch('https://api.tosspayments.com/v1/payments/confirm', {
    method: 'POST',
    headers: {
      Authorization: `Basic ${secret}`,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({ paymentKey, orderId, amount }),
  });

  if (!tossRes.ok) return res.status(400).json(await tossRes.json());

  const payment = await tossRes.json();
  await db.orders.markPaid(orderId, payment);
  res.json({ ok: true });
});
```

포인트는 **클라이언트가 보낸 금액을 그대로 믿지 말고 서버 DB의 주문 금액과 대조한 후 confirm**한다는 것이다. 안 그러면 결제창에서 금액이 조작될 수 있다.

## 결제 라우트 인증 미들웨어

confirm 엔드포인트는 로그인된 사용자가 호출하는 것이므로 앞단에 인증 미들웨어를 깐다. 세션이 없거나 만료됐으면 confirm을 호출하지 않고 401로 끊는다. 결제는 멱등성·금액 정합성이 중요해서 어중간한 자동 갱신으로 두 번 결제되는 것이 가장 위험하다.

```js
function requireAuth(req, res, next) {
  if (!req.session?.userId) return res.status(401).json({ error: 'login required' });
  next();
}

app.post('/payments/confirm', requireAuth, async (req, res) => {
  // 위의 confirm 로직, 그리고 order.userId === req.session.userId 도 같이 확인
});
```

주문 소유자도 세션의 userId와 대조한다. 안 그러면 남의 주문을 자기 세션으로 confirm 치는 시나리오가 열린다.
