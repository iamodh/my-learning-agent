# 세션 관리

세션은 로그인한 사용자가 매 요청마다 누구인지 서버가 식별하기 위한 상태 유지 메커니즘이다. OAuth로 사용자 정보를 받은 뒤, 매 요청마다 외부 인증 제공자에 다시 묻지 않기 위해 사용한다.

## 동작 방식

콜백에서 사용자 정보를 받자마자 서버가 세션 ID를 발급하고, 그것을 HttpOnly 쿠키로 브라우저에 심는다. 서버는 세션 저장소(Redis나 DB)에 `{ sessionId → userId }`를 들고 있다가, 이후 요청이 올 때마다 쿠키의 세션 ID로 사용자를 식별한다.

```js
req.session.userId = profile.email; // express-session 기준
```

## JWT 대신 세션을 선택한 이유 (탈락한 대안 기록)

토큰(JWT) 방식도 가능하지만, 이 케이스에서는 세션 방식이 만료·강제 로그아웃 컨트롤이 더 직관적이라 세션으로 결정했다. JWT는 채택하지 않았다.

## 만료 정책

너무 짧으면 자주 로그아웃되어 UX가 나빠지고, 너무 길면 보안 위험이 커진다. 보통 두 전략을 섞어 쓴다.

- **Sliding expiration**: 요청이 올 때마다 만료 시각을 갱신해 활동 중인 사용자는 끊기지 않게 한다.
- **Absolute expiration**: 활동 여부와 무관하게 "최초 로그인 후 N시간"으로 상한을 둔다.

일반 서비스라면 sliding 30분 + absolute 14일 정도가 무난하다. 결제 같은 민감 액션 직전에는 더 짧은 재인증을 별도로 요구하기도 한다.

Redis 같은 저장소면 키에 TTL을 걸어두면 알아서 만료되어 구현이 깔끔하다.
