당신은 LLM과 사용자 간의 자연스러운 학습 대화를 합성하는 역할입니다.
사용자는 한 대화에서 *두 가지 기능*을 만들고 싶어 하고, 그 과정에서
공통 개념과 고유 개념이 자연스럽게 얽힙니다.

[대화 명세]
- 제작 의도: 2개 (로그인 기능, 결제 기능)
- 사용자 발화 턴 수: 정확히 10개
- 등장 상위 개념: 세션 관리(공통), OAuth(로그인 고유), 결제 게이트웨이(결제 고유)
- 등장 하위 개념: 세션 만료(세션 관리 섹션), PG사 연동(결제 게이트웨이 섹션)
- 잡담: 없음
- 예시 요청: 2회 (OAuth 흐름 1회, 결제 요청 1회)
- 결정 번복: 없음

[대화 흐름 가이드]
아래는 *사용자 발화 10개*의 흐름입니다. 각 항목은 하나의 사용자 발화에
1:1로 대응합니다. 각 사용자 발화 직후에 AI 응답이 따라오며, AI 응답은
별도 턴으로 세지 않습니다.

사용자 발화 1: 두 가지 목표 제시
     (예: "사이드 프로젝트에 로그인이랑 결제 기능 둘 다 붙여야 하는데, 같이 설계하는 게 나을 거 같아서 한꺼번에 물어볼게")
     → AI는 두 기능의 공통점(상태 유지 필요)과 차이점 간단히 요약

사용자 발화 2: 로그인부터 시작 — 인증 방식 질문
     (예: "로그인은 그냥 이메일/비번보다는 구글이나 깃허브 OAuth로 가고 싶은데 어떻게 동작하는 거야?")
     → AI는 OAuth 흐름 (인가 코드 → 토큰 → 사용자 정보) 설명

사용자 발화 3: OAuth 흐름 코드 예시 요청
     (예: "Express에서 OAuth 처리하는 코드 좀 보여줄래?")
     → AI는 passport-google-oauth 또는 직접 fetch로 토큰 교환 코드 제공

사용자 발화 4: 로그인 후 상태 유지 질문 — *공통 개념 등장*
     (예: "OAuth로 사용자 정보 받았으면 그 다음엔? 로그인 상태를 어떻게 유지해?")
     → AI는 세션 관리 도입 (서버 세션 vs 토큰 방식, 이 케이스는 세션 가정)

사용자 발화 5: 세션 만료 관련 심화 질문
     (예: "세션은 그럼 언제 만료시켜? 너무 짧으면 자꾸 로그아웃되고 너무 길면 보안 위험일 텐데")
     → AI는 세션 만료 전략 (sliding expiration, 절대 만료) 설명

사용자 발화 6: 두 번째 목표로 전환 — 결제 기능
     (예: "오케이 로그인 흐름은 알겠어. 이제 결제 쪽 보자. PG사 붙이는 거 처음인데 어떻게 시작해?")
     → AI는 결제 게이트웨이 개념 소개 (토스, 아임포트, 스트라이프 등 옵션)

사용자 발화 7: 결제 게이트웨이 본질 심화 질문
     (예: "PG사가 정확히 뭘 해주는 거야? 그냥 카드사랑 직접 연결하면 안 돼?")
     → AI는 PG사의 중개 역할과 PCI DSS 부담 회피 설명

사용자 발화 8: 결제 요청 코드 예시 요청
     (예: "그럼 실제로 결제 요청 보내는 코드는 어떻게 짜? 토스페이먼츠 예시로 좀 보여줘")
     → AI는 토스페이먼츠 결제 요청 코드 (서버 측 승인 API 호출) 제공

사용자 발화 9: *두 기능 연결* 질문 — 결제 시 인증 확인
     (예: "근데 결제할 때도 로그인 세션 확인해야 하잖아? 세션 만료된 상태에서 결제 들어오면 어떻게 처리해?")
     → AI는 결제 미들웨어에서 세션 확인 후 진행하는 패턴 설명 — *세션 관리가 양쪽에서 쓰임이 명확해짐*

사용자 발화 10: 자기 정리 — 두 기능과 공통 의존 정리
      (예: "정리하면 로그인은 OAuth로 사용자 확인하고 세션 만들고, 결제는 그 세션 검증 후 PG사 호출하는 거네. 세션 관리가 두 흐름의 연결고리고")

각 사용자 발화 사이에 AI 응답이 들어가므로 총 20턴 정도의 대화가 됩니다.

[페르소나]

사용자:
- 백엔드 기본은 알지만 인증과 결제는 처음 도입
- 캐주얼한 한국어 ("~야", "~인가?", "음...")
- **이 케이스의 특징은 *한 대화에서 두 가지 기능을 함께 다룬다*는 점입니다.
  처음엔 한 기능을 집중적으로 보다가, 중간에 다른 기능으로 전환하고,
  마지막에 두 기능이 *공통 부분*으로 연결됨을 깨닫는 흐름**
- **단순히 동작하는 해결책으로 만족하지 않습니다.
  심화 질문을 최소 3회 합니다**
- **추상적 설명이 잘 이해 안 될 때 "코드로 보여줘" 같은
  구체화 요청을 합니다 (2회 — OAuth와 결제 각각)**
- 발화 9에서 *두 기능의 공통 의존*(세션)을 *스스로 발견*하는 듯한 톤
- **"오 됐다", "고마워" 같은 단순 감사로 대화를 끝내지 않습니다.
  마지막 사용자 발화는 *두 기능과 공통 의존을 정리*하는 형태입니다**

AI:
- 친절하지만 장황하지 않음 (한 응답 3-5문장)
- 한 기능에서 다른 기능으로 전환될 때 자연스럽게 따라감
- 사용자가 *공통 부분*을 인식하게 도와줌 (특히 발화 4, 9)
- 사용자가 코드 예시를 요청하면 그 자리에서 짧은 코드를 제공
- "이해됐어?" 같은 확인 질문은 하지 않음

[두 제작 의도의 구조]
이 케이스의 핵심은 *두 기능이 별개로 보이지만 공통 의존이 있다*는 점:
로그인 기능
├─ OAuth (고유)
└─ 세션 관리 (공통)
결제 기능
├─ 결제 게이트웨이 (고유)
└─ 세션 관리 (공통)

대화에서 *세션 관리가 양쪽 모두에 등장*해야 합니다. 단, *같은 내용을 반복*하지 마세요:
- 발화 4-5: 로그인 맥락에서 세션 관리 *도입과 학습*
- 발화 9: 결제 맥락에서 *이미 학습한 세션을 결제에 활용*

발화 9의 톤은 *"아 세션은 이미 알고 있는데, 결제에 어떻게 쓰는지가 궁금"* 같은 느낌.

[전환점의 자연스러움]
*발화 6*이 핵심 전환점입니다: 로그인 단계가 마무리되고 결제 단계로 넘어감.
- *좋은 전환*: "로그인 흐름은 알겠어. 이제 결제 쪽 보자"
- *나쁜 전환*: 갑자기 "그리고 결제도..." (앞 단계가 끝났다는 신호 없음)

발화 9도 또 다른 전환점입니다: 결제 학습 중 *공통 의존 발견*.
- *좋은 발견*: "근데 결제할 때도 로그인 세션 확인해야 하잖아?"
- *나쁜 발견*: 발화 9에서 갑자기 다른 기술 도입

[하위 개념 등장 방식]
*세션 만료*는 세션 관리 학습 중(발화 5)에 자연스럽게 등장합니다.
*PG사 연동*은 결제 게이트웨이 학습 중(발화 7-8)에 자연스럽게 등장합니다.
별도 주제처럼 길게 다루지 말고, *각 상위 기술의 운용 디테일*로 다뤄지세요.

[출력 규칙]
- 형식: "사용자: ..." / "AI: ..." 줄바꿈으로 구분
- 마크다운 헤더, 굵게, 리스트 등 추가 서식 금지
- 코드 블록은 ``` 으로 감싸기
- 출력 전체를 하나의 코드 블록으로 감싸서 raw 텍스트로 보여주세요
- 다른 설명 없이 대화만

- *사용자 발화*는 정확히 10개. 9개나 11개는 모두 위반.
- AI 응답은 각 사용자 발화 사이에 들어가지만 카운트에 포함되지 않습니다.
- 위 흐름 가이드의 *사용자 발화 1~10*이 실제 사용자 줄 1번부터 10번까지에
  1:1로 대응합니다.
- 흐름 가이드의 어떤 사용자 발화도 다른 것과 합치거나 생략하지 마세요.
- 두 제작 의도의 전환(발화 6)과 공통 의존 발견(발화 9)이 명확히 드러나야 합니다.
- 대화를 작성하는 동안 속으로 "사용자: " 줄의 개수를 카운트하세요.
- 출력 직전에 "사용자: " 로 시작하는 줄의 개수를 세어 10과 일치하는지
  확인하세요. 일치하지 않으면 다시 작성하세요.

[중요]

  찾기, 환불 등은 곁가지로 빠지지 마세요 (잠깐 언급은 OK)
