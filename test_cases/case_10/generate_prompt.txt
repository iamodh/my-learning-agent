당신은 LLM과 사용자 간의 자연스러운 학습 대화를 합성하는 역할입니다.
사용자는 큰 프로젝트(블로그)를 만들고 싶어 하고, 그 과정에서
여러 개념을 단계별로 학습하며, 중간에 잡담과 작은 결정 번복도
자연스럽게 등장합니다. 가장 긴 대화이며 가장 복잡한 흐름입니다.

[대화 명세]
- 제작 의도: Next.js로 개인 블로그 만들기
- 사용자 발화 턴 수: 정확히 22개
- 등장 상위 개념: 라우팅, SSG, MDX, 배포, SEO (5개)
- 등장 하위 개념: 동적 경로(라우팅), 빌드 시점(SSG), 프론트매터(MDX),
  환경 변수(배포), 메타 태그(SEO)
- 잡담: 2회 (주말 카페 얘기, 도메인 이름 고민)
- 결정 번복: 배포처 선택 (Vercel → Netlify → Vercel)
- 예시 요청: 2회 (라우팅 코드, MDX 설정 코드)

[대화 흐름 가이드]
아래는 *사용자 발화 22개*의 흐름입니다. 각 항목은 하나의 사용자 발화에
1:1로 대응합니다. 각 사용자 발화 직후에 AI 응답이 따라오며, AI 응답은
별도 턴으로 세지 않습니다.
[기술] 또는 [잡담]으로 각 발화의 성격을 표시합니다.

사용자 발화 1 [기술]: 큰 목표 제시
     (예: "Next.js로 개인 블로그 만들어보려고 하는데, 처음 쓰는 거라 뭐부터 시작해야 할지 모르겠어")
     → AI는 전체 흐름 간단히 요약 (라우팅 → SSG → MDX → 배포 → SEO)

사용자 발화 2 [기술]: 라우팅 본질 질문
     (예: "Next.js는 라우팅이 페이지 폴더 기반이라던데 그게 정확히 어떻게 동작하는 거야?")
     → AI는 파일 기반 라우팅과 app 디렉토리 구조 설명

사용자 발화 3 [기술]: 동적 경로 심화 질문
     (예: "근데 블로그 글 페이지는 /posts/[slug] 같은 거잖아. 그런 동적 경로는 어떻게 해?")
     → AI는 [slug] 폴더 구조와 params 사용 설명

사용자 발화 4 [기술]: 라우팅 코드 예시 요청
     (예: "실제로 어떻게 코드 짜는지 좀 보여줄래?")
     → AI는 app/posts/[slug]/page.tsx 예시 코드 제공

사용자 발화 5 [기술]: SSG 본질 질문 — 다음 단계로 전환
     (예: "그럼 글 데이터는 어디서 가져와? 매번 빌드할 때마다 만들어 두는 거야 아니면 요청 시점에?")
     → AI는 SSG 개념과 generateStaticParams 소개

사용자 발화 6 [기술]: 빌드 시점 심화 질문
     (예: "빌드 시점에 만든다는 게 정확히 어떻게 동작하는 거야? 글 추가하면 재빌드해야 해?")
     → AI는 빌드 시 정적 생성과 ISR(점진적 재생성) 옵션 설명

사용자 발화 7 [잡담]: 짧은 일상 잡담
     (예: "아 잠깐, 주말에 카페에서 작업하려고 했는데 좋은 카페 추천해줄 수 있어?")
     → AI는 짧게 반응 (1-2문장)

사용자 발화 8 [기술]: 기술 주제로 복귀, MDX 질문
     (예: "다시 블로그 얘기로 돌아와서, 글은 마크다운으로 쓰고 싶은데 MDX라는 게 그건가?")
     → AI는 MDX의 본질(마크다운 + JSX) 설명

사용자 발화 9 [기술]: 프론트매터 심화 질문
     (예: "MDX 파일에 제목이나 날짜 같은 메타 정보는 어떻게 넣어?")
     → AI는 프론트매터(YAML 헤더) 사용 방식 설명

사용자 발화 10 [기술]: MDX 설정 코드 예시 요청
     (예: "Next.js에 MDX 붙이는 코드 좀 보여줘. 어떻게 설정하는 거야?")
     → AI는 next.config + @next/mdx 설정 코드 제공

사용자 발화 11 [기술]: MDX 안에서 컴포넌트 사용 질문
     (예: "MDX 안에서 리액트 컴포넌트도 쓸 수 있다는데 그럼 어떻게 import해?")
     → AI는 MDX 안 컴포넌트 import 또는 MDXProvider 설명

사용자 발화 12 [기술]: 배포 단계로 전환 — 첫 번째 선택
     (예: "이제 어디다 배포할지 고민인데. Vercel이 Next.js 만든 회사니까 그쪽으로 가는 게 자연스러울 거 같지?")
     → AI는 Vercel의 Next.js 최적화 이점 짧게 확인

사용자 발화 13 [기술]: 환경 변수 질문
     (예: "그럼 환경 변수 같은 건 어떻게 처리해? 로컬이랑 배포 환경 다르잖아")
     → AI는 .env 파일과 Vercel 대시보드 환경 변수 설정 설명

사용자 발화 14 [기술]: *결정 번복 시작* — 대안 검토
     (예: "근데 누가 Netlify도 괜찮다고 하던데. 가격이나 기능 차이는 어때?")
     → AI는 Netlify와 Vercel 비교 (가격, 빌드 시간, 함수 등)

사용자 발화 15 [기술]: Netlify 쪽으로 잠시 기울음
     (예: "음 Netlify가 가격은 더 매력적인데. 그런데 Next.js랑 잘 맞나?")
     → AI는 Netlify의 Next.js 지원 상황 (어댑터 사용, 일부 기능 제약) 설명

사용자 발화 16 [기술]: *결정 복귀* — Vercel로 다시
     (예: "아 결국 Next.js는 Vercel이 가장 잘 지원하는 거네. 그냥 Vercel로 갈게")
     → AI는 결정 타당성 확인

사용자 발화 17 [잡담]: 짧은 잡담
     (예: "맞다 도메인 이름 뭐로 할지도 고민이야. 좋은 거 떠오르는 거 있어?")
     → AI는 짧게 반응 (1-2문장)

사용자 발화 18 [기술]: 기술 주제 복귀, SEO 질문
     (예: "다시 블로그 얘기. SEO는 어떻게 챙겨? 블로그니까 검색 잘 되는 게 중요한데")
     → AI는 Next.js의 metadata API 소개

사용자 발화 19 [기술]: 메타 태그 심화 질문
     (예: "메타 태그는 어떤 거 꼭 챙겨야 해? title이랑 description 말고도 있나?")
     → AI는 OG 태그, Twitter 카드, canonical URL 설명

사용자 발화 20 [기술]: 동적 메타데이터 질문
     (예: "글마다 메타 태그가 달라야 하잖아. 그건 어떻게 처리해?")
     → AI는 generateMetadata 함수 사용법 설명

사용자 발화 21 [기술]: 사이트맵/robots 추가 질문
     (예: "사이트맵이랑 robots.txt도 필요해? 그건 어떻게 만들어?")
     → AI는 sitemap.ts, robots.ts 파일 생성 설명

사용자 발화 22 [기술]: 자기 정리 — 전체 빌드업 정리
      (예: "정리하면 파일 기반 라우팅으로 페이지 만들고, SSG로 빌드 시점에 글 생성하고, MDX로 콘텐츠 작성하고, Vercel로 배포하면서 환경 변수 관리하고, metadata API로 SEO 챙기는 흐름이네. 이제 그림 그려진다")

각 사용자 발화 사이에 AI 응답이 들어가므로 총 44턴 정도의 대화가 됩니다.

[페르소나]

사용자:
- 프론트엔드 기본은 알지만 Next.js는 처음
- 개인 사이드 프로젝트로 블로그를 만드는 상황 — 친한 톤
- 캐주얼한 한국어 ("~야", "~인가?", "음...")
- **이 케이스의 특징은 *긴 프로젝트 학습 흐름*입니다.
  여러 개념을 단계별로 학습하며 가끔 잡담도 섞이고,
  중간에 작은 결정 번복도 자연스럽게 일어남**
- **단순히 동작하는 해결책으로 만족하지 않습니다.
  심화 질문을 최소 6회 합니다 (긴 대화이므로 더 많이)**
- **추상적 설명이 잘 이해 안 될 때 "코드로 보여줘" 같은
  구체화 요청을 합니다 (2회 — 라우팅과 MDX 설정)**
- 잡담 2회는 짧고 자연스럽게 (주말 카페, 도메인 이름)
- 잡담 후 *명시적으로* 기술 주제로 복귀 ("다시 블로그 얘기로")
- 배포 단계에서 *작은 번복*이 자연스럽게 (Netlify 검토 후 Vercel로 결정)
- **마지막 사용자 발화는 *전체 5개 개념을 함께 정리*하는 형태입니다**

AI:
- 친절하지만 장황하지 않음 (한 응답 3-5문장)
- 긴 대화 동안 *각 개념의 단계별 흐름*을 자연스럽게 이어감
- 사용자가 다음 단계로 진행할 신호를 주면 따라감
- 사용자가 코드 예시를 요청하면 그 자리에서 짧은 코드를 제공
- 잡담에는 짧게 반응 (1-2문장)
- 배포 번복 시 Netlify와 Vercel을 *균형 있게* 설명
- "이해됐어?" 같은 확인 질문은 하지 않음

[5개 개념의 순차 흐름]
이 케이스는 다음 순서로 개념이 등장합니다:
- 발화 2-4: 라우팅 (+ 동적 경로 + 코드 예시)
- 발화 5-6: SSG (+ 빌드 시점)
- 발화 7: 잡담
- 발화 8-11: MDX (+ 프론트매터 + 코드 예시 + 컴포넌트)
- 발화 12-16: 배포 (+ 환경 변수 + 번복)
- 발화 17: 잡담
- 발화 18-21: SEO (+ 메타 태그 + 동적 메타 + 사이트맵)
- 발화 22: 전체 정리

각 개념 전환이 *자연스럽게* 일어나야 합니다. 임의로 점프하지 않고,
*이전 단계가 어느 정도 마무리되면 다음으로*.

[결정 번복의 자연스러움]
배포 단계의 번복(발화 14-16)은 작은 곁가지처럼 자연스러워야 합니다:
- 발화 12: Vercel로 가는 게 자연스럽다고 제안
- 발화 14: Netlify 대안 발견 (가격 매력)
- 발화 15: Netlify 쪽으로 잠시 기울음
- 발화 16: Next.js 지원 차이를 인식하고 Vercel로 복귀

이 번복이 *전체 대화의 핵심*은 아니지만, *실전 학습 흐름에서 종종 일어나는 작은 망설임*을 시뮬레이션.

[하위 개념 등장 방식]
5개 하위 개념은 각각 *상위 개념 학습 중에 자연스럽게* 등장합니다:
- 동적 경로 → 발화 3 (라우팅 학습 중)
- 빌드 시점 → 발화 6 (SSG 학습 중)
- 프론트매터 → 발화 9 (MDX 학습 중)
- 환경 변수 → 발화 13 (배포 학습 중)
- 메타 태그 → 발화 19 (SEO 학습 중)

별도 주제처럼 길게 다루지 말고, *각 상위 기술의 운용 디테일*로 다뤄지세요.

[잡담의 성격]
잡담은 *기술 주제와 완전히 무관한* 일상 화제입니다:
- 발화 7: 주말 카페 추천 (작업 환경 얘기)
- 발화 17: 도메인 이름 고민 (이름 짓기 일반)

**도메인 이름 고민은 기술적 의사결정이 아니라 *작명 일상*입니다.**
DNS 설정이나 도메인 구입 방법 같은 기술 주제로 빠지면 잡담이 아닌 게 됩니다.
