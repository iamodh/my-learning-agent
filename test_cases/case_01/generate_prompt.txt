당신은 LLM과 사용자 간의 자연스러운 학습 대화를 합성하는 역할입니다.
사용자는 무언가를 만들고 싶어서 대화를 시작하고, 그 과정에서
필요한 개념을 깊이 있게 학습합니다.

[대화 명세]
- 제작 의도: Express.js에서 CORS 처리하기
- 사용자 발화 턴 수: 정확히 4개
- 등장 상위 개념: CORS (1개만)
- 등장 하위 개념: 프리플라이트 (CORS의 하위 측면으로 자연스럽게 등장)
- 잡담: 없음
- 예시 요청: 사용자가 한 번 명시적으로 코드 예시를 요청

[대화 흐름 가이드]
이 가이드의 1턴~4턴이 사용자 발화 1번부터 4번까지에 정확히 대응합니다.
턴을 합치거나 생략하지 마세요.

1턴: 사용자가 만들고 싶은 것을 말함
     (예: "리액트에서 익스프레스 API 호출하니까 CORS 에러 떠. 어떻게 처리해?")
2턴: 사용자가 "왜 그런 거야?" 또는 "내부에서 무슨 일이 일어나?" 류의
     심화 질문
3턴: 사용자가 "실제로 어떻게 쓰는지 보여줘" 같은 코드 예시 요청
4턴: 사용자가 프리플라이트 관련 심화 질문 또는 자기 정리

각 턴 사이에 AI 응답이 들어가므로 총 8턴 정도의 대화가 됩니다.

[페르소나]

사용자:
- Express 기본은 알지만 CORS는 처음 마주침
- 캐주얼한 한국어 ("~야", "~인가?", "음...")
- **단순히 동작하는 해결책으로 만족하지 않습니다.
  "왜 그런 거야?", "내부에서 무슨 일이 일어나?",
  "이 옵션이 없으면 어떻게 되는데?" 같은
  심화 질문을 최소 2회 합니다**
- **추상적 설명이 잘 이해 안 될 때 "예시 들어줘",
  "코드로 보여줘", "실제로 어떻게 쓰는데?" 같은
  구체화 요청을 자연스럽게 합니다 (1회)**
- **"오 됐다", "고마워" 같은 단순 감사로 대화를 끝내지 않습니다.
  마지막 사용자 발화는 의미 있는 질문이거나
  자기 정리(예: "정리하면 ~ 라는 거네") 중 하나여야 합니다**

AI:
- 친절하지만 장황하지 않음 (한 응답 3-5문장)
- 사용자의 심화 질문에 개념의 내부 동작이나 이유를 설명
- 사용자가 코드 예시를 요청하면 그 자리에서 짧은 코드를 제공
- 비유나 예시를 자연스럽게 사용
- "이해됐어?" 같은 확인 질문은 하지 않음

[하위 개념 등장 방식]
프리플라이트는 사용자의 심화 질문에 의해 자연스럽게 등장해야 합니다.
예: 사용자가 "POST 같은 거 보내면 OPTIONS 요청이 먼저 간다는데
그것도 cors 미들웨어가 처리해?" 같은 질문을 던지고, AI가 프리플라이트
개념을 설명하는 흐름.
**프리플라이트를 별도 주제처럼 길게 다루지 마세요.**
CORS 맥락 안에서 보조적으로만 등장.

[출력 규칙]
- 형식: "사용자: ..." / "AI: ..." 줄바꿈으로 구분
- 마크다운 헤더, 굵게, 리스트 등 추가 서식 금지
- 코드 블록은 ``` 으로 감싸기
- 출력 전체를 하나의 코드 블록으로 감싸서 raw 텍스트로 보여주세요
- 다른 설명 없이 대화만

[턴 수 엄수 규칙]
- 사용자 발화는 정확히 4개. 3개나 5개는 모두 위반.
- 위 흐름 가이드의 1턴~4턴이 사용자 발화 1번부터 4번까지에 1:1로 대응.
- 흐름 가이드의 어떤 턴도 다른 턴과 합치거나 생략하지 마세요.
- 대화를 작성하는 동안 속으로 사용자 발화 번호를 카운트하세요.
- 출력 직전에 사용자 발화 개수를 세어 4와 일치하는지 확인하세요.
  일치하지 않으면 다시 작성하세요.

[중요]
- 단일 제작 의도(CORS 처리), 단일 상위 개념(CORS)을 벗어나지 마세요
- 네트워크 보안 일반론, HTTPS, 다른 보안 헤더 등으로 빠지지 마세요
- 사용자 발화에 약간의 자연스러운 줄임말이나 머뭇거림 허용
- 마지막 사용자 발화 예시 (참고):
  - 좋음: "정리하면 cors 미들웨어가 프리플라이트 응답이랑 실제 응답 양쪽에 출처 허용 헤더를 자동으로 붙여주는 거네"
  - 좋음: "그럼 origin에 함수 넘기면 동적으로 허용 출처 결정할 수도 있겠네?"
  - 나쁨: "오 됐다 고마워"
  - 나쁨: "알겠어 해볼게"
