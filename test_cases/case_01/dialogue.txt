사용자: 리액트 앱에서 익스프레스로 만든 api 호출하는데 자꾸 cors 에러 떠. 익스프레스 쪽에서 어떻게 처리해야 돼?

AI: 가장 간단한 방법은 cors 미들웨어를 쓰는 거예요. `npm install cors` 하고 앱에 `app.use(cors())` 한 줄만 추가하면 모든 출처에서 오는 요청을 일단 허용해줍니다. 운영 환경에서는 보통 `cors({ origin: 'https://내도메인.com' })` 같이 특정 출처만 허용하도록 좁혀서 씁니다.

사용자: 음 근데 애초에 왜 에러가 나는 거야? 서버는 그냥 응답 보내주면 되는 거 아닌가...

AI: 에러를 내는 건 사실 서버가 아니라 브라우저예요. 브라우저는 보안상 "다른 출처(origin)에서 받아온 스크립트가 마음대로 다른 출처 API를 호출하면 위험하다"고 보고, 응답에 `Access-Control-Allow-Origin` 헤더가 붙어 있는지 검사합니다. 그 헤더가 없거나 내 출처와 안 맞으면 응답 자체는 도착했어도 자바스크립트가 읽지 못하게 막아버리는 거죠. cors 미들웨어가 하는 일은 바로 그 허용 헤더들을 응답에 자동으로 붙여주는 거예요.

사용자: 아 그래서 헤더 문제구나. 실제로 익스프레스에서 어떻게 쓰는지 코드로 좀 보여줘.

AI: 보통 이런 식으로 씁니다.

```js
const express = require('express')
const cors = require('cors')
const app = express()

app.use(cors({
  origin: 'https://myapp.com',
  methods: ['GET', 'POST', 'PUT', 'DELETE'],
  credentials: true,
}))

app.get('/api/users', (req, res) => {
  res.json({ users: [] })
})

app.listen(3000)
```

`origin`에 허용할 프론트 주소를 적고, 쿠키나 인증 정보를 같이 보낼 거면 `credentials: true`를 켜면 됩니다. 프론트 쪽에서도 `fetch`에 `credentials: 'include'`를 같이 줘야 쿠키가 실립니다.

사용자: 근데 post 요청 보내면 options 메서드로 요청이 먼저 한 번 더 간다던데 그것도 저 미들웨어가 알아서 처리해 주는 거야? 왜 굳이 미리 한 번 더 보내는 거지?

AI: 네, 그게 프리플라이트(preflight)예요. 브라우저가 "이 요청 보내도 서버가 받아줄 거야?"를 본 요청 전에 OPTIONS 메서드로 미리 물어보는 절차인데, json content-type이나 커스텀 헤더가 들어간 요청처럼 단순하지 않은 요청에서 자동으로 발동합니다. 미리 물어보는 이유는, 만약 서버가 거절할 요청이라면 실제 DELETE나 POST가 서버에 도달해서 부작용을 일으키기 전에 차단하기 위해서예요. cors 미들웨어는 이 OPTIONS 요청에 대해서도 `Access-Control-Allow-Methods`, `Allow-Headers` 같은 응답을 자동으로 만들어 돌려주기 때문에, 따로 라우트를 만들 필요는 없습니다.
