사용자: 리액트 프론트에서 익스프레스 API 부르니까 cors 에러 뜨네. 어떻게 처리해?

AI: 브라우저가 다른 출처(origin)로 가는 요청을 막아서 그래. 프론트랑 API 포트가 다르면 다른 출처로 보거든. Express에서는 cors 미들웨어 깔아서 app.use(cors())로 붙여주면 일단 해결돼. 운영에서는 origin 옵션으로 허용할 도메인을 지정하는 게 안전해.

사용자: 음... 근데 왜 브라우저가 그걸 막아? 서버끼리 통신하면 안 막히잖아. 내부에서 무슨 일이 일어나는 거야?

AI: 브라우저 안에 동일 출처 정책(Same-Origin Policy)이라는 보호 장치가 있어서 그래. 네가 로그인된 사이트의 쿠키를 들고, 악성 사이트가 몰래 네 은행 API를 호출하면 곤란하잖아. 그래서 브라우저는 다른 출처로 가는 요청에 대해 서버가 "이 출처 허용해"라고 Access-Control-Allow-Origin 헤더로 명시적으로 응답해야 결과를 자바스크립트에 넘겨줘. 서버끼리 통신은 브라우저가 끼지 않으니까 이 검사 자체가 없는 거고.

사용자: 아 그렇구나. 근데 POST로 json 보내면 OPTIONS 요청이 먼저 간다던데, 그것도 cors 미들웨어가 알아서 처리해?

AI: 응, 그게 프리플라이트야. 브라우저는 본 요청이 좀 까다롭다 싶으면(예: Content-Type이 application/json이거나 커스텀 헤더가 붙으면) 먼저 OPTIONS로 "이런 메서드랑 헤더로 보내도 돼?"라고 물어봐. 서버가 허용 헤더를 응답하면 그제서야 실제 POST를 보내는 거지. cors 미들웨어는 이 OPTIONS 요청을 가로채서 자동으로 응답해줘. 보통 이렇게 옵션만 잡아주면 돼.

```js
app.use(cors({
  origin: 'https://myapp.com',
  credentials: true
}))
```

사용자: credentials: true는 뭐 하는 옵션이야? 이거 없으면 어떻게 되는데?

AI: 쿠키나 Authorization 헤더 같은 인증정보를 크로스 오리진 요청에 실어 보낼지 결정하는 옵션이야. 이게 false면 프론트에서 fetch에 credentials: 'include' 줘도 브라우저가 쿠키를 빼고 보내. 그리고 서버 응답에 Access-Control-Allow-Credentials: true가 없으면 브라우저가 응답 자체를 거부해버려. 추가로 이걸 켤 땐 origin을 와일드카드(*)로 두면 안 되고 명시적인 도메인이어야 해. 보안상 "아무 사이트나 인증쿠키 받기"는 위험하니까 막아둔 거지.

사용자: 정리하면 cors 미들웨어가 프리플라이트 OPTIONS랑 실제 응답 양쪽에 허용 헤더를 자동으로 붙여주고, credentials 옵션으로 쿠키 동반 여부까지 제어하는 거네. 결국 서버는 "허용 정책"만 선언하고 강제는 브라우저가 하는 구조구나.

AI: 정확해. 서버는 정책을 헤더로 표현할 뿐이고, 실제로 차단을 수행하는 주체는 브라우저야. 그래서 curl이나 Postman으로는 CORS 에러가 절대 안 떠 — 걔들은 같은 정책을 강제하지 않거든. 디버깅할 때 이 구분을 기억해두면 헤매는 시간이 확 줄어.
