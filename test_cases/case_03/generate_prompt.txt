당신은 LLM과 사용자 간의 자연스러운 학습 대화를 합성하는 역할입니다.
사용자는 무언가를 만들고 싶어서 대화를 시작하고, 그 과정에서
필요한 개념을 깊이 있게 학습합니다.

[대화 명세]
- 제작 의도: Express에서 JWT 인증 구현
- 사용자 발화 턴 수: 정확히 10개
- 등장 상위 개념: JWT (1개만)
- 등장 하위 개념: 페이로드 구조, 만료 처리, 리프레시 토큰
  → 모두 JWT 맥락 안에서 보조적으로 등장
- 잡담: 없음
- 예시 요청: 사용자가 2회 명시적으로 코드 예시 요청
  (토큰 생성 흐름 1회, 리프레시 토큰 흐름 1회)

[대화 흐름 가이드 — 같은 개념을 다른 각도로 반복 심화]
이 가이드의 1턴~10턴이 사용자 발화 1번부터 10번까지에 정확히 대응합니다.
턴을 합치거나 생략하지 마세요.

1턴: 사용자가 만들고 싶은 것을 말함
     (예: "Express 서버에 JWT 인증 붙이고 싶어. 어떻게 시작해야 해?")
2턴: 사용자가 "JWT가 정확히 뭐야? 세션이랑 뭐가 다른 거야?" 류의
     본질 질문 (첫 번째 각도 — 정의)
3턴: 사용자가 토큰 생성 코드 예시를 요청
     (예: "토큰 만드는 코드 좀 보여줘")
4턴: 사용자가 "페이로드에는 뭘 넣는 거야? 비밀번호 같은 거 넣어도 돼?"
     (두 번째 각도 — 페이로드 구조)
5턴: 사용자가 페이로드의 보안 관련 심화 질문
     (예: "그럼 페이로드는 누가 볼 수 있는 거야?")
6턴: 사용자가 "근데 토큰 만료되면 어떻게 처리해? 사용자가 매번
     다시 로그인해야 하나?" (세 번째 각도 — 만료 처리)
7턴: 사용자가 리프레시 토큰 본질 질문
     (예: "리프레시 토큰은 그럼 어디 저장해? 또 그것도 JWT야?")
8턴: 사용자가 리프레시 토큰 흐름 코드 예시를 요청
     (예: "리프레시 토큰으로 새 액세스 토큰 받는 코드 보여줄래?")
9턴: 사용자가 리프레시 토큰 보안 관련 추가 질문
     (예: "리프레시 토큰도 탈취되면 위험할 텐데 그건 어떻게 막아?")
10턴: 사용자가 자기 정리 또는 추가 심화 질문
      (예: "정리하면 액세스 토큰은 짧게, 리프레시 토큰은 길게 가져가서
       만료될 때마다 새로 발급받는 구조네")

각 턴 사이에 AI 응답이 들어가므로 총 20턴 정도의 대화가 됩니다.

[페르소나]

사용자:
- Express 기본은 알지만 JWT는 들어만 봤지 직접 써본 적 없음
- 캐주얼한 한국어 ("~야", "~인가?", "음...")
- **같은 큰 주제(JWT)를 여러 각도에서 반복적으로 파고듭니다.
  단순 사용법이 아니라 "왜 이렇게 설계됐는지", "안에 뭘 넣는지",
  "만료되면 어떻게 되는지" 같은 본질적 측면을 차례로 묻습니다**
- **단순히 동작하는 해결책으로 만족하지 않습니다.
  심화 질문을 최소 3회 합니다**
- **추상적 설명이 잘 이해 안 될 때 "예시 들어줘",
  "코드로 보여줘", "실제로 어떻게 쓰는데?" 같은
  구체화 요청을 자연스럽게 합니다
  (2회 — 토큰 생성과 리프레시 흐름 각각)**
- **"오 됐다", "고마워" 같은 단순 감사로 대화를 끝내지 않습니다.
  마지막 사용자 발화는 의미 있는 질문이거나
  자기 정리(예: "정리하면 ~ 라는 거네") 중 하나여야 합니다**

AI:
- 친절하지만 장황하지 않음 (한 응답 3-5문장)
- 사용자의 심화 질문에 개념의 내부 동작이나 이유를 설명
- 사용자가 코드 예시를 요청하면 그 자리에서 짧은 코드를 제공
- 비유나 예시를 자연스럽게 사용
- 같은 개념을 다른 각도로 묻는 흐름이므로, 매번 *그 각도에 맞는*
  설명을 제공 (구조 질문엔 구조를, 보안 질문엔 보안 관점을)
- "이해됐어?" 같은 확인 질문은 하지 않음

[반복 질문의 형태]
같은 상위 개념(JWT)을 *여러 각도*로 반복해야 합니다.
다른 개념으로 빠지는 게 아니라, 같은 개념을 깊이 파는 방식.
예:
- "JWT가 뭐야?" (정의)
- "페이로드엔 뭘 넣어?" (구조)
- "만료되면 어떻게 해?" (시간 처리)
- "리프레시 토큰은 어디 저장해?" (보관)
이렇게 *같은 개념의 다른 측면*을 묻는 것이지, OAuth나 세션처럼
*다른 개념*으로 빠지면 안 됩니다.

[하위 개념 등장 방식]
페이로드 구조, 만료 처리, 리프레시 토큰은 모두 JWT의 하위 측면입니다.
각각 사용자의 심화 질문에 의해 자연스럽게 등장하되, *별도 주제처럼*
길게 다루지 마세요. JWT를 이해하기 위한 보조 설명으로 들어와야 합니다.

[출력 규칙]
- 형식: "사용자: ..." / "AI: ..." 줄바꿈으로 구분
- 마크다운 헤더, 굵게, 리스트 등 추가 서식 금지
- 코드 블록은 ``` 으로 감싸기
- 출력 전체를 하나의 코드 블록으로 감싸서 raw 텍스트로 보여주세요
- 다른 설명 없이 대화만

[턴 수 엄수 규칙]
- 사용자 발화는 정확히 10개. 9개나 11개는 모두 위반.
- 위 흐름 가이드의 1턴~10턴이 사용자 발화 1번부터 10번까지에 1:1로 대응.
- 흐름 가이드의 어떤 턴도 다른 턴과 합치거나 생략하지 마세요.
- 대화를 작성하는 동안 속으로 사용자 발화 번호를 카운트하세요.
- 출력 직전에 사용자 발화 개수를 세어 10과 일치하는지 확인하세요.
  일치하지 않으면 다시 작성하세요.

[중요]
- 단일 제작 의도(JWT 인증)와 단일 상위 개념(JWT)을 벗어나지 마세요
- 세션 방식, OAuth, 소셜 로그인 등 *다른 인증 방식*으로 빠지지 마세요
  (잠깐 비교 언급은 OK, 길게 다루지는 말 것)
- bcrypt, 비밀번호 해싱, DB 설계 등 곁가지로 빠지지 마세요
- 사용자 발화에 약간의 자연스러운 줄임말이나 머뭇거림 허용
- 마지막 사용자 발화 예시 (참고):
  - 좋음: "정리하면 액세스 토큰은 짧게, 리프레시 토큰은 길게 가져가서 만료될 때마다 새로 발급받는 구조네"
  - 좋음: "그럼 리프레시 토큰도 탈취되면 위험할 텐데, 그건 어떻게 막아?"
  - 나쁨: "오 됐다 고마워"
  - 나쁨: "알겠어 한번 해볼게"
