# JWT (JSON Web Token)

JWT는 서버가 서명해서 발급한, 그 자체로 정보를 담고 있는 문자열이다. 세션 방식이 서버에 실제 정보를 보관하고 세션 ID만 클라이언트에 주는 것과 달리, JWT는 사용자 정보를 토큰 안에 직접 담아 보내고 서버는 별도 저장 없이 서명만 검증한다. 그래서 서버를 여러 대로 늘려도 세션 동기화 고민이 줄어드는 대신, 한번 발급된 토큰을 중간에 무효화하기는 어렵다는 트레이드오프가 있다.

## 구조

`헤더.페이로드.서명` 세 부분이 점으로 이어진 형태이며, 점 단위로 base64로 인코딩되어 있다.

## 서명 vs 암호화

JWT는 *암호화*가 아니라 *서명*이다. 페이로드는 base64 인코딩일 뿐이라 jwt.io 같은 데에 붙여넣으면 누구나 내용을 읽을 수 있다. 서명이 보장하는 것은 "이 내용이 발급 이후 변조되지 않았다"는 점이지 "내용이 비밀이다"가 아니다.

정말 숨겨야 하는 데이터라면 JWE라는 별도 방식을 쓰거나, 서버에 두고 ID로만 조회해야 한다.

## 페이로드에 무엇을 담는가

페이로드에는 "이 사용자가 누군지 식별할 수 있는 최소 정보"만 넣는다. 비밀번호·주민번호 같은 민감 정보는 절대 넣지 않는다 (위의 서명 vs 암호화 참고).

- 표준 클레임: `iss`(발급자), `exp`(만료), `iat`(발급시각), `sub`(주체)
- 커스텀 클레임: 본인이 정의 (예: `role`)

너무 많이 넣으면 토큰이 커져서 매 요청마다 전송 비용이 생기므로, 자주 쓰는 식별 정보만 넣는다.

## 발급

`jwt.sign(payload, secret, options)` 한 줄이 핵심이다.

```js
const jwt = require('jsonwebtoken');

const accessToken = jwt.sign(
  { sub: user.id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: '15m' }
);
```

- 첫 번째 인자: 페이로드
- 두 번째 인자: 서명용 비밀키
- 세 번째 인자: 옵션. `expiresIn`은 토큰에 만료시각(`exp` 클레임)을 자동으로 박아준다.

## 검증과 만료

검증 시점에 jsonwebtoken이 `exp` 클레임을 보고 자동으로 `TokenExpiredError`를 던진다. 이 에러를 잡아서 처리 흐름(예: 재발급 요청)으로 이어가면 된다.
