# 리프레시 토큰 저장과 회전

리프레시 토큰은 수명이 길기 때문에 탈취 시 위험이 크다. 그래서 액세스 토큰과 달리 **서버가 토큰의 생존 여부를 추적할 수 있어야 한다**. 형식이 JWT인지 단순 랜덤 문자열인지보다 이 추적 가능성이 더 중요하다.

## 서버 측 저장

리프레시 토큰은 DB에 저장해두고 발급/폐기를 관리한다. 이래야 로그아웃이나 탈취 시 무효화가 가능하다. 사용자ID·발급시각·기기정보를 함께 저장하면 의심스러운 패턴 감지 시 강제 로그아웃에 활용할 수 있다.

## 클라이언트 측 저장

클라이언트에는 httpOnly 쿠키로 담아 보내는 것이 일반적이다. JavaScript에서 접근할 수 없으므로 XSS 공격으로 훔쳐가기 어렵다. 함께 쓰는 옵션은 httpOnly + Secure + SameSite.

## 이중 검증

리프레시 요청이 들어오면 두 단계로 검증한다.

1. `jwt.verify`로 서명·만료를 확인
2. DB에 그 토큰이 아직 살아있는지(폐기되지 않았는지) 확인

서명만으로는 폐기 여부를 알 수 없기 때문에 DB 조회가 필수다.

## 리프레시 토큰 회전(rotation)

리프레시 요청이 들어올 때마다 새 리프레시 토큰을 발급하고 이전 것은 즉시 폐기한다.

방어 효과: 공격자가 훔친 토큰을 한 번이라도 쓰면 정상 사용자의 토큰은 이미 폐기된 상태가 된다. 그러면 다음에 정상 사용자가 요청할 때 인증 실패가 나고, 이를 신호로 해당 사용자의 모든 리프레시 토큰을 한꺼번에 무효화할 수 있다.
