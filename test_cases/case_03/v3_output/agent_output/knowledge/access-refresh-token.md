# 액세스 토큰 / 리프레시 토큰 이중 구조

토큰 하나로 인증을 운영하면, 수명을 짧게 잡으면 사용자가 자주 다시 로그인해야 하고, 길게 잡으면 탈취 시 피해 시간이 길어진다. 이 트레이드오프를 풀기 위해 **수명이 다른 두 종류 토큰을 함께 쓴다**.

- **액세스 토큰**: 짧은 수명(예: 15분). 평소 API 호출에 사용. 자주 노출되므로 탈취돼도 피해 시간을 최소화하기 위해 짧게 유지.
- **리프레시 토큰**: 긴 수명(예: 2주). 액세스 토큰이 만료되면 이걸로 새 액세스 토큰을 받아온다. 사용자가 매번 다시 로그인하지 않도록 하는 백업.

## 신뢰 수준의 분리

이 구조의 본질은 "각 토큰이 서로 다른 신뢰 수준을 갖는다"는 점이다.

- 액세스 토큰: 자주 쓰여서 노출 위험이 높음 → 짧고 가볍게
- 리프레시 토큰: 드물게 쓰임 → 길고 DB 추적이 가능한 형태로 (자세한 보관·회전 방식은 [리프레시 토큰 저장과 회전](./refresh-token-storage.md) 참고)

## 시크릿 분리

액세스 토큰용 시크릿과 리프레시 토큰용 시크릿은 분리하는 것이 좋다.

## 만료 시 흐름

액세스 토큰이 만료되면 jsonwebtoken이 `TokenExpiredError`를 던지므로, 이를 잡아 클라이언트가 리프레시 엔드포인트로 재발급 요청을 보내게 한다.
