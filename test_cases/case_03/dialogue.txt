사용자: 잠깐, 그 전에 JWT가 정확히 뭐야? 세션이랑 뭐가 다른 건데? 이름만 들어봤지 본질이 뭔지 모르겠어.

AI: JWT는 한마디로 "서버가 서명해서 발급한, 그 자체로 정보를 담고 있는 문자열"이에요. 세션 방식은 서버가 세션 ID만 주고 실제 정보는 서버 메모리/DB에 보관하지만, JWT는 사용자 정보를 토큰 안에 직접 담아서 보내고 서버는 별도 저장 없이 서명만 검증해요. 그래서 서버를 여러 대로 늘려도 세션 동기화 같은 고민이 줄어드는 대신, 한번 발급된 토큰을 중간에 무효화하기는 어렵다는 트레이드오프가 있어요. 구조는 `헤더.페이로드.서명` 세 부분이 점으로 이어진 형태고, 점 단위로 base64로 인코딩돼 있어요.

사용자: 음 일단 토큰 만드는 코드 좀 보여줘. 로그인 성공했다 치고 어떻게 발급하는지.

AI: 핵심은 `jwt.sign(payload, secret, options)` 한 줄이에요.

```js
const jwt = require('jsonwebtoken');

app.post('/login', async (req, res) => {
  const user = await findUser(req.body.email, req.body.password);
  if (!user) return res.status(401).json({ message: '인증 실패' });

  const accessToken = jwt.sign(
    { sub: user.id, role: user.role },
    process.env.JWT_SECRET,
    { expiresIn: '15m' }
  );

  res.json({ accessToken });
});
```

여기서 첫 번째 인자가 페이로드, 두 번째가 서명용 비밀키, 세 번째가 옵션이에요. `expiresIn`은 토큰에 만료시각(exp 클레임)을 자동으로 박아줘요.

사용자: 페이로드에는 뭘 넣는 거야? 사용자 정보 다 넣어도 돼? 비밀번호 같은 거 넣어도 되나?

AI: 페이로드에는 "이 사용자가 누군지 식별할 수 있는 최소 정보"만 넣는 게 원칙이에요. 보통 `sub`(사용자 ID), `role` 정도면 충분하고, 비밀번호나 주민번호 같은 민감 정보는 절대 넣지 않아요. 표준 클레임으로 `iss`(발급자), `exp`(만료), `iat`(발급시각) 같은 게 있고, 나머지는 본인이 정의하는 커스텀 클레임이에요. 너무 많이 넣으면 토큰이 커져서 매 요청마다 전송 비용이 생기니까 정말 자주 쓸 식별 정보만 넣는다고 생각하면 돼요.

사용자: 잠깐, 비밀번호 같은 거 넣지 말라고 했는데 그 말은 페이로드를 누군가가 볼 수 있다는 얘기야? 암호화된 거 아니야?

사용자: 아 그렇구나. 근데 아까 만료시간 15분으로 했잖아. 그럼 15분 지나면 어떻게 처리해? 사용자가 매번 다시 로그인해야 하는 거야?

AI: 그러면 사용자 경험이 너무 안 좋죠. 그래서 보통 두 종류 토큰을 같이 써요. 짧은 수명(예: 15분)의 액세스 토큰으로 평소 API를 호출하고, 그게 만료되면 별도로 갖고 있던 긴 수명(예: 2주)의 리프레시 토큰으로 새 액세스 토큰을 받아오는 구조예요. 액세스 토큰을 짧게 유지하는 이유는 탈취돼도 피해 시간을 최소화하기 위함이고, 매번 로그인 안 시키려고 리프레시 토큰이라는 백업을 두는 거예요. 검증 시점에 jsonwebtoken이 `exp` 클레임을 보고 자동으로 `TokenExpiredError`를 던져주니까, 그 에러를 잡아서 클라이언트가 재발급 요청을 보내게 하면 돼요.

사용자: 리프레시 토큰은 그럼 어디 저장해? 그것도 JWT야 아니면 다른 거야?

사용자: 리프레시 토큰으로 새 액세스 토큰 받는 코드 보여줄래? 흐름이 잘 안 그려져.

AI: 보통 `/refresh` 엔드포인트를 따로 만들어요.

```js
app.post('/refresh', async (req, res) => {
  const token = req.cookies.refreshToken;
  if (!token) return res.status(401).json({ message: '토큰 없음' });

  try {
    const payload = jwt.verify(token, process.env.REFRESH_SECRET);

    const stored = await db.refreshTokens.findOne({ token });
    if (!stored) return res.status(401).json({ message: '폐기된 토큰' });

    const newAccessToken = jwt.sign(
      { sub: payload.sub, role: payload.role },
      process.env.JWT_SECRET,
      { expiresIn: '15m' }
    );

    res.json({ accessToken: newAccessToken });
  } catch {
    res.status(401).json({ message: '리프레시 실패' });
  }
});
```

핵심은 두 단계 검증이에요. `jwt.verify`로 서명·만료를 보고, 그 다음 DB에 그 토큰이 아직 살아있는지 한 번 더 확인해요. 액세스 토큰용 시크릿과 리프레시 토큰용 시크릿은 분리하는 게 좋고요.

사용자: 근데 리프레시 토큰이 수명이 길면 그게 탈취됐을 때 더 위험한 거 아니야? 그건 어떻게 방어해?

AI: 핵심 대응은 두 가지예요. 첫째, "리프레시 토큰 회전(rotation)"이라고 해서 리프레시 요청이 들어올 때마다 새 리프레시 토큰을 발급하고 이전 건 즉시 폐기해요. 그러면 만약 공격자가 훔친 토큰을 한 번이라도 쓰면 정상 사용자의 토큰은 이미 폐기된 상태가 돼서 다음에 정상 사용자가 요청할 때 인증 실패가 나고, 그걸 감지해서 해당 사용자의 모든 리프레시 토큰을 한꺼번에 무효화할 수 있어요. 둘째, 위에서 말한 httpOnly + Secure + SameSite 쿠키로 저장하는 것, 그리고 DB에 사용자ID·발급시각·기기정보 같이 저장해서 의심스러운 패턴이 보이면 강제 로그아웃시키는 거예요. 액세스 토큰을 짧게 유지하는 이유도 결국 이 방어선의 일부예요.

사용자: 정리하면 액세스 토큰은 짧게 가져가서 탈취 피해를 줄이고, 그 불편함은 긴 수명의 리프레시 토큰으로 메우되 그 리프레시 토큰은 DB에 추적 가능하게 두고 회전시켜서 위험을 분산시키는 구조네. 페이로드는 노출되니까 식별자만 넣고. 이렇게 이해하면 맞는 거지?
