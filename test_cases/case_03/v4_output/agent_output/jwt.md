# JWT (JSON Web Token)

JWT는 서버가 서명해서 발급한, 그 자체로 정보를 담고 있는 문자열이다. 사용자 정보를 토큰 안에 직접 담아 클라이언트에 내려보내고, 이후 매 요청마다 토큰이 함께 전송되면 서버는 별도 저장 없이 서명만 검증해 사용자를 식별한다.

## 세션 방식과의 차이

세션 방식은 서버가 세션 ID만 주고 실제 사용자 정보는 서버 메모리·DB에 보관한다. JWT는 정보 자체를 토큰에 담아 보내므로 서버 측 저장이 필요 없다.

- 장점: 서버를 여러 대로 늘려도 세션 동기화를 고민할 필요가 줄어든다.
- 트레이드오프: 한번 발급된 토큰을 중간에 무효화하기 어렵다.

## 구조: 헤더.페이로드.서명

세 부분이 점(`.`)으로 이어진 형태이고, 각 부분은 base64로 인코딩돼 있다.

중요한 점은 JWT가 **암호화가 아니라 서명**이라는 것이다. 페이로드는 base64 인코딩일 뿐이라 jwt.io 같은 도구에 붙여넣으면 누구나 내용을 읽을 수 있다. 서명이 보장하는 것은 "발급 이후 변조되지 않았다"이지 "내용이 비밀이다"가 아니다. 정말 숨겨야 하는 데이터라면 JWE라는 별도 방식을 쓰거나, 서버에 두고 ID로만 조회해야 한다.

## 페이로드에 넣을 것

"이 사용자가 누군지 식별할 수 있는 최소 정보"만 넣는 것이 원칙이다.

- 보통 `sub`(사용자 ID), `role` 정도면 충분하다.
- 비밀번호, 주민번호 같은 민감 정보는 절대 넣지 않는다 (위의 노출 이유).
- 표준 클레임으로 `iss`(발급자), `exp`(만료), `iat`(발급시각) 등이 있고, 그 외는 커스텀 클레임이다.
- 너무 많이 넣으면 토큰이 커져 매 요청 전송 비용이 늘어난다.

## 토큰 발급: `jwt.sign`

핵심은 `jwt.sign(payload, secret, options)` 한 줄이다.

```js
const jwt = require('jsonwebtoken');

const accessToken = jwt.sign(
  { sub: user.id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: '15m' }
);
```

- 첫 번째 인자: 페이로드.
- 두 번째 인자: 서명용 비밀키.
- 세 번째 인자: 옵션. `expiresIn`은 토큰에 만료시각(`exp` 클레임)을 자동으로 박아준다.

## 만료 처리 — 액세스 토큰과 리프레시 토큰

액세스 토큰만 짧게 두면 사용자가 만료될 때마다 다시 로그인해야 해서 경험이 나쁘다. 그래서 두 종류 토큰을 함께 쓴다.

- **액세스 토큰**: 짧은 수명(예: 15분). 평소 API 호출에 사용. 탈취되더라도 피해 시간이 짧다.
- **리프레시 토큰**: 긴 수명(예: 2주). 액세스 토큰이 만료됐을 때 새 액세스 토큰을 받기 위해 쓰인다.

검증 시점에 `jsonwebtoken`이 `exp` 클레임을 보고 자동으로 `TokenExpiredError`를 던진다. 그 에러를 잡아 클라이언트가 재발급 요청을 보내게 한다.

## 리프레시 토큰의 형식과 저장

- 형식: JWT여도 되고, 그냥 랜덤한 긴 문자열이어도 된다. 형식보다 중요한 것은 "서버가 이 토큰이 살아있는지 추적할 수 있어야 한다"는 점이다.
- 서버 측: DB에 저장해두고 발급/폐기를 관리한다. 그래야 로그아웃이나 탈취 시 무효화할 수 있다.
- 클라이언트 측: httpOnly 쿠키에 담는 것이 일반적이다. JavaScript에서 접근 못 하므로 XSS로 훔쳐가기 어렵다.

## 리프레시 토큰 회전 (rotation)

리프레시 토큰이 수명이 길어 탈취 시 위험이 큰 것을 보완하기 위한 방어이다.

- 리프레시 요청이 들어올 때마다 새 리프레시 토큰을 발급하고 이전 것은 즉시 폐기한다.
- 공격자가 훔친 토큰을 한 번이라도 쓰면, 정상 사용자의 토큰은 이미 폐기된 상태가 된다. 정상 사용자가 다음에 요청할 때 인증 실패가 나면 이를 감지해 그 사용자의 모든 리프레시 토큰을 한꺼번에 무효화할 수 있다.
- 추가로 httpOnly + Secure + SameSite 쿠키 저장, DB에 사용자ID·발급시각·기기정보 같이 저장해두고 의심스러운 패턴이 보이면 강제 로그아웃하는 운영을 결합한다.

## 검증 시 이중 안전장치

리프레시 토큰을 검증할 때는 두 단계를 모두 본다.

1. `jwt.verify`로 서명과 만료를 확인한다.
2. DB에 그 토큰이 아직 살아있는지 한 번 더 확인한다.

또한 액세스 토큰용 시크릿과 리프레시 토큰용 시크릿은 분리한다.

## 핵심 포인트 요약

- 각 토큰이 서로 다른 신뢰 수준을 갖는다: 액세스는 자주 쓰여 노출 위험이 높으므로 짧고 가볍게, 리프레시는 드물게 쓰이므로 길고 DB 추적이 가능한 형태로.
- 페이로드는 식별자만.
- 서명 검증 + DB 검증의 이중 안전장치.
- 토큰별로 시크릿 분리.
