# JWT 인증

JWT(JSON Web Token)는 서버가 서명해서 발급한, 그 자체로 정보를 담고 있는 문자열이다. 세션 방식이 서버에 정보를 두고 세션 ID만 주는 것과 달리, JWT는 사용자 정보를 토큰 안에 직접 담아 보내고 서버는 별도 저장 없이 서명만 검증한다. 그래서 서버를 여러 대로 늘려도 세션 동기화 고민이 줄어드는 대신, 한번 발급된 토큰을 중간에 무효화하기는 어렵다는 트레이드오프가 있다.

## 구조

`헤더.페이로드.서명` 세 부분이 점으로 이어진 형태이고, 점 단위로 base64로 인코딩되어 있다.

## 서명이지 암호화가 아니다

JWT는 *암호화*가 아니라 *서명*이다. 페이로드는 그냥 base64로 인코딩된 것이라 jwt.io 같은 데에 붙여넣으면 누구나 내용을 읽을 수 있다. 서명이 보장하는 것은 "이 내용이 발급 이후로 변조되지 않았다"는 것이지 "내용이 비밀이다"가 아니다. 정말 숨겨야 하는 데이터라면 JWE라는 별도 방식을 쓰거나, 서버에 두고 ID로만 조회해야 한다.

## 페이로드에 넣을 것

페이로드에는 "이 사용자가 누군지 식별할 수 있는 최소 정보"만 넣는다. 보통 `sub`(사용자 ID), `role` 정도면 충분하다. 비밀번호나 주민번호 같은 민감 정보는 절대 넣지 않는다(페이로드가 노출되기 때문).

표준 클레임:
- `iss`: 발급자
- `exp`: 만료
- `iat`: 발급시각
- `sub`: 주체(보통 사용자 ID)

나머지는 본인이 정의하는 커스텀 클레임이다. 너무 많이 넣으면 토큰이 커져서 매 요청마다 전송 비용이 생기므로, 자주 쓸 식별 정보만 넣는다.

## 토큰 발급 — `jwt.sign`

`jsonwebtoken` 라이브러리에서 `jwt.sign(payload, secret, options)` 한 줄로 토큰을 만든다.

```js
const jwt = require('jsonwebtoken');

const accessToken = jwt.sign(
  { sub: user.id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: '15m' }
);
```

- 1번째 인자: 페이로드
- 2번째 인자: 서명용 비밀키
- 3번째 인자: 옵션 (`expiresIn`은 토큰에 만료시각(`exp` 클레임)을 자동으로 박아준다)

## 액세스 토큰 + 리프레시 토큰 패턴

만료시간을 15분 정도로 짧게 두면 사용자가 매번 다시 로그인해야 하므로, 보통 두 종류 토큰을 같이 쓴다.

- **액세스 토큰**: 짧은 수명(예: 15분). 평소 API 호출에 사용.
- **리프레시 토큰**: 긴 수명(예: 2주). 액세스 토큰이 만료되면 이걸로 새 액세스 토큰을 받는다.

액세스 토큰을 짧게 유지하는 이유는 탈취돼도 피해 시간을 최소화하기 위함이고, 매번 로그인 안 시키려고 리프레시 토큰이라는 백업을 둔다. 검증 시점에 `jsonwebtoken`이 `exp` 클레임을 보고 자동으로 `TokenExpiredError`를 던지므로, 그 에러를 잡아 클라이언트가 재발급 요청을 보내게 한다.

### 리프레시 토큰의 형식과 저장

리프레시 토큰도 JWT로 만들어도 되고, 그냥 랜덤한 긴 문자열이어도 된다. 형식보다 중요한 것은 "서버가 이 토큰이 살아있는지 추적할 수 있어야 한다"는 점이다.

- 서버: DB에 저장해두고 발급/폐기를 관리한다. 그래야 로그아웃이나 탈취 시 무효화할 수 있다.
- 클라이언트: 보안상 httpOnly 쿠키에 담아 보내는 것이 일반적이다. JavaScript에서 접근 못 하므로 XSS로 훔쳐가기 어렵다.

### 리프레시 토큰 회전 (rotation)

리프레시 토큰이 수명이 길어서 탈취 시 위험이 크므로, 회전 패턴으로 방어한다.

- 리프레시 요청이 들어올 때마다 새 리프레시 토큰을 발급하고 이전 건 즉시 폐기한다.
- 공격자가 훔친 토큰을 한 번이라도 쓰면, 정상 사용자의 토큰은 이미 폐기된 상태가 되어 다음 요청에서 인증 실패가 난다. 이걸 감지해 해당 사용자의 모든 리프레시 토큰을 한꺼번에 무효화할 수 있다.

추가 방어선:
- httpOnly + Secure + SameSite 쿠키 저장
- DB에 사용자ID·발급시각·기기정보 같이 저장해 의심 패턴 감지 시 강제 로그아웃
- 액세스 토큰 자체를 짧게 유지

### 시크릿 분리

액세스 토큰용 시크릿과 리프레시 토큰용 시크릿은 분리하는 것이 좋다.

## 검증의 이중 안전장치

리프레시 토큰을 받았을 때는 두 단계로 검증한다.

1. `jwt.verify`로 서명·만료 확인
2. DB에 그 토큰이 아직 살아있는지 확인 (폐기되지 않았는지)

서명 검증만으로는 폐기된 토큰을 거를 수 없으므로 DB 조회를 함께 한다.

## 본질 정리

이 구조의 본질은 "각 토큰이 서로 다른 신뢰 수준을 갖는다"는 점이다. 액세스 토큰은 자주 쓰여서 노출 위험이 높은 대신 짧고 가볍게, 리프레시 토큰은 드물게 쓰이는 대신 길고 DB 추적이 가능한 형태로 둔다. 실무에서 빠뜨리면 안 되는 포인트는 세 가지다.

- 페이로드는 식별자만
- 서명 검증 + DB 검증의 이중 안전장치
- 토큰별로 시크릿 분리
