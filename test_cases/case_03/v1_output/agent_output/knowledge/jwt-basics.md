# JWT 기초

## 핵심 요약

- JWT는 "서버가 서명해서 발급한, 그 자체로 정보를 담은 문자열"이다.
- JWT는 **암호화가 아니라 서명**이다. 페이로드는 누구나 읽을 수 있다.
- 페이로드에는 사용자를 식별할 **최소 정보(식별자)만** 넣는다. 민감 정보 금지.

## 세션 방식과의 차이

| 구분 | 세션 | JWT |
|------|------|-----|
| 정보 위치 | 서버 메모리/DB | 토큰 자체 |
| 서버 저장 | 세션 ID 매핑 필요 | 서명만 검증, 별도 저장 불필요 |
| 확장성 | 서버 늘리면 세션 동기화 필요 | 동기화 부담 적음 |
| 무효화 | 서버에서 즉시 가능 | 발급된 토큰 무효화 어려움 |

JWT는 무상태(stateless) 검증이 가능해 서버 수평 확장에 유리하지만,
한번 발급된 토큰을 중간에 무효화하기 어렵다는 트레이드오프가 있다.

## 토큰 구조

```
헤더.페이로드.서명
```

- 세 부분이 점(`.`)으로 이어지고, 각 부분은 base64로 인코딩된다.
- base64는 인코딩일 뿐 암호화가 아니므로 jwt.io 같은 곳에 붙여넣으면
  페이로드 내용을 누구나 그대로 읽을 수 있다.
- 서명이 보장하는 것은 "발급 이후 변조되지 않았다"는 무결성이지
  "내용이 비밀"이라는 기밀성이 아니다.

## 페이로드에 넣을 것 / 넣지 말 것

- 넣을 것: `sub`(사용자 ID), `role` 등 자주 쓰는 최소 식별 정보
- 표준 클레임: `iss`(발급자), `exp`(만료), `iat`(발급시각)
- 절대 금지: 비밀번호, 주민번호 등 민감 정보 (페이로드는 노출되므로)
- 너무 많이 넣으면 토큰이 커져 매 요청 전송 비용이 늘어난다.
- 정말 숨겨야 하는 데이터는 JWE를 쓰거나, 서버에 두고 ID로만 조회한다.

## 발급 핵심

핵심은 `jwt.sign(payload, secret, options)` 한 줄이다.

```js
const accessToken = jwt.sign(
  { sub: user.id, role: user.role },
  process.env.JWT_SECRET,
  { expiresIn: '15m' }
);
```

- 1번째 인자: 페이로드
- 2번째 인자: 서명용 비밀키
- 3번째 인자: 옵션. `expiresIn`은 만료시각(`exp` 클레임)을 자동으로 박는다.
