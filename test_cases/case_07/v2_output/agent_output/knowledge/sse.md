# SSE (Server-Sent Events)

> 본 시나리오에서는 **검토 후 탈락한 대안**이다. 양방향 통신이 필요해 최종적으로 WebSocket으로 결정되었다.

SSE는 서버가 클라이언트로 데이터를 스트림 형태로 흘려보내는 방식이다. 별도 프로토콜이 아니라 HTTP 위에서 동작하며, 서버 → 클라이언트 **단방향**만 지원한다. 브라우저에서는 `EventSource` API로 바로 수신할 수 있다.

## 동작 방식

서버가 응답할 때 `Content-Type: text/event-stream`으로 설정하고 응답을 닫지 않은 채 유지한다. 열린 연결에 `data: ...\n\n` 형식으로 이벤트를 계속 흘려보내면, 클라이언트(`EventSource`)가 스트림으로 읽으며 `onmessage` 같은 핸들러로 처리한다. 연결이 끊기면 클라이언트가 자동으로 재연결을 시도하고, `Last-Event-ID` 헤더로 어디까지 받았는지 서버에 알려 끊김 복구도 자연스럽게 이루어진다.

## 장점

- HTTP를 그대로 쓰므로 프록시·로드밸런서 호환성이 좋다.
- 브라우저가 자동 재연결을 처리한다.
- 인증/CORS가 기존 HTTP 방식 그대로 적용된다.
- 구현이 단순하고 가볍다.

## 한계

서버 → 클라이언트 한 방향만 가능하다. 클라이언트가 서버로 무언가 보내야 한다면(예: '읽음' 표시) 별도의 HTTP 요청을 따로 보내야 하므로 "받는 건 SSE, 보내는 건 fetch/axios" 식의 하이브리드 구조가 된다. 양방향 상호작용이 빈번하면 이 분리가 번거로워진다.
