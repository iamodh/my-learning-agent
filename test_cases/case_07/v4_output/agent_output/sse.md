# SSE (Server-Sent Events)

SSE는 서버가 클라이언트로 데이터를 스트림 형태로 흘려보내는 방식이다. HTTP 응답을 닫지 않고 계속 열어둔 채로 이벤트를 흘려보내는 구조이며, 별도 프로토콜이 아니라 HTTP 위에서 동작한다. **본 알림 시스템에서는 검토 후 탈락한 대안**이다 (사유는 아래 한계 참고).

## 동작 방식

서버가 응답할 때 Content-Type을 `text/event-stream`으로 설정하고 응답을 닫지 않은 채로 유지한다. 그 열린 연결에 `data: ...\n\n` 형식으로 이벤트를 계속 흘려보내면, 클라이언트(`EventSource`)는 그것을 스트림으로 읽으면서 `onmessage` 같은 핸들러로 처리한다.

연결이 끊기면 클라이언트가 자동으로 재연결을 시도하고, `Last-Event-ID` 헤더로 어디까지 받았는지 서버에 알려줄 수도 있어 끊김 복구가 자연스럽다.

## 장점

- HTTP를 그대로 쓰므로 프록시·로드밸런서 호환성이 좋다.
- 자동 재연결을 브라우저가 알아서 해준다.
- 인증/CORS도 기존 HTTP 방식 그대로 적용된다.
- 브라우저의 `EventSource` API로 바로 받을 수 있어 구현이 단순하다.

## 한계 (본 시나리오에서 탈락한 이유)

SSE는 **서버 → 클라이언트 단방향**만 지원한다. 클라이언트가 서버로 무언가 보내야 하면 별도 HTTP 요청(예: fetch/axios POST)을 따로 보내야 한다. 알림 수신 외에 "읽음" 표시 같은 클라이언트 → 서버 액션이 필요한 시나리오에서는 "받는 건 SSE, 보내는 건 별도 HTTP"의 하이브리드 구조가 되며, 양방향 상호작용이 빈번할수록 이 분리가 번거롭다.

본 알림 시스템은 "읽음" 처리와 같은 양방향이 필요하므로 SSE 대신 WebSocket을 선택했다.
