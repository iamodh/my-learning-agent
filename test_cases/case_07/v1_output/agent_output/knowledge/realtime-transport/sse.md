# SSE (Server-Sent Events)

## 핵심
- 서버가 클라이언트로 데이터를 **스트림**으로 흘려보내는 **단방향**(서버 → 클라이언트) 방식.
- 별도 프로토콜이 아니라 **HTTP 위에서** 그대로 동작한다.
- 브라우저의 `EventSource` API로 바로 수신 → 구현이 단순하고 가볍다.

## 연결 유지 방식
- 서버가 응답 `Content-Type`을 `text/event-stream`으로 설정하고 응답을 닫지 않은 채 유지.
- 열린 연결에 `data: ...\n\n` 형식으로 이벤트를 계속 흘려보냄.
- 클라이언트(`EventSource`)는 스트림으로 읽으며 `onmessage` 핸들러로 처리.
- 연결이 끊기면 클라이언트가 자동 재연결, `Last-Event-ID` 헤더로 복구 지점 전달 가능.

## 장점
- HTTP 그대로라 프록시/로드밸런서 호환성 좋음.
- 자동 재연결을 브라우저가 처리.
- 인증/CORS를 기존 HTTP 방식 그대로 적용.

## 한계
- 단방향이라 클라이언트 → 서버 액션(예: '읽음' 표시)은 불가.
- 클라이언트가 서버로 보낼 일이 생기면 별도 HTTP POST를 따로 보내야 함 → "받는 건 SSE, 보내는 건 fetch/axios" 하이브리드 구조.
