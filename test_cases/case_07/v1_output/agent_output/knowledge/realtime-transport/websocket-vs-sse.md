# WebSocket vs SSE 선택 기준

## 핵심
- **단방향(서버 → 클라이언트)으로 충분** → SSE: 가볍고 HTTP 친화적.
- **양방향이 필요** → WebSocket: 클라이언트 → 서버 액션을 같은 연결로 처리.

## 비교

| 항목 | SSE | WebSocket |
|------|-----|-----------|
| 방향 | 단방향 (서버 → 클라이언트) | 양방향 |
| 프로토콜 | HTTP 그대로 | HTTP 핸드셰이크 후 `ws://` 전환 |
| 구현 난이도 | 단순 (`EventSource`) | 상대적으로 무거움 |
| 인프라 호환 | 프록시/LB 친화적 | 별도 고려 필요 |
| 재연결 | 브라우저 자동 | 라이브러리/직접 처리 |
| 클라이언트 → 서버 | 별도 HTTP 요청 필요 | 같은 연결로 가능 |

## 판단 가이드
- "서버가 일방적으로 던지기만 하면 되는" 단순 알림 → SSE가 깔끔.
- 알림 수신 + '읽음' 처리 + 채팅/입력 상태 등 확장 가능성 → 처음부터 WebSocket으로 통일하는 게 복잡도가 낮음.
