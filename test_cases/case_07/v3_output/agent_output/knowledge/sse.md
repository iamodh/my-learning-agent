# SSE (Server-Sent Events)

> 검토했으나 최종적으로 채택하지 않은 대안. 단방향(서버 → 클라이언트)이라서 클라이언트가 서버로 액션을 보내야 하는 시나리오와 맞지 않았다.

SSE는 서버가 클라이언트로 데이터를 스트림 형태로 흘려보내는 방식이다. HTTP 응답을 닫지 않고 계속 열어둔 채로 이벤트를 흘려보내는 구조라, 별도 프로토콜이 아니라 HTTP 위에서 그대로 동작한다.

## 동작 원리

서버는 응답의 `Content-Type`을 `text/event-stream`으로 설정하고 응답을 닫지 않은 채로 유지한다. 그 열린 연결에 `data: ...\n\n` 형식으로 이벤트를 계속 흘려보내면, 클라이언트(EventSource)는 그걸 스트림으로 읽으면서 `onmessage` 같은 핸들러로 처리한다.

연결이 끊기면 클라이언트가 자동으로 재연결을 시도하고, `Last-Event-ID` 헤더로 어디까지 받았는지 서버에 알려줄 수 있어 끊김 복구도 자연스럽다.

## 특성

- **단방향**: 서버 → 클라이언트만 가능. 클라이언트가 서버로 무언가 보내려면 별도 HTTP 요청을 따로 보내야 한다.
- **HTTP 친화적**: 별도 프로토콜이 아니라 HTTP 위에서 동작하므로 프록시·로드밸런서 호환성이 좋고, 인증/CORS도 기존 HTTP 방식 그대로 적용된다.
- **브라우저 내장 지원**: `EventSource` API로 바로 받을 수 있어 구현이 단순하다.
- **자동 재연결**: 브라우저가 알아서 재연결을 시도한다.

## WebSocket과의 비교

서버가 일방적으로 흘려보내기만 하면 되는 시나리오에서는 SSE가 깔끔한 선택지다. 다만 클라이언트 → 서버 액션이 필요한 순간 별도 HTTP 요청을 섞어야 하는 하이브리드 구조가 되며, 양방향 상호작용이 빈번하면 이 분리가 번거롭다.
