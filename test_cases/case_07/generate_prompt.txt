당신은 LLM과 사용자 간의 자연스러운 학습 대화를 합성하는 역할입니다.
사용자는 무언가를 만들고 싶어서 대화를 시작하고, 그 과정에서
기술 선택을 *번복*하는 경험을 합니다.

[대화 명세]
- 제작 의도: 실시간 알림 기능 구현
- 사용자 발화 턴 수: 정확히 10개
- 등장 상위 개념: WebSocket, SSE (2개)
- 등장 하위 개념: 없음 (각 개념을 별도로 다룸)
- 잡담: 없음
- 결정 번복: 있음 (WebSocket → SSE → WebSocket)
- 최종 결정: WebSocket
- 예시 요청: 최종 결정 후 WebSocket 구현 코드를 1회 요청

[대화 흐름 가이드]
아래는 *사용자 발화 10개*의 흐름입니다. 각 항목은 하나의 사용자 발화에
1:1로 대응합니다. 각 사용자 발화 직후에 AI 응답이 따라오며, AI 응답은
별도 턴으로 세지 않습니다.

사용자 발화 1: 만들고 싶은 것을 말함, 첫 번째 선택 (WebSocket) 가정
     (예: "서버에서 클라이언트로 실시간 알림 보내고 싶어. WebSocket 쓰면 되겠지?")
     → AI는 WebSocket이 적합함을 짧게 확인하고 기본 방향 제시

사용자 발화 2: WebSocket 본질 심화 질문
     (예: "WebSocket이 정확히 일반 HTTP랑 뭐가 다른 거야? 왜 실시간이 가능해?")
     → AI는 핸드셰이크 후 지속 연결 유지 방식 설명

사용자 발화 3: *대안*에 대해 질문 — 결정 흔들리기 시작
     (예: "근데 누가 SSE라는 것도 있다던데 그건 뭐야? 더 간단하다고 들었어")
     → AI는 SSE를 소개하고 WebSocket과의 차이 요약

사용자 발화 4: SSE의 장점 듣고 *번복 고려*
     (예: "오 그럼 SSE가 더 가벼우면 그걸 쓰는 게 낫지 않아? 어떤 차이가 있는데?")
     → AI는 SSE의 장단점 (HTTP 기반, 단방향) 구체적으로 설명

사용자 발화 5: SSE 상세 질문 — 번복 방향으로 기울어짐
     (예: "SSE는 그럼 어떻게 연결을 유지하는 거야? HTTP 위에서 어떻게 실시간이 돼?")
     → AI는 SSE의 EventStream과 자동 재연결 메커니즘 설명

사용자 발화 6: *한계 인식* — 결정 되돌리기 시작
     (예: "근데 알림은 받기만 하면 되는 게 아니라 클라이언트에서 '읽음' 표시도 서버로 보내야 하는데, SSE로도 그게 가능해?")
     → AI는 SSE의 단방향 한계를 설명 (별도 HTTP 요청 필요)

사용자 발화 7: *양방향 필요성* 깨닫고 WebSocket으로 복귀 결정
     (예: "아 그럼 결국 WebSocket이 맞겠네. 양방향이 필요한 시나리오니까")
     → AI는 결정의 타당성 확인, WebSocket으로 가는 게 적절함을 짧게 정리

사용자 발화 8: WebSocket 구현 코드 예시 요청
     (예: "그럼 WebSocket으로 가는 걸로 정했고, Express에서 어떻게 설정하는지 코드 좀 보여줄래?")
     → AI는 ws 또는 socket.io 사용한 간단한 서버 설정 코드 제공

사용자 발화 9: WebSocket 라이브러리 선택 질문
     (예: "ws랑 socket.io 중에 뭘 쓰는 게 좋아? 차이가 뭐야?")
     → AI는 두 라이브러리 비교 (기능 차이, 폴백 지원 등)

사용자 발화 10: 자기 정리 — 번복 경로와 최종 결정을 정리
      (예: "정리하면 SSE도 가벼워서 매력적이지만 양방향 필요해서 WebSocket으로 가는 거고, 라이브러리는 socket.io 쓰는 걸로")

각 사용자 발화 사이에 AI 응답이 들어가므로 총 20턴 정도의 대화가 됩니다.

[페르소나]

사용자:
- 백엔드 기본은 알지만 실시간 통신은 처음 도입
- 캐주얼한 한국어 ("~야", "~인가?", "음...")
- **이 케이스의 특징은 *기술 선택을 번복*하는 사고 과정입니다.
  처음 결정 → 대안 검토 → 한계 인식 → 원래 결정으로 복귀**
- **단순히 동작하는 해결책으로 만족하지 않습니다.
  심화 질문을 최소 3회 합니다 (특히 양 기술의 본질과 차이)**
- **추상적 설명이 잘 이해 안 될 때 "예시 들어줘", "코드로 보여줘" 같은
  구체화 요청을 합니다 (1회 — 최종 결정 후 WebSocket 코드)**
- **"오 됐다", "고마워" 같은 단순 감사로 대화를 끝내지 않습니다.
  마지막 사용자 발화는 *번복 경로와 최종 결정을 정리*하는 형태입니다
  (예: "정리하면 SSE도 매력적이지만 ~ 때문에 WebSocket으로 가는 거고")**

AI:
- 친절하지만 장황하지 않음 (한 응답 3-5문장)
- 각 기술의 본질과 한계를 *균형 있게* 설명
  (한쪽을 일방적으로 추천하지 않음)
- 사용자가 SSE로 기울 때도 SSE의 장점을 솔직히 설명
- 사용자가 양방향 필요성을 깨달았을 때 WebSocket의 적합성 확인
- 사용자가 코드 예시를 요청하면 그 자리에서 짧은 코드를 제공
- "이해됐어?" 같은 확인 질문은 하지 않음

[번복의 자연스러움]
이 케이스의 핵심은 *번복이 자연스러워야* 한다는 점입니다:
- 사용자 발화 1: WebSocket으로 시작 (직관적 첫 선택)
- 사용자 발화 3-4: SSE 검토 (가벼움이라는 장점으로 기울음)
- 사용자 발화 6: 양방향 필요성 깨달음 (한계 인식)
- 사용자 발화 7: WebSocket으로 복귀

각 번복 지점에 *명확한 이유*가 있어야 합니다.
"그냥 마음이 바뀜"이 아니라 *기술 비교에서 발견한 사실*이 번복의 근거.

[두 개념의 분리]
WebSocket과 SSE는 둘 다 *독립적으로 의미 있는* 기술입니다.
- WebSocket: 양방향 통신 프로토콜
- SSE: 서버 → 클라이언트 단방향 스트림 (HTTP 기반)
대화에서 둘이 *왜 다른지*가 명확히 드러나야 하며, 사용자가
양쪽 모두에 대해 *충분한 깊이*로 이해하는 흐름이어야 합니다.
