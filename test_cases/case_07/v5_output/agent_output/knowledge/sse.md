# SSE (Server-Sent Events)

SSE는 서버가 클라이언트로 데이터를 "스트림"으로 흘려보내는 방식이다. HTTP 응답을 끊지 않고 계속 열어둔 채로 이벤트를 흘려보내는 구조라, 별도 프로토콜이 아니라 HTTP 위에서 그대로 동작한다.

> 이 문서는 **검토 후 탈락한 대안**이다. 단방향 제약 때문에 이번 알림 시스템에는 채택하지 않았다(아래 "탈락 사유" 참고).

## 동작 방식

서버가 응답할 때 `Content-Type`을 `text/event-stream`으로 설정하고 응답을 닫지 않은 채 유지한다. 그 열린 연결에 `data: ...\n\n` 형식으로 이벤트를 계속 흘려보내면, 클라이언트(EventSource)는 그것을 스트림으로 읽으면서 `onmessage` 같은 핸들러로 처리한다.

연결이 끊기면 클라이언트가 자동으로 재연결을 시도하고, `Last-Event-ID` 헤더로 어디까지 받았는지 서버에 알려줄 수도 있어 끊김 복구가 자연스럽다.

## WebSocket과의 차이

- 별도 프로토콜이 아닌 HTTP 위에서 동작하므로 프록시·로드밸런서 호환성이 좋다.
- 자동 재연결을 브라우저가 알아서 해준다.
- 인증·CORS도 기존 HTTP 방식을 그대로 쓸 수 있다.
- **단방향**(서버 → 클라이언트)만 된다. 클라이언트가 서버로 뭔가 보내려면 별도 HTTP 요청을 따로 보내야 한다.

브라우저의 `EventSource` API로 바로 받을 수 있어 구현이 단순하고 가볍다는 평가를 받는다.

## 탈락 사유

이번 알림 시스템은 "받는 것"뿐 아니라 클라이언트의 **읽음** 처리(클라이언트 → 서버)가 필요했다. SSE 자체로는 클라이언트 → 서버 액션이 불가능해서 "받는 건 SSE, 보내는 건 fetch/axios" 식의 하이브리드 구조가 된다. 이 분리를 피하기 위해 양방향이 가능한 WebSocket으로 최종 결정했다.
