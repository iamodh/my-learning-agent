② RSC Payload — Server Component를 서버에서 실행해 React Tree를 바이너리 스트림(RSC Payload)으로 직렬화합니다. 여기에는 렌더 트리, Client Component 참조, 서버 측 데이터가 포함됩니다.

③ 데이터 패칭 — `fetch()`가 세 가지 모드로 분기됩니다.

④ HTML 생성 — RSC Payload를 `renderToString`으로 변환하고 Client JS 번들을 인라인 스크립트로 삽입해 최종 HTML을 만듭니다.

사용자: cdn 캐싱은 vercel이 해 주는 거야? 새로고침 하거나 link 이동했을 때 캐싱은 어떻게 돼?

CDN 캐싱 — Vercel만이 아닙니다.

Vercel은 CDN 캐싱을 자동으로 해줍니다. Next.js 앱을 배포하면 별도 설정 없이 Edge Network(전 세계 PoP)에 Full Route Cache가 배포됩니다.

하지만 직접 운영하는 CDN에서도 똑같이 동작합니다.

| 배포 환경 | CDN 캐싱 방식 |
| --- | --- |
| Vercel | 자동. `s-maxage`, `stale-while-revalidate` 헤더 자동 설정 |
| Cloudflare / AWS CloudFront | Next.js가 내보내는 `Cache-Control` 헤더를 CDN이 그대로 읽음 |
| self-hosted (Node / Docker) | CDN 없이 Origin 서버가 직접 응답, Full Route Cache는 서버 파일시스템에 저장 |

핵심은 Next.js가 헤더를 설정하고, CDN은 그 헤더를 따른다는 것입니다. Vercel은 그 과정을 가장 투명하게 처리해줄 뿐입니다.

새로고침 vs Link 이동 — 캐싱이 다르게 동작합니다.

두 경로의 핵심 차이는 캐시를 어느 레이어에서 읽느냐입니다. 새로고침은 서버(CDN 포함) 캐시를 활용하고, `<Link>` 이동은 브라우저 메모리 안의 Router Cache를 먼저 씁니다.

포인트만 짚으면 이렇습니다.

새로고침 — 브라우저가 Router Cache를 버리고 처음부터 HTTP 요청을 보냅니다. CDN → Full Route Cache → 서버 재렌더링 순서로 타고 올라가며, 응답은 완성된 HTML입니다.

Link 이동 — 브라우저 메모리에 있는 Router Cache를 먼저 씁니다. Cache miss 시에는 CDN이나 HTML 재생성 없이 Origin에서 RSC Payload만 받아와 클라이언트에서 합성합니다. 그래서 새로고침보다 훨씬 빠릅니다.

Router Cache 수명은 prefetch 여부와 라우트 종류로 갈립니다. 정적 라우트는 5분, 동적 라우트는 30초 동안 메모리에 살아 있습니다. 이 시간 안에 같은 페이지로 이동하면 서버 요청 자체가 발생하지 않습니다. 데이터를 즉시 최신화하려면 `router.refresh()`를 호출해야 합니다.

사용자: router cache랑 full route cache는 다른 거야? 그리고 data cache도 있지 않아?

AI: 맞습니다. Next.js에는 캐시 레이어가 총 4개 있고, 각각 위치·저장 내용·수명이 다릅니다.

네 레이어를 한 문장씩 구분하면 이렇습니다.

Request Memoization — React가 관리하는 가장 단명한 캐시입니다. 하나의 렌더 패스 안에서 같은 URL로 `fetch()`를 여러 번 호출해도 실제 요청은 딱 한 번만 나갑니다. 렌더가 끝나면 즉시 사라집니다.

Data Cache — `fetch()` 응답 자체를 서버 파일시스템에 영구 저장합니다. `revalidate: 60`이나 `revalidateTag('posts')` 같은 명시적 설정이 없으면 빌드 이후 계속 살아 있습니다. "데이터 원본"을 캐시하는 레이어입니다.

Full Route Cache — Data Cache를 사용해 렌더링한 결과물(HTML + RSC Payload)을 서버에 저장합니다. Data Cache가 무효화되면 Full Route Cache도 자동으로 무효화되는 의존 관계가 있습니다.

Router Cache — 브라우저 메모리에만 존재합니다. `<Link>` 이동 시 RSC Payload를 저장해두고, 같은 경로로 다시 이동할 때 서버 요청을 생략합니다. 새로고침하면 초기화됩니다.

요약하면, Data Cache와 Full Route Cache는 둘 다 서버에 있지만 저장하는 것이 다릅니다 — Data Cache는 `fetch()` 응답 원본, Full Route Cache는 그것으로 만든 렌더 결과물입니다. Router Cache는 완전히 별개로 브라우저 안에만 존재합니다.

사용자: revalidate됐을 때 prerender가 다시 발생하는데, 이 때 revalidate 대상은 뭐야? full route cache의 데이터? 아님 data cache의 데이터? 아님 둘 다?

AI: 둘 다입니다. 그런데 순서와 인과관계가 있습니다.

핵심은 revalidate의 직접 대상은 항상 Data Cache라는 점입니다.

`revalidateTag('posts')`나 `revalidate: 60`은 "이 `fetch()` 응답 캐시를 버려라"는 명령입니다. Full Route Cache는 Data Cache를 재료로 만든 결과물이기 때문에, 재료가 stale 처리되면 결과물도 자동으로 stale이 됩니다.

그리고 한 가지 중요한 타이밍이 있는데, prerender 재실행은 무효화 즉시 일어나지 않습니다. stale 처리만 해두고, 다음 요청이 들어올 때 그 요청을 처리하면서 ①②③을 다시 실행합니다. ISR의 stale-while-revalidate가 바로 이 동작입니다 — 첫 번째 요청은 stale한 응답을 받고, 그 사이 백그라운드에서 재생성이 완료되면 그 다음 요청부터 새 결과를 받게 됩니다.

옛 Pages Router의 ISR(`revalidate: 60`)은 백그라운드에서 페이지 HTML 자체를 재생성합니다 — 이건 "prerender"의 사촌이라 부를 수 있습니다. 하지만:

- 이 프로젝트는 ISR이 아닙니다 (페이지가 dynamic)
- Cache Components 모델에서 캐시 단위는 페이지가 아닌 데이터입니다 — 그래서 "re-prerender"가 아니라 "cache refill"이라 부르는 게 정확합니다.

| --- | --- | --- |

그래서 프로젝트가 dynamic 페이지 기반이라면 "re-prerender"라는 표현은 맞지 않고, "cache refill"이 정확한 표현입니다.

cache refill — Data Cache를 새로 채우는 것 (`fetch()` → 외부 API 호출 → 결과 저장)

dynamic render — 매 요청마다 일어나는 것으로, cache refill 이후 그 데이터로 RSC Payload와 HTML(홀 부분)을 생성

즉 "cache refill"과 "RSC Payload·HTML 생성"은 별개입니다. cache refill은 ① 데이터 패칭만을 가리키는 말이고, ②③은 cache refill 여부와 무관하게 dynamic 홀이라면 매 요청마다 항상 실행됩니다.

그러니까 정확한 구분은 이렇습니다.

| 구분 | re-prerender | dynamic render |
| --- | --- | --- |
| ① 데이터 패칭 | cache miss 시만 | cache miss 시만 |
| ② RSC Payload 생성 | 매번 | 매번 |
| ③ HTML 생성 | 매번 | 매번 |
| 결과를 Full Route Cache에 저장 | O | X |

둘의 차이는 ②③을 실행하느냐가 아니라, 그 결과를 Full Route Cache에 저장하느냐입니다. re-prerender는 새 HTML을 Full Route Cache에 다시 써넣고, dynamic render는 쓰지 않습니다.
