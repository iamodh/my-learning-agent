# 평가 케이스 자동 생성 워크플로우

## 프로젝트 목적
대화 기반으로 학습한 내용을 문서화하는 에이전트를 평가하기 위해,
테스트 케이스 입력 파일을 일괄 생성하는 작업이다.

각 테스트 케이스는 다음 3개 입력 파일로 구성된다:
- `spec.json`: 정답지 (생성 명세 + 채점용 정답)
- `generate_prompt.txt`: 대화 합성을 위한 LLM 프롬프트
- `dialogue.txt`: 위 프롬프트로 생성된 합성 대화 (빈 파일로 시작)

## 폴더 구조
프로젝트 루트에 다음 구조를 만든다. 이미 존재하면 덮어쓰지 말고 보존한다.

```
test_cases/
  case_01/
    spec.json
    generate_prompt.txt
    dialogue.txt
    v1_output/
      agent_output/
      score.json
    v2_output/
      agent_output/
      score.json
  case_02/
    ...
```

- 케이스 번호는 항상 두 자리 0 패딩 (`case_01`, `case_02`, ..., `case_10`)
- `v1_output/`, `v2_output/`, `agent_output/` 폴더는 *생성만* 해두고 내용은 비워둠
- `score.json`은 빈 객체 `{}`로 초기화
- `dialogue.txt`는 빈 파일로 생성

## 입력 형식

사용자는 케이스 하나당 다음 두 블록을 붙여넣는다.

### 블록 1: spec.json
JSON 코드 블록으로 시작. 첫 줄에 `// case_01/spec.json` 같은 경로 주석이 있을 수도 있고 없을 수도 있다. 경로 주석이 없으면 JSON 안의 `case_id` 필드를 보고 케이스 번호를 판정한다.

예시:
```json
{
  "case_id": 1,
  "case_name": "...",
  ...
}
```

### 블록 2: generate_prompt.txt
일반 텍스트. 코드 블록으로 감싸져 있을 수 있다. 감싸져 있으면 ```` ``` ```` 안의 내용만 추출한다.

## 출력 동작

입력을 받으면:

1. `spec.json`의 `case_id` 값으로 폴더명 결정 (`case_id: 1` → `case_01`)
2. 해당 폴더 구조를 만든다 (이미 있으면 건너뜀)
3. 다음 파일들을 쓴다:
   - `case_NN/spec.json` ← 받은 JSON 내용 그대로
   - `case_NN/generate_prompt.txt` ← 받은 프롬프트 텍스트 (코드 블록 마커 제거)
   - `case_NN/dialogue.txt` ← 빈 파일
   - `case_NN/v1_output/score.json` ← `{}`
   - `case_NN/v2_output/score.json` ← `{}`
   - `case_NN/v1_output/agent_output/.gitkeep` ← 빈 파일
   - `case_NN/v2_output/agent_output/.gitkeep` ← 빈 파일

4. 작업 완료 후 다음을 출력한다:
   - 생성된 파일 목록 (전체 경로)
   - JSON 유효성 검사 결과
   - 케이스 번호와 이름 확인

## 입력 처리 규칙

### 한 번에 여러 케이스가 들어올 때
사용자가 여러 케이스를 한 메시지에 붙여넣을 수 있다. 각 케이스는 spec.json + generate_prompt.txt 쌍으로 구성된다. 쌍의 경계는 다음 신호로 판정:

- 새 `case_id`가 등장하는 JSON 블록
- `// case_XX/...` 형식의 경로 주석
- 사용자가 명시한 구분선 (예: `---`, `### 케이스 N`)

각 쌍을 독립적으로 처리한다.

### spec.json 검증
파일 쓰기 전에 다음을 확인:

- 유효한 JSON 형식인가
- 다음 필수 필드가 모두 있는가:
  - `case_id` (정수)
  - `case_name` (문자열)
  - `생성_명세` (객체)
  - `정답지` (객체)
- `case_id`가 기존 케이스와 중복되지 않는가 (중복 시 사용자에게 확인 요청)

검증 실패 시 파일을 쓰지 않고 사용자에게 어떤 필드가 문제인지 알린다.

### generate_prompt.txt 추출
입력이 코드 블록(```)으로 감싸져 있으면 마커를 제거하고 내부 내용만 저장한다. 감싸져 있지 않으면 입력을 그대로 저장한다.

앞뒤 공백 줄은 한 줄까지만 보존한다.

### dialogue.txt 처리
기본은 빈 파일이다. 단, 사용자가 합성된 대화를 함께 붙여넣고 명시적으로 채워달라고 요청하면, 받은 대화를 **그대로** `dialogue.txt`에 쓴다 (내용을 임의로 지어내거나 보강하지 않는다). 코드 펜스가 깨져 들어온 경우(예: `javascriptconst ...`)에만 정상 마크다운 펜스(```)로 복원한다.

generate_prompt.txt 텍스트와 dialogue가 한 입력에 섞여 오면, 마지막 `[...]` 지시 블록까지가 프롬프트이고 첫 `사용자:` 줄부터가 대화다.

### dialogue.txt ↔ spec.json 대조 검증 (대화가 함께 제공된 경우)
대화를 파일에 쓰기 **전후로** 다음을 대조하고 결과를 보고한다. 불일치해도 파일은 쓰되(사용자가 준 원본 보존), 어떤 항목이 어긋났는지 반드시 알린다. spec.json이나 dialogue를 임의로 고쳐 일치시키지 않는다.

대조 항목 (spec에 해당 필드가 있을 때만):

- **사용자 발화 턴 수**: `생성_명세.사용자_발화_턴_수` == 대화에서 `^사용자:`로 시작하는 줄 수
- **잡담 턴**: `잡담_턴_수`/`잡담_턴_인덱스`가 대화의 잡담 발화와 일관되는가 (빈 배열이면 잡담 없음)
- **코드 스니펫**: `코드_스니펫_포함_턴` 배열 길이 == 대화의 코드 블록(``` 쌍) 개수. 턴 인덱스 위치도 대략 일치하는가
- **예시 요청**: `예시_요청_포함`이 true면 대화에 사용자의 명시적 코드/예시 요청 발화가 1회 있는가
- **상위/하위 개념**: `등장_상위_개념`, `등장_하위_개념`이 대화 주제 범위와 맞는가
- **결정 번복**: `결정_번복` 값이 대화 내 입장 번복 여부와 맞는가

대조 결과는 아래 보고 형식의 표로 출력한다.

## 하지 말 것

- `dialogue.txt`에 임의로 지어낸 내용을 채워넣지 않는다 (사용자가 대화를 제공하지 않으면 빈 파일, 제공하면 받은 원본 그대로만)
- `agent_output/` 폴더 안에 어떤 파일도 만들지 않는다 (`.gitkeep` 외)
- 기존 케이스 폴더를 덮어쓰지 않는다 (사용자가 명시적으로 요청하지 않는 한)
- spec.json 내용을 임의로 수정하지 않는다 (정렬, 포맷팅 포함)
- 한국어 필드명을 영문으로 바꾸지 않는다 (`생성_명세`, `정답지` 등 그대로 유지)

## 작업 후 보고 형식

각 케이스 처리 후 다음 형식으로 짧게 보고:

```
[OK] case_01 (단순 제작, 적은 개념)
  - spec.json: 유효 (정답지 8개 필드)
  - generate_prompt.txt: 1247자
  - 폴더 구조 생성됨

[OK] case_02 (단순 제작, 다중 개념)
  ...
```

문제가 있으면:

```
[ERROR] case_03
  - 사유: spec.json의 case_id 필드 누락
  - 처리: 파일 쓰기 건너뜀
```

대화가 함께 제공된 경우, dialogue ↔ spec 대조 표를 같이 출력한다:

```
[OK] case_01 (단순 제작, 적은 개념)
  - spec.json: 유효
  - dialogue.txt: 사용자 제공 원본 기록

  | 항목            | spec | 대화 | 판정 |
  |-----------------|------|------|------|
  | 사용자 발화 턴 수 | 4    | 4    | ✓    |
  | 코드 스니펫       | [3]  | 1개  | ✓    |
  | 잡담 턴          | 0    | 0    | ✓    |
```

불일치가 있으면 판정에 `✗`를 적고 어긋난 양(예: `+1`, `-2`)을 표기한다. 파일은 원본대로 쓰되 사용자에게 처리 방향(spec 수정/대화 보강/대기)을 묻는다.

여러 케이스를 처리했으면 마지막에 요약:

```
처리 완료: 3건 성공, 1건 실패
test_cases/ 폴더 트리:
  case_01/ ...
```

## 평가 실행 사이클

위 워크플로우는 *케이스 입력 파일 생성*(step 0)까지다. 입력이 갖춰진 뒤
실제 평가는 다음 사이클로 진행한다.

```
1단계 실행:  prompts/vN.md + case_NN/dialogue.txt
                → 학습 에이전트 → case_NN/vN_output/agent_output/
             (spec.json 미투입 — 정답지를 에이전트에 절대 노출하지 않음)

2단계 채점:  vN_output/agent_output  ↔  case_NN/spec.json 의 정답지
                → case_NN/vN_output/score.json

반복:        v1/v2 score 비교 → 깨진 규칙(L규칙·특이_검증)을
             프롬프트로 명문화 → prompts/v(N+1).md → 재실행
```

- 1단계 입력은 `prompts/vN.md`와 해당 케이스의 `dialogue.txt` **둘뿐**이다.
  `spec.json`은 정답지이므로 에이전트 입력에 넣지 않는다.
- 2단계에서 `agent_output/`을 `정답지`(`예상_결과_문서`,
  `예상_참조_매핑`, `특이_검증` 등)와 대조한 결과를 `score.json`에 쓴다.
  즉 `score.json`의 `{}`는 *초기값*이고, 채점 단계에서 채워진다.
- v1/v2는 동일 케이스를 서로 다른 프롬프트 버전으로 돌린 결과이며,
  점수 차이가 프롬프트 변경을 정당화한다.

**범위 주의**: 위 "`agent_output/` 폴더는 생성만·내용 비워둠"(폴더 구조
절)과 "`agent_output/` 안에 어떤 파일도 만들지 않는다"(하지 말 것 절)는
*케이스 생성 단계* 한정 규칙이다. **1단계 실행·2단계 채점 단계에는
적용되지 않으며**, 그 단계에서는 `agent_output/`과 `score.json`을
정상적으로 채운다.

## 참고

- 정답지의 한국어 필드명(`예상_결과_문서`, `예상_지식_문서` 등)은 의도된 것이다. 채점 단계에서 LLM이 한국어 키로 채점하는 게 자연스러워서 이렇게 설계됐다.
- `dialogue.txt`가 빈 파일인 이유는, 사용자가 `generate_prompt.txt`를 별도 LLM 세션에 넣어 대화를 합성한 뒤 결과를 직접 복사해 채우기 때문이다.
- `v1_output`, `v2_output`은 에이전트 프롬프트의 버전별 결과를 비교하기 위한 폴더다. 지금은 비어 있지만 나중에 채워진다.
