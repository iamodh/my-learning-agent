# 학습 에이전트 프롬프트 v3

첨부한 대화를 기반으로 내가 어떤 지식을 학습하려고 했고, 이 지식을 통해 무엇을 만들려고 했는지 지식 문서와 결과 문서로 분류해줘.

## 1. 산출물 범위 (대화에서 실제로 확립된 것만)

- 산출물은 대화에서 명시적으로 다뤄진 내용으로 **엄격히 한정한다**. 대화에 없던 설정값·옵션·대안 비교·모범 사례는 추가하지 마라.
- 얕게 언급된 주제는 얕게 남긴다. 추측으로 상세화하거나 일반 지식을 끌어와 보강하지 마라.
- 잡담·여담은 어떤 문서의 본문에도 유입시키지 마라.
- 설치 명령, 설정값, API 사용 흐름, 라이브러리 옵션, 운영 모범 사례는 대화에서 직접 다뤘을 때만 쓴다.

## 2. 결과 문서 (조건부 산출물)

결과 문서는 **항상 만드는 산출물이 아니다.** 먼저 제작 의도 게이트를 통과해야 한다.

### 2.1 제작 의도 게이트

다음이 모두 확인될 때만 결과 문서를 만든다.

- 사용자가 만들 대상의 이름·형태·기능 중 하나를 명확히 말했다.
- 그 대상에 어떤 지식을 적용하려는지 대화에서 확인됐다.
- 단순 호기심·가정·잡담·스치는 언급이 아니라 실제 구현 방향이 정해졌다.

위 조건을 통과하지 못하면 **지식 문서만 만들고 결과 문서·참조 매핑·참조 이유를 생성하지 마라.** 이때 지식 문서도 대화에서 명시된 수준까지만 짧게 유지한다.

### 2.2 결과 문서 분리 기준

결과 문서는 제작 대상 단위로 나눈다. **문서 수 최소화는 서로 다른 결과 산출물을 하나로 합치라는 뜻이 아니다.**

- 서로 다른 기능·산출물·사용 흐름이 있고, 각 대상이 다른 고유 지식을 사용하면 **결과 문서를 분리한다**.
- 여러 결과가 같은 지식을 공유하면, 지식 문서는 하나만 만들고 각 결과 문서가 그 공통 지식을 각자의 이유로 참조한다.
- 하나의 제작 대상 안에서 단계만 나뉜 경우에는 결과 문서를 하나로 유지한다.
- 결과 문서에는 내가 만들려는 것이 무엇인지, 어떤 지식을 참조하는지, 왜 그 지식이 필요한지 명시한다.

### 2.3 최종 결정과 탈락 대안

- 참조는 **최종 결정의 근거가 된 지식만** 포함한다.
- 검토 끝에 탈락한 대안은 지식 문서로 남길 수 있으나, 그것이 탈락했다는 사실을 본문에 명시한다.
- 탈락한 대안은 결과 문서의 참조 매핑에 넣지 마라.
- 사용자가 선택을 번복한 경우, 결정 과정을 시간순으로 길게 재현하지 마라. 한 줄짜리 요약("초기에는 X를 검토했으나 최종적으로 Y로 결정") 정도로만 남긴다.

## 3. 지식 문서

지식 문서는 하나의 개념 범위여야 한다. 핵심 내용은 앞에 배치하고, 첫 문단은 코드가 아니라 정의 텍스트로 시작한다. 이해가 어려운 부분, 특히 사용자가 명시적으로 예시·코드를 요청한 개념에는 예시나 코드 스니펫을 넣어도 된다.

### 3.1 독립 문서 vs 섹션 기준

문서 수 최소화는 **하위 지식 개념의 과분할을 막기 위한 규칙**이다. 독립 구현 책임을 갖는 개념까지 부모 문서에 흡수하지 마라.

다음 조건 중 하나라도 강하면 독립 지식 문서로 둔다.

- 사용자가 그 개념을 별도 학습 대상으로 반복해 물었다.
- 결과 문서에서 그 개념을 별도 선택·구성 요소·구현 책임으로 사용한다.
- 같은 부모 도구 안의 단순 옵션 설명이 아니라, 저장 방식·인증 방식·배포 방식처럼 결과 설계에서 독립 결정을 담당한다.
- 서로 다른 결과 문서에서 재사용될 수 있다.

다음 경우에는 새 파일이 아니라 부모 지식 문서의 `##` 섹션으로 통합한다.

- 부모 개념을 떼면 단독으로 의미가 성립하지 않는다.
- 사용자가 부모 개념을 학습하다 마주친 세부 측면이다.
- 옵션·제약·검증 규칙·만료 처리·키 네이밍·프리플라이트처럼 부모 개념의 작동 조건이나 하위 메커니즘이다.
- 한 결과 문서에서만 쓰이는 부수 개념이고 독립 구현 책임이 약하다.

판단이 애매하면 먼저 "이 개념이 결과 설계에서 독립 책임을 갖는가, 아니면 부모 도구의 옵션인가?"를 확인한다. 독립 책임이면 문서로 두고, 옵션·하위 메커니즘이면 섹션으로 둔다.

### 3.2 중복 방지

- 같은 지식을 결과별로 중복 문서화하지 마라.
- 공통 지식은 한 번만 문서화하고, 여러 결과 문서가 같은 지식 문서를 참조할 수 있다.
- 두 번째 이상의 관련 지식 파일을 만들기 전에, 기존 문서의 섹션으로 표현 가능한지 확인한다.

## 4. 결과 문서와 지식 문서의 역할 분리

명확한 제작 의도가 있으면 결과 문서는 지식 문서와 분리한다.

- 결과 문서: 무엇을 만들려 하는가, 어떤 지식을 왜 참조하는가.
- 지식 문서: 개념 정의, 핵심 원리, 대화에서 다룬 예시·코드.

결과 문서가 필요할 정도로 제작 의도가 명확하다면, 결과 내용을 지식 문서의 섹션으로 흡수하지 마라. 반대로 제작 의도 게이트를 통과하지 못하면 결과 문서를 만들지 마라.

## 5. 길이와 형식

- 모든 문서는 간결한 길이를 유지한다.
- 핵심을 앞에 두고, 곁가지 설명은 뒤로 미루거나 잘라낸다.
- 사용자가 명시적으로 예시·코드를 요청한 개념에는 구체적 예시나 코드 스니펫을 포함한다.
- 코드 스니펫은 대화에서 다룬 수준을 넘어서 과하게 확장하지 않는다.

## 6. 폴더 구조와 README

문서를 모두 생성한 뒤 인과 관계에 따라 폴더 구조를 만든다.

- 폴더 깊이는 3 이하로 제한한다. 다만 이는 상한일 뿐, **인과 관계를 표현하는 가장 얕은 구조**를 택한다.
- 파일이 1~2개뿐인 분류 폴더는 만들지 마라. 그 정도는 상위 폴더에 평탄하게 둔다.
- 카테고리를 만들기 위해 카테고리를 만들지 마라. 묶을 실체가 있을 때만 폴더로 분리한다.
- 루트 디렉토리에 문서 배치 구조를 표시한 README를 만든다.
- README는 배치 안내용 메타 문서다. README에 새 지식·새 결과·대화에 없던 내용을 추가하지 마라.
- 결과/지식 문서 수를 판단할 때 README는 메타 문서로 취급한다.

## 7. 자가 점검 (출력 직전)

문서를 확정하기 전에 스스로 확인한다.

1. 결과 문서를 만들었다면, 제작 의도 게이트를 통과했는가? 아니면 결과 문서·참조 매핑·참조 이유를 제거한다.
2. 서로 다른 제작 대상이 하나의 결과 문서로 합쳐지지 않았는가? 기능·산출물·사용 흐름이 분리되면 결과 문서도 분리한다.
3. 공통 지식이 결과별로 중복 문서화되지 않았는가? 공통 지식은 하나만 두고 여러 결과가 참조하게 한다.
4. 독립 구현 책임을 갖는 지식이 부모 문서 섹션으로 흡수되지 않았는가?
5. 부모 개념의 옵션·제약·하위 메커니즘이 불필요하게 별도 지식 문서로 분리되지 않았는가?
6. 참조 매핑에 탈락한 대안이 들어가 있지 않은가?
7. 대화에 없던 설정값·옵션·대안 비교·구체 API 흐름이 본문에 섞이지 않았는가?
8. README가 새 지식이나 새 결과를 확장하지 않고 배치 안내만 하는가?
