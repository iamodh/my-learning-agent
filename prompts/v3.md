# 학습 에이전트 프롬프트 v3

첨부한 대화를 기반으로 내가 어떤 지식을 학습하려고 했고, 이 지식을 통해 무엇을 만드려고 했는지 **지식 문서**와 **결과 문서**로 분류해줘.

지식 문서와 결과 문서는 **서로 독립된 산출물 축**이다. 한쪽의 분할/통합 기준을 다른 쪽에 적용하지 마라.

## 1. 산출물 범위 (대화에서 실제로 확립된 것만)

- 산출물은 대화에서 명시적으로 다뤄진 내용으로 **엄격히 한정한다**. 대화에 없던 설정값·옵션·대안 비교·모범 사례는 어떤 문서에도 추가하지 마라.
- 얕게 언급된 주제는 얕게 남긴다. 추측으로 상세화하거나 일반 지식을 끌어와 보강하지 마라.
- 잡담·여담은 어떤 문서의 본문에도 유입시키지 마라.

### 1.1 환각 보강 금지 (구체 항목)

다음은 대화에서 **직접 다루지 않았다면** 어떤 문서에도 쓰지 않는다. "관행적 기본값"·"일반 지식"이라는 이유로도 끌어오지 마라.

- 설치 명령, 패키지 매니저 명령
- 호스트·포트·경로 등 설정값과 환경 변수
- API 호출 옵션, 함수 시그니처의 미언급 인자
- 인증 방식·토큰 발급 흐름의 디테일
- 라이브러리 권장 사용법·모범 사례·튜닝 팁
- 대안 라이브러리 비교나 "왜 이걸 안 쓰는지"

대화에 등장한 키워드가 있더라도, 그 키워드의 **구체 동작**이 대화에서 설명되지 않았다면 동작 설명을 지어내지 마라. 키워드 수준에서 멈춘다.

## 2. 지식 문서

지식 문서는 하나의 개념 범위여야 해. 핵심 내용은 앞에 배치하고, **첫 문단은 코드가 아니라 정의 텍스트로 시작**해줘. 이해가 어려운 부분(특히 사용자가 명시적으로 예시·코드를 요청한 개념)에는 예시나 코드 스니펫을 넣어도 돼.

### 2.1 분할/통합 — 양방향 기준

지식 문서를 별도 파일로 분리할지, 부모 문서의 `##` 섹션으로 통합할지는 **양방향 질문을 모두** 거쳐서 정한다. 한쪽만 보고 결정하지 마라.

**[A] 섹션 통합 조건** — 다음 중 하나라도 해당하면 부모 문서의 섹션으로 둔다.

- 부모 개념을 떼면 단독으로 의미가 성립하지 않는다.
- 부모 도구·라이브러리의 **옵션·플래그·제약·검증 규칙·하위 동작 양식**이다.
- 같은 부모 영역의 서로 다른 측면(예: 같은 도구의 비동기 처리 양식과 에러 처리 양식)이라 함께 묶이는 게 자연스럽다.
- 한 결과 문서에서만 부수적으로 쓰이며 다른 맥락에서 재사용 여지가 없다.

**[B] 독립 문서 조건** — 다음 중 **하나라도 강하게** 해당하면 별도 지식 문서로 분리한다. [A]보다 [B]가 우선이다.

- 사용자가 그 개념을 **별도 학습 대상**으로 다루었다 (반복 질문, 별도 설명 요청, 개념 자체에 대한 호기심).
- 부모 도구의 옵션이 아니라 **독립 결정**을 담당한다 (예: 저장 방식 선택, 인증 방식 선택, 배포 방식 선택, 통신 방식 선택).
- 결과 문서에서 **독립 구성 요소·구현 책임**으로 등장한다.
- 다른 결과·다른 맥락에서도 **재사용 가능**한 일반 지식이다.

**판단 순서**

1. 후보 개념마다 [B]를 먼저 점검한다. [B]가 강하면 → 독립 문서로 분리.
2. [B]가 약하면 [A]를 점검한다. [A]가 해당하면 → 섹션으로 통합.
3. 둘 다 약해서 정말 애매하면 → 통합한다 (지식 문서 한정 기본 편향).

기본 편향 "애매하면 통합"은 **지식 문서 분할에만** 적용된다. 결과 문서에는 적용하지 마라 (§3 참고).

### 2.2 탈락한 대안의 처리

사용자가 검토했다가 채택하지 않은 대안은 지식 문서로 남길 수 있으나, 그것이 *탈락했다는 사실*을 본문에 명시한다. 결과 문서의 참조 매핑에는 넣지 않는다(§3.4 참고).

## 3. 결과 문서 (조건부 산출물, 지식 문서와 독립)

결과 문서의 생성·분할 기준은 §2의 지식 문서 통합 편향과 **독립적으로** 적용한다. "문서 수 최소화"는 지식 문서 한정 규칙이며, 결과 문서를 합치는 근거로 쓰지 마라.

### 3.1 생성 게이트 (체크리스트)

결과 문서는 **항상 만드는 산출물이 아니다.** 다음 세 항목이 **모두** 충족될 때만 만든다. 하나라도 빠지면 결과 문서·참조 매핑·참조 이유를 만들지 않는다.

1. **대상 명시**: 사용자가 만들 대상의 **이름이나 형태**를 한 번 이상 명확히 지목했다. 스치는 기술 키워드 언급은 해당하지 않는다.
2. **사용 방식 확인**: 그 대상에 어떤 지식을 어떻게 적용할지 대화에서 확인됐다.
3. **구현 방향 결정**: 단순 호기심·가정·"~라면 어떨까" 비교만이 아니라, **실제로 만들겠다는 방향**이 정해졌다.

게이트를 통과하지 못하면 지식 문서만 만든다. 이때 지식 문서는 대화에서 명시된 수준까지만 짧게 유지한다 (보강 금지).

### 3.2 결과 문서 N개 분할 — "한 결과당 한 문서"

게이트를 통과한 뒤, 서로 다른 제작 대상이 **N개** 식별되면 결과 문서도 **N개로 분리**해서 만든다. 하나로 합치지 마라.

**다른 대상으로 보는 기준** — 다음 중 하나라도 해당하면 별개 결과 문서로 분리한다.

- 서로 다른 **기능·산출물 단위** (예: 업로드 기능 vs 다운로드 기능, 클라이언트 vs 서버).
- 서로 다른 **사용 흐름·진입점**.
- **고유 지식 구성**이 다르다 (참조 매핑 집합이 다르면 다른 대상).

같은 시스템의 일부라는 이유로 묶지 마라. 공통 지식이 있다는 것은 한 결과 문서로 합치는 근거가 되지 않는다.

### 3.3 결과 문서와 지식 문서의 역할 분리

결과 문서가 생기는 경우, 결과 내용을 지식 문서 안에 섞어 넣지 마라.

- 지식 문서: 개념 정의, 동작 원리, 사용 방법.
- 결과 문서: 무엇을 만들었는가, 어떤 지식이 사용되었는가, 왜 그 지식이 사용되었는가.

결과 문서가 없는 경우(게이트 미통과)에만 결과적 맥락이 지식 문서에 자연스럽게 흘러들어도 된다 — 단, 이 경우에도 §1.1 환각 보강 금지는 그대로 지킨다.

### 3.4 참조 매핑과 참조 이유

- 결과 문서에는 **무엇을 만들었는가**, **어떤 지식이 사용되었는가**(참조), **왜 그 지식이 사용되었는가**(참조 이유)를 적는다.
- 참조는 **최종 결정의 근거가 된 지식만** 포함한다. 검토 끝에 탈락한 대안은 참조 매핑에 넣지 마라.
- 사용자가 선택을 번복한 경우, 결정 과정을 시간순으로 재현하지 마라. 한 줄짜리 요약("초기에는 X를 검토했으나 최종적으로 Y로 결정") 정도로만 남긴다.
- **공통 지식**이 여러 결과 문서에서 쓰이면, 지식 문서는 하나만 두고 각 결과 문서가 **자기 맥락의 참조 이유**를 별도로 서술한다. 같은 지식을 복제하거나 결과별로 나누지 마라.
- 참조 매핑의 항목은 **독립된 지식 문서 파일**을 가리켜야 한다. 다른 지식 문서의 *섹션*을 참조 매핑 항목으로 가리키는 형태는 허용하지 않는다. 그런 일이 발생한다면 그 섹션은 §2.1 [B] 기준에 따라 별도 지식 문서로 분리해야 한다.

## 4. 길이와 형식

- 모든 문서는 간결한 길이를 유지한다. 핵심을 앞에 두고, 곁가지 설명은 뒤로 미루거나 잘라낸다.
- 대화에서 얕게 다뤄진 주제는 얕게 남긴다. 분량을 채우기 위해 보강하지 마라.

## 5. 폴더 구조

문서를 모두 생성한 뒤 인과 관계에 따라 폴더 구조를 만든다.

- 폴더 깊이는 3 이하로 제한한다. 다만 이는 상한일 뿐, **인과 관계를 표현하는 가장 얕은 구조**를 택한다.
- **파일이 1~2개뿐인 분류 폴더는 만들지 마라.** 그 정도는 상위 폴더에 평탄하게 둔다.
- 카테고리를 만들기 위해 카테고리를 만들지 마라. 묶을 실체가 있을 때만 폴더로 분리한다.

마지막으로 각 문서를 알맞은 폴더에 배치하고, 루트 디렉토리에 문서 배치 구조를 표시한 README를 만들어줘.

## 6. 자가 점검 (출력 직전 — 모두 통과해야 확정)

### 6.1 결과 문서 축 (D1·D4)

1. 결과 문서 생성 게이트(§3.1) 세 항목이 모두 충족됐는가? 하나라도 빠지면 결과 문서·참조 매핑·참조 이유를 **제거**한다.
2. 게이트를 통과한 대상이 N개인데 결과 문서를 1개로 합치지 않았는가? 합쳤다면 N개로 **분리**한다.
3. 결과 구현 내용이 지식 문서 본문에 섞여 있지 않은가? 섞여 있다면 결과 문서로 **이전**한다.
4. 결과 문서의 참조 매핑 각 항목이 **별도 지식 문서 파일**을 가리키는가? 어떤 항목이 다른 지식 문서의 섹션을 가리키고 있다면, 그 섹션을 별도 지식 문서로 **분리**한다.

### 6.2 지식 문서 축 (D2)

5. 각 섹션에 대해 자문: **"이 섹션은 독립 결정·독립 구현 책임을 담당하는가? 다른 맥락에서 재사용되는가?"** YES면 §2.1 [B]에 따라 별도 문서로 **분리**한다.
6. 각 문서 쌍에 대해 자문: **"이 두 문서는 같은 부모 영역의 서로 다른 측면인가? 부모 도구의 옵션·제약인가?"** YES면 §2.1 [A]에 따라 한 문서로 **통합**한다.
7. 참조 매핑에 탈락한 대안이 들어가 있지 않은가? 들어가 있으면 **제거**하고 본문에 한 줄 요약만 남긴다.

### 6.3 환각·범위 (D3·§1)

8. 대화에 없던 설치 명령·설정값·API 옵션·인증 디테일·모범 사례가 본문에 섞이지 않았는가? 섞였다면 **삭제**한다.
9. 잡담·여담이 본문에 유입되지 않았는가?
10. 사용자가 예시·코드를 요청한 개념에 코드 스니펫이 들어가 있는가? (요청이 있었다면 충족 필요)

### 6.4 구조 (§5)

11. 파일이 1~2개뿐인 분류 폴더가 있지 않은가? 있으면 상위로 **평탄화**한다.
12. README가 루트에 있고 모든 문서의 배치 구조를 반영하는가?
