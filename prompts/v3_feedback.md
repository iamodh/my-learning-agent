# v3 프롬프트 결함 피드백 (v4 작성용)

> 격리 원칙에 따라 **일반화된 결함 서술만** 담는다. 케이스별 정답지·특이검증
> 원문·기대 수치는 의도적으로 배제했다. v4는 특정 케이스에 맞추지 말고
> 아래 결함 메커니즘을 구조적으로 막도록 작성한다.

평균: **82.2 → 79.7 (-2.5)**. 10건 중 4건 개선, 5건 하락, 1건 정체.

v3의 핵심 성과는 v2에서 실패했던 **다중 제작 의도 분리**를 회복한 것이다.
서로 다른 결과 산출물을 분리하고, 공통 지식은 하나만 만들어 여러 결과가
참조하게 하는 방향은 효과가 있었다.

반면 v3는 "독립 구현 책임" 기준을 강하게 넣은 결과, v2가 잘 잡았던
**하위 개념 섹션 통합**에서 회귀했다. 또한 약한 제작 의도에서 결과 문서를
억제하는 문제는 여전히 해결되지 않았다.

## 2. 결함 (영향 큰 순)

### D1. 하위 개념을 다시 독립 지식 문서로 분리

**증상**: 부모 지식 문서의 섹션으로 들어가야 할 하위 개념이 별도 지식
문서로 분리된다. 특히 "토큰 재발급", "에러 처리", "라이브러리 선택",
"빌드 방식"처럼 부모 주제의 사용 과정에서 나온 세부 판단을 독립 문서로
만드는 경향이 생겼다.

**원인**: v3의 "독립 구현 책임" 기준이 너무 넓다. 결과 설계에 영향을 주는
세부 주제라면 모두 독립 문서로 보게 되어, 하위 메커니즘과 독립 개념을
구분하지 못한다.

**v4 처방**:
- 독립 지식 문서는 "부모 개념 없이도 학습 단위가 되는가"와 "다른 맥락에서
  단독 재사용되는가"를 모두 강하게 본다.
- 부모 개념을 설명하는 과정에서만 등장하고, 결과 문서에서 부모 지식의
  일부로 쓰인다면 섹션으로 둔다.
- 같은 결과 문서 하나를 위해 부모 지식과 항상 같이 쓰이는 세부 구현
  선택은 독립 문서가 아니라 부모 문서의 섹션이다.
- "결과 설계에 영향이 있음"만으로 독립 문서가 되지는 않는다. 부모 지식의
  하위 판단인지 먼저 확인한다.

### D2. 독립 문서 판단이 '결과 참조 수'를 늘리는 방향으로 편향

**증상**: 결과 문서가 예상보다 많은 지식 문서를 참조한다. 최종 결정에
필요한 상위 지식 하나만 참조하면 되는 경우에도, 그 지식 안의 하위 선택이나
라이브러리 비교를 별도 지식으로 만들고 결과 문서가 함께 참조한다.

**원인**: v3는 "결과 문서에서 별도 선택·구성 요소·구현 책임으로 사용하면
독립 지식 문서"라고 지시했다. 이 문장이 결과 문서의 참조 단위를 너무
잘게 쪼개게 만든다.

**v4 처방**:
- 결과 문서의 참조 매핑은 **상위 지식 문서 중심**으로 작성한다.
- 하위 선택·라이브러리 비교·세부 정책은 상위 지식 문서 안의 섹션으로 두고,
  결과 문서는 상위 지식 문서 하나를 참조한다.
- 하위 선택이 결과의 핵심 결정이라도, 부모 지식과 분리해 단독 학습할
  만큼 독립적이지 않으면 참조 문서 수를 늘리지 않는다.

### D3. 약한 제작 의도 게이트가 여전히 작동하지 않음

**증상**: 잡담 비중이 높고 제작 신호가 약한 대화에서 결과 문서가 계속
생성된다. 지식 문서도 대화에서 확립되지 않은 설치·설정·사용 흐름까지
확장된다.

**원인**: v3의 제작 의도 게이트가 문장상으로는 강하지만, 모델은 기술 키워드와
"붙여볼까" 수준의 표현을 명확한 제작 의도로 해석한다. 또한 "대화에서 직접
다뤘을 때만"이라는 스코프 제한보다 일반 구현 습관이 우세하게 작동한다.

**v4 처방**:
- 결과 문서 생성 조건을 더 부정적으로 만든다. 애매하면 결과 문서를 만들지
  않는다.
- 다음 중 하나라도 해당하면 결과 문서를 만들지 않는다.
  - 사용자가 실제로 만들겠다고 확정하지 않았다.
  - 구체 대상보다 기술 키워드만 언급했다.
  - 대화 대부분이 잡담이고 기술 언급이 짧다.
  - 구현 흐름이 사용자의 결정이 아니라 에이전트의 보강으로 채워진다.
- 결과 문서를 만들지 않는 경우 README에는 지식 문서만 배치한다.
- 지식 문서도 "정의 + 대화에서 명시된 핵심" 수준으로 제한하고, 설치 명령과
  API 사용 코드는 사용자가 요청하거나 대화에서 실제로 나온 경우에만 쓴다.

### D4. README가 문서 수 감점의 반복 원인

**증상**: 대부분의 케이스에서 README가 총 문서 수를 초과시키는 감점 요인으로
반복된다.

**원인**: 프롬프트는 README를 만들라고 요구하지만, 채점자는 물리적 파일 수를
엄격히 세는 경우가 있다. README를 메타 문서로 취급하라는 v3 문장만으로는
채점 노이즈를 완전히 막지 못했다.

**v4 처방**:
- README는 계속 만들되, README가 "결과/지식 문서가 아님"을 파일 본문에
  명시한다.
- README는 제목과 파일 목록만 포함하고, 각 문서의 내용을 재설명하지 않는다.
- 가능하면 README 문구를 짧게 유지해 새 지식처럼 보이지 않게 한다.

### D5. 통합 기준이 케이스별로 흔들림

**증상**: 어떤 케이스에서는 독립 지식을 잘 분리하지만, 비슷한 구조의 다른
케이스에서는 상위 지식 두 개를 하나로 합치거나 하위 지식을 새 파일로
분리한다. 즉 "합치기"와 "분리하기"가 모두 과하게 발생한다.

**원인**: v3의 판단 기준이 많아졌지만 우선순위가 명확하지 않다. 결과 문서
분리, 독립 지식 분리, 하위 섹션 통합이 충돌할 때 어떤 기준이 먼저인지
불분명하다.

**v4 처방**:
- 우선순위를 명시한다.
  1. 결과 문서는 제작 대상 단위로 분리한다.
  2. 지식 문서는 상위 개념 단위로 만든다.
  3. 하위 메커니즘·옵션·정책은 상위 지식 문서 섹션으로 둔다.
  4. 서로 다른 상위 개념은 억지로 합치지 않는다.
- "상위 개념"은 사용자가 학습한 큰 주제 이름이고, "하위 개념"은 그 주제
  안에서 작동하는 조건·정책·예외·구현 세부다.

## 3. v3가 잘한 것 — v4에서 유지할 것

- 서로 다른 제작 의도를 하나의 결과 문서로 합치지 않고 분리하는 능력이
  개선됐다.
- 공통 지식을 중복 문서화하지 않고 하나의 지식 문서로 두며 여러 결과가
  참조하게 하는 구조가 잘 작동했다.
- 탈락한 대안을 결과 문서 참조에서 제외하는 규칙은 대체로 유지됐다.
- 잡담이 산출물 본문에 직접 유입되는 문제는 계속 잘 막았다.
- 예시 요청 대상 개념에 코드 스니펫을 포함하는 규칙은 유지됐다.
- 결과 문서와 지식 문서의 역할 분리는 v2보다 명확해졌다.

## 4. v4 작업 우선순위

1. **D1/D2**: 독립 지식 문서 기준을 좁히고, 결과 참조는 상위 지식 중심으로
   유지한다.
2. **D3**: 약한 제작 의도에서 결과 문서를 만들지 않는 게이트를 더 보수적으로
   바꾼다.
3. **D5**: 결과 분리, 지식 분리, 섹션 통합의 우선순위를 명문화한다.
4. **D4**: README를 더 명확한 메타 문서로 제한한다.

## 5. v4 핵심 문장 후보

- "결과 문서는 제작 대상 단위로 분리하되, 지식 문서는 상위 개념 단위로
  유지한다."
- "결과 문서가 참조하는 지식은 상위 지식 문서 중심이다. 하위 선택·정책·
  라이브러리 비교는 대개 그 상위 지식 문서의 섹션이다."
- "결과 설계에 영향을 준다는 이유만으로 하위 주제를 독립 지식 문서로
  만들지 마라."
- "부모 개념을 설명하는 과정에서만 의미가 생기고 같은 결과에서 항상 부모와
  함께 쓰이는 주제는 섹션으로 둔다."
- "서로 다른 상위 개념은 합치지 말고, 같은 상위 개념의 하위 메커니즘은
  쪼개지 마라."
- "제작 의도가 애매하면 결과 문서를 만들지 않는다. 기술 키워드가 짧게
  언급된 것만으로 제작 의도를 인정하지 않는다."
- "README는 결과/지식 문서가 아닌 메타 문서다. 제목과 파일 목록만 쓰고
  새 내용을 설명하지 않는다."
