# 학습 에이전트 프롬프트 v5

첨부한 대화를 기반으로 내가 어떤 지식을 학습하려고 했고, 이 지식을 통해 무엇을 만드려고 했는지 **지식 문서**와 **결과 문서**로 분류해줘.

지식 문서와 결과 문서는 **서로 독립된 산출물 축**이다. 한쪽의 분할/통합 기준을 다른 쪽에 적용하지 마라.

## 1. 산출물 범위 (대화에서 실제로 확립된 것만)

- 산출물은 대화에서 명시적으로 다뤄진 내용으로 **엄격히 한정한다**. 대화에 없던 설정값·옵션·대안 비교·모범 사례는 어떤 문서에도 추가하지 마라.
- 얕게 언급된 주제는 얕게 남긴다. 추측으로 상세화하거나 일반 지식을 끌어와 보강하지 마라.
- 잡담·여담은 어떤 문서의 본문에도 유입시키지 마라.

### 1.1 환각 보강 금지 (구체 항목)

다음은 대화에서 **직접 다루지 않았다면** 어떤 문서에도 쓰지 않는다. "관행적 기본값"·"일반 지식"이라는 이유로도 끌어오지 마라.

- 설치 명령, 패키지 매니저 명령
- 호스트·포트·경로 등 설정값과 환경 변수
- API 호출 옵션, 함수 시그니처의 미언급 인자
- 인증 방식·토큰 발급 흐름의 디테일
- 라이브러리 권장 사용법·모범 사례·튜닝 팁
- 대안 라이브러리 비교나 "왜 이걸 안 쓰는지"

대화에 등장한 키워드가 있더라도, 그 키워드의 **구체 동작**이 대화에서 설명되지 않았다면 동작 설명을 지어내지 마라. 키워드 수준에서 멈춘다.

## 2. 지식 문서

지식 문서는 하나의 개념 범위여야 해. 핵심 내용은 앞에 배치하고, **첫 문단은 코드가 아니라 정의 텍스트로 시작**해줘. 이해가 어려운 부분(특히 사용자가 명시적으로 예시·코드를 요청한 개념)에는 예시나 코드 스니펫을 넣어도 돼.

### 2.1 분할/통합 — 통합 우선, 그다음 분리

지식 문서의 분할/통합 판단은 **순서가 중요하다**. 먼저 통합 후보인지를 보고, 통합 조건 *어디에도* 해당하지 않을 때만 분리 여부를 본다. 분리는 강한 관찰 신호가 있을 때만 한다.

**[1단계 — 통합 우선 점검]** 다음 중 **하나라도** 해당하면 부모 문서의 `##` 섹션으로 통합한다. 별도 파일로 만들지 마라.

- 부모 개념을 떼면 단독으로 의미가 성립하지 않는다.
- 부모 도구·라이브러리의 **옵션·플래그·제약·검증 규칙·하위 동작 양식**이다 (예: 어떤 라이브러리의 메서드, 어떤 형식의 파서 동작, 어떤 알고리즘의 매개변수).
- 같은 부모 영역의 서로 다른 **측면**이다 (예: 같은 도구에서의 비동기 처리 양식과 에러 처리 양식, 같은 인증 방식에서의 토큰 발급과 만료 관리).
- **한 코드 흐름에서 *동시에* 다뤄지는 두 측면이다** — 한 함수·한 모듈·한 라우터·한 미들웨어·한 리졸버·한 라이브러리 호출 안에서 *같이 작동*하는 측면들. 의미상 서로 다른 영역처럼 보여도, *같은 코드 자리에서 함께 처리되어야* 한다면 한 문서의 섹션으로 묶는다. 예: 같은 리졸버 함수 안의 비동기 처리와 에러 처리, 같은 미들웨어 안의 입력 검증과 응답 변환, 같은 핸들러 안의 인증과 권한 확인, 같은 라이브러리 호출 안의 페이로드 구성과 옵션 설정. **이 신호는 의미적 거리감보다 우선한다**: 두 주제가 의미상 충분히 독립적으로 *보이더라도*, 실제 코드에서 같이 다뤄지면 같은 영역이다.
- 한 결과 문서를 이해하기 위한 **보조·맥락 개념**이다 (왜 기본값으로 안 되는지, 왜 이걸 도입했는지, 어떤 한계가 있는지). 이런 내용은 관련 지식 문서의 도입 섹션 또는 결과 문서 본문에 녹인다.
- **추상 개념 + 그 구체 구현**의 관계다 (예: 결제 게이트웨이 일반 개념 + 특정 PG의 사용법, 캐시 일반 개념 + 특정 라이브러리 호출, 인증 방식 + 특정 라이브러리 사용 패턴, 통신 방식 + 특정 라이브러리 선택). **분리하면 추상 문서에 코드 예시가 빠지고 코드 문서는 더 좁은 주제만 다루게 된다.** 한 문서의 섹션으로 묶는다.

**[2단계 — 분리 점검]** 위 통합 조건에 *어디에도* 해당하지 않을 때만 분리 여부를 본다. 다음 **필수 신호**가 명시적으로 관찰돼야 별도 문서로 분리한다.

- **필수 신호 — 사용자가 그 개념을 별도 학습 대상으로 다루었다**: 반복 질문, 별도 설명 요청, 그 개념 자체에 대한 호기심, 다른 개념과 비교하기 위해 분리해 학습한 흔적 중 하나가 대화에 명시적으로 드러나야 한다.
- "이론적 재사용 가능성"은 신호로 치지 않는다. "독립 결정을 담당할 수 있다"는 추상 판단도 신호로 치지 않는다. 관찰 가능한 학습 행위만 신호다.

필수 신호가 없으면 분리하지 않는다. 통합 조건과 분리 신호 모두 약하면 **통합한다** (지식 문서 한정 기본 편향: "애매하면 통합").

**자가 점검 — 분리 후보가 떠오를 때마다 자문하라**

1. "이 개념이 [1단계] 통합 조건 어딘가에 해당하는가? 특히 — 결과 코드에서 같은 자리에 같이 등장하는가?" — YES면 즉시 통합, 더 보지 마라.
2. "사용자가 이 개념을 별도 학습 대상으로 다룬 *관찰 가능한* 신호가 있는가?" — NO면 통합.
3. 1·2를 통과해 분리를 정당화한 뒤에도, 분리 결과로 부모 문서에서 정의·코드 예시가 빠지지 않는지 확인. 빠진다면 분리를 철회하고 섹션으로 둔다.

기본 편향 "애매하면 통합"은 **지식 문서 분할에만** 적용된다. 결과 문서에는 적용하지 마라 (§3 참고).

### 2.2 탈락한 대안의 처리

사용자가 검토했다가 채택하지 않은 대안은 지식 문서로 남길 수 있으나, 그것이 *탈락했다는 사실*을 본문에 명시한다. 결과 문서의 참조 매핑에는 넣지 않는다(§3.4 참고).

## 3. 결과 문서 (조건부 산출물, 지식 문서와 독립)

결과 문서의 생성·분할 기준은 §2의 지식 문서 통합 편향과 **독립적으로** 적용한다. "문서 수 최소화"는 지식 문서 한정 규칙이며, 결과 문서를 합치는 근거로 쓰지 마라.

### 3.1 생성 게이트 — 부정 신호 강/약 분류, 그다음 양성 신호

결과 문서는 **항상 만드는 산출물이 아니다.** 게이트를 두 단계로 본다. 부정 신호를 강/약으로 분류해 차단 여부를 보고, 통과하면 양성 신호 셋을 본다.

**[부정 신호 게이트 — 강/약 분류]**

부정 신호는 두 부류다. **강한 신호가 하나라도 해당하거나, 약한 신호가 두 개 이상 동시에 해당**하면 결과 문서·참조 매핑·참조 이유를 만들지 않는다. 약한 신호 하나만으로는 차단하지 않는다.

**강한 신호** (1개 해당 시 차단):

- 대화의 **절반 이상이 잡담·여담**이다.
- 사용자가 **학습 자체에 더 관심**이 있어 보이고, 만들기는 미래의 가능성("나중에 필요해지면", "이런 게 있나 보네"), 가설, 가정 수준으로만 언급된다.

**약한 신호** (2개 이상 동시 해당 시 차단):

- 사용자가 만들 대상의 **구체 형태**(어떤 인터페이스, 어떤 입력·출력, 언제 호출되는지, 어떤 흐름에 들어가는지)를 한 번도 묘사하지 않았다. 단순 이름 언급("X 기능 좀 붙여야지", "Y 라이브러리 써볼까")만 있다.
- 만들기 결정이 "쓰면 되겠네", "오케이" 수준에 그치고 **어떻게 적용할지의 맥락**(어느 흐름의 어느 시점에 들어가는지)이 비어 있다.

**왜 강/약 분리인가**: 약한 신호 하나만으로 차단하면, 명확히 학습 + 만들기를 모두 진행한 대화(예: "에러가 났는데 이 라이브러리로 해결하겠다"는 흐름)에서 구체 인터페이스가 묘사되지 않았다는 이유만으로 결과 문서가 누락된다. 강한 신호(잡담 절대 다수, 학습-주된-관심)는 한 가지로도 결정적이지만, 구체성·맥락 부족 같은 약한 신호는 *두 가지가 동시*에 나타날 때만 결정적이다.

**[양성 신호 체크리스트]** 부정 신호 게이트를 통과한 경우에만 본다. 다음 셋이 **모두** 충족돼야 결과 문서를 만든다.

1. **대상 명시**: 만들 대상의 **이름과 형태**가 한 번 이상 명확히 지목됐다 (스치는 키워드 아님).
2. **사용 방식 확인**: 그 대상에 어떤 지식을 어떻게 적용할지 대화에서 확인됐다.
3. **구현 방향 결정**: 단순 호기심·가정·"~라면 어떨까" 비교가 아니라, **실제로 만들겠다는 방향**이 정해졌다.

게이트 어느 단계에서든 통과하지 못하면 지식 문서만 만든다. 이때 지식 문서는 대화에서 명시된 수준까지만 짧게 유지한다 (보강 금지).

**borderline 처리 원칙**: 양성 신호 셋이 모두 분명하면 만들고, 양성 신호조차 분명하지 않으면 만들지 않는다. 부정 신호가 강/약 합쳐 임계에 못 미치면서 양성 신호 셋이 다 명확하다면, *만든다*가 기본이다 — 결과 의도가 보이는 대화에서 결과 문서를 누락하지 않는다.

### 3.2 결과 문서 N개 분할 — "한 결과당 한 문서"

게이트를 통과한 뒤, 서로 다른 제작 대상이 **N개** 식별되면 결과 문서도 **N개로 분리**해서 만든다. 하나로 합치지 마라.

**다른 대상으로 보는 기준** — 다음 중 하나라도 해당하면 별개 결과 문서로 분리한다.

- 서로 다른 **기능·산출물 단위** (예: 업로드 기능 vs 다운로드 기능, 클라이언트 vs 서버).
- 서로 다른 **사용 흐름·진입점**.
- **고유 지식 구성**이 다르다 (참조 매핑 집합이 다르면 다른 대상).

같은 시스템의 일부라는 이유로 묶지 마라. 공통 지식이 있다는 것은 한 결과 문서로 합치는 근거가 되지 않는다.

### 3.3 결과 문서와 지식 문서의 역할 분리

결과 문서가 생기는 경우, 결과 내용을 지식 문서 안에 섞어 넣지 마라. 그러나 그 반대 방향 — **지식의 코드 예시를 결과 문서로만 보내는 형태** — 도 금지다.

- 지식 문서: 개념 정의, 동작 원리, 사용 방법, **그 개념의 코드 예시**.
- 결과 문서: 무엇을 만들었는가, 어떤 지식이 사용되었는가, 왜 그 지식이 사용되었는가, **최종 구현 코드**.

사용자가 예시·코드를 요청한 개념의 코드 스니펫은 **지식 문서 쪽**에 두는 것이 기본이다. 지식 문서가 정의 텍스트만 담고 코드는 결과 문서에만 있는 형태는 피한다 — 그 개념의 사용 방법은 그 개념의 지식 문서에 있어야 한다.

결과 문서가 없는 경우(게이트 미통과)에만 결과적 맥락이 지식 문서에 자연스럽게 흘러들어도 된다 — 단, 이 경우에도 §1.1 환각 보강 금지는 그대로 지킨다.

### 3.4 참조 매핑과 참조 이유

- 결과 문서에는 **무엇을 만들었는가**, **어떤 지식이 사용되었는가**(참조), **왜 그 지식이 사용되었는가**(참조 이유)를 적는다.
- 참조 매핑은 **결과 작성에 직접 필수적인 지식만** 포함한다. 단순 맥락 이해를 돕는 보조 지식·"비교용 대안"·"검토는 했으나 결과에 안 쓰인 개념"은 매핑에 넣지 마라. 그런 보조 개념은 §2.1 [1단계]에 따라 다른 문서의 섹션이어야 한다.
- 검토 끝에 탈락한 대안은 참조 매핑에 넣지 마라. 사용자가 선택을 번복한 경우, 결정 과정을 시간순으로 재현하지 마라. 한 줄짜리 요약("초기에는 X를 검토했으나 최종적으로 Y로 결정") 정도로만 남긴다.
- **공통 지식**이 여러 결과 문서에서 쓰이면, 지식 문서는 하나만 두고 각 결과 문서가 **자기 맥락의 참조 이유**를 별도로 서술한다. 같은 지식을 복제하거나 결과별로 나누지 마라.
- 참조 매핑의 항목은 **독립된 지식 문서 파일**을 가리켜야 한다. 다른 지식 문서의 *섹션*을 참조 매핑 항목으로 가리키는 형태는 허용하지 않는다. 단, 이를 이유로 그 섹션을 §2.1 [2단계] 필수 신호 없이 분리하지도 마라. 신호가 약하면 그 항목은 참조 매핑에서 빼고 본문 내 맥락으로 녹인다.

## 4. 길이와 형식

- 모든 문서는 간결한 길이를 유지한다. 핵심을 앞에 두고, 곁가지 설명은 뒤로 미루거나 잘라낸다.
- 대화에서 얕게 다뤄진 주제는 얕게 남긴다. 분량을 채우기 위해 보강하지 마라.

## 5. 폴더 구조

문서를 모두 생성한 뒤 인과 관계에 따라 폴더 구조를 만든다.

- 폴더 깊이는 3 이하로 제한한다. 다만 이는 상한일 뿐, **인과 관계를 표현하는 가장 얕은 구조**를 택한다.
- **파일이 1~2개뿐인 분류 폴더는 만들지 마라.** 그 정도는 상위 폴더에 평탄하게 둔다.
- 카테고리를 만들기 위해 카테고리를 만들지 마라. 묶을 실체가 있을 때만 폴더로 분리한다.

마지막으로 각 문서를 알맞은 폴더에 배치하고, 루트 디렉토리에 문서 배치 구조를 표시한 README를 만들어줘.

## 6. 자가 점검 (출력 직전 — 모두 통과해야 확정)

### 6.1 결과 문서 게이트 (강/약 부정 신호)

1. **부정 신호 게이트** — 강한 신호 1개라도 해당하는가? (잡담 절반 이상 / 학습이 주된 관심) → 차단. 결과 문서·참조 매핑·참조 이유 제거.
2. 강한 신호가 없다면, **약한 신호가 2개 이상 동시에** 해당하는가? (구체 형태 묘사 없음 + 적용 맥락 비어 있음) → 차단.
3. 위 두 단계를 모두 통과한 뒤, 양성 신호 셋(대상 명시·사용 방식·구현 방향)이 *모두* 충족되는가? 하나라도 빠지면 결과 문서를 만들지 않는다.
4. 양성 신호 셋이 분명한데 부정 신호가 임계 미만이라면 결과 문서를 *만든다*. 약한 신호 하나만으로 누락하지 마라.

### 6.2 결과 문서 축

5. 게이트를 통과한 대상이 N개인데 결과 문서를 1개로 합치지 않았는가? 합쳤다면 N개로 **분리**한다.
6. 결과 구현 내용이 지식 문서 본문에 섞여 있지 않은가? 섞여 있다면 결과 문서로 **이전**한다.
7. 사용자가 예시·코드를 요청한 개념의 코드 스니펫이 *지식 문서 쪽*에 있는가? 결과 문서에만 코드가 있고 지식 문서가 정의 텍스트뿐이라면 코드 일부를 **지식 문서로 이동**한다.

### 6.3 지식 문서 축 (분리 과다 방지 + 같은 코드 자리 통합)

8. 각 별도 문서 쌍에 대해 자문: **"이 두 개념이 같은 함수·라우터·미들웨어·리졸버·라이브러리 호출 안에서 같이 작동하는가?"** YES면 한 문서로 **통합**한다. 의미적으로 달라 보이더라도 코드 자리가 같으면 같은 영역이다.
9. 각 별도 문서에 대해 자문: **"이 개념이 §2.1 [1단계] 다른 통합 조건(부모 도구 옵션 / 같은 부모 영역 측면 / 보조 맥락 / 추상+구체) 어딘가에 해당하는가?"** YES면 부모 문서의 섹션으로 **흡수**한다.
10. 별도 문서가 정당화되는가? **"사용자가 이 개념을 별도 학습 대상으로 다룬 관찰 가능한 신호가 있는가?"** NO면 통합한다.
11. **추상 개념과 그 구체 구현**이 별도 문서로 쪼개져 있지 않은가? 쪼개져 있다면 한 문서의 섹션으로 **통합**한다.
12. 참조 매핑에 **단순 맥락 이해를 위한 보조 지식**이 들어가 있지 않은가? 들어가 있다면 그 항목은 매핑에서 **제거**하고 해당 보조 지식은 본문 섹션으로 흡수한다.
13. 참조 매핑에 탈락한 대안이 들어가 있지 않은가? 들어가 있으면 **제거**하고 본문에 한 줄 요약만 남긴다.

### 6.4 환각·범위
