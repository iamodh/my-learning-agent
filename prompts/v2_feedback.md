# v2 프롬프트 결함 피드백 (v3 작성용)

> 격리 원칙에 따라 **일반화된 결함 서술만** 담는다. 케이스별 정답지·특이검증
> 원문·기대 수치는 의도적으로 배제했다. v3는 특정 케이스에 맞추지 말고
> 아래 결함 메커니즘을 구조적으로 막도록 작성한다.

평균: **70.7 → 82.2 (+11.5)**. 10건 중 7건 개선, 3건 하락.

v2의 핵심 성과는 v1의 주된 실패 원인이던 **하위 개념 과분할**을 크게
줄인 것이다. 특히 하위 개념이 부모 지식 문서의 섹션으로 들어가야 하는
케이스에서 점수가 크게 올랐다. 반면 v2는 "문서 수 최소화" 편향이 강해져
**독립 문서로 남아야 할 지식·결과 문서까지 합치는 회귀**가 생겼다.

## 2. 결함 (영향 큰 순)

### D1. 독립 결과 문서 병합 — 최우선

**증상**: 대화에 서로 다른 제작 의도가 여러 개 있을 때, 결과 문서를
각각 만들지 않고 하나의 통합 결과 문서로 합친다. 그 결과 문서 수가
부족해지고, 결과별 참조 매핑과 참조 이유도 흐려진다.

**원인**: v2의 "문서 수 최소화"와 "애매하면 합친다"가 지식 문서 분할에는
효과적이었지만, 결과 문서에도 과하게 적용됐다. 서로 다른 산출물 단위까지
하나의 결과 문서로 합쳐 버린다.

**v3 처방**:
- 결과 문서에는 별도 분할 기준을 둔다.
- 사용자가 만들려는 대상이 기능·산출물·사용 흐름 단위로 분리되고, 각
  대상이 서로 다른 고유 지식을 필요로 하면 **결과 문서는 분리한다**.
- 공통 지식은 중복 문서로 만들지 말고 하나의 지식 문서로 유지하되, 각
  결과 문서가 그 공통 지식을 각자의 이유로 참조하게 한다.
- "문서 수 최소화"는 주로 지식 문서의 하위 개념 통합에 적용하며, 독립
  결과 문서 병합의 근거로 쓰지 않는다.

### D2. 독립 지식까지 부모 문서에 흡수

**증상**: 하위 개념은 잘 통합하지만, 독립 지식 문서로 남아야 하는 개념도
부모 문서의 섹션으로 흡수한다. 이 경우 결과 문서의 참조 매핑에서 필요한
지식 하나가 사라진다.

**원인**: v2의 "부모 개념을 떼면 단독으로 의미가 성립하지 않으면 섹션"
규칙은 맞지만, 반대 조건이 약하다. 즉 어떤 개념이 독립 지식으로 남아야
하는지 판단하는 기준이 부족하다.

**v3 처방**:
- 다음 조건 중 하나라도 강하면 독립 지식 문서로 둔다.
  - 사용자가 그 개념을 별도 학습 대상으로 반복해 물었다.
  - 결과 문서에서 그 개념을 별도 선택·구성 요소·구현 책임으로 사용한다.
  - 같은 부모 도구 안의 옵션 설명이 아니라, 저장 방식·인증 방식·배포
    방식처럼 결과 설계에서 독립 결정을 담당한다.
  - 다른 결과에서도 재사용 가능한 개념이다.
- 단순 옵션·제약·검증 규칙은 부모 문서 섹션으로 통합한다.
- "독립 구현 책임"과 "부모 도구의 사용 옵션"을 구분하라.

### D3. 약한 제작 의도에서 결과 문서 생성 지속

**증상**: 잡담 비중이 높고 제작 의도가 약한 대화에서도 결과 문서를 만든다.
또한 대화에서 확립되지 않은 구현 세부를 보강해 지식 문서를 과하게
상세화한다.

**원인**: v2에 결과 문서 생성 조건은 추가됐지만, "약한 신호"를 기각하는
판단 절차가 충분히 강하지 않다. 에이전트가 기술 키워드와 스치는 제작
언급을 실제 제작 의도로 과해석한다.

**v3 처방**:
- 결과 문서를 만들기 전 "제작 의도 게이트"를 통과시킨다.
- 다음이 모두 없으면 결과 문서를 만들지 않는다.
  - 사용자가 만들 대상의 이름이나 형태를 명확히 말했다.
  - 그 대상에 적용할 지식의 사용 방식이 대화에서 확인됐다.
  - 단순 호기심·가정·잡담이 아니라 실제 구현 방향이 정해졌다.
- 게이트를 통과하지 못하면 지식 문서만 만들고, 지식 문서도 대화에서
  명시된 수준까지만 짧게 유지한다.
- 설치 명령, 설정값, 구체 API 사용 흐름은 대화에서 직접 다뤘을 때만 쓴다.

### D4. 결과 문서와 지식 문서 역할의 과통합

**증상**: 단순 케이스에서 결과 문서가 별도 파일로 존재하지 않고 지식
문서 안에 결과 구현 내용이 섞인다. 문서 수는 맞아 보여도 결과↔지식
참조 매핑과 참조 이유가 약해진다.

**원인**: v2가 폴더/문서 수 최소화를 강조하면서, 결과 문서와 지식 문서의
역할 분리를 약화시켰다.

**v3 처방**:
- 명확한 제작 의도가 있으면 결과 문서는 지식 문서와 분리한다.
- 결과 문서는 "무엇을 만들었는가/만들려 하는가"와 "어떤 지식을 왜
  참조하는가"를 담당한다.
- 지식 문서는 개념 설명과 예시를 담당한다.
- 결과 문서가 필요할 정도로 제작 의도가 명확하다면, 결과 내용을 지식
  문서의 섹션으로 흡수하지 않는다.

### D5. README 보조 문서가 총 문서 수 판정에 노이즈

**증상**: 거의 모든 v2 산출물에 README가 추가되어, 채점자가 총 문서 수를
엄격히 셀 때 감점 요인이 된다. 기존 프롬프트가 README 생성을 요구하므로
완전한 오류는 아니지만, "문서 수" 기대와 충돌한다.

**v3 처방**:
- README는 배치 안내용 메타 문서임을 명시하고, 결과/지식 문서 수에는
  포함하지 않는다는 의도를 프롬프트나 채점 기준 중 한쪽에 명확히 둔다.
- 프롬프트에서는 README 본문이 새 지식·결과 내용을 확장하지 않게 제한한다.

## 3. v2가 잘한 것 — v3에서 유지할 것

- 하위 개념을 독립 파일로 과분할하지 않고 부모 지식 문서 섹션으로
  통합하는 능력이 크게 개선됐다.
- 탈락한 대안을 결과 문서 참조에서 제외하는 처리가 좋아졌다.
- 결정 번복을 장황한 시간순 기록이 아니라 최종 결정 중심으로 정리했다.
- 잡담이 결과/지식 문서 본문에 유입되는 문제는 대체로 잘 막았다.
- 예시 요청 대상 개념에 코드 스니펫을 포함하는 규칙은 유지됐다.
- 지식 문서 첫 문단이 코드가 아니라 정의 텍스트로 시작하는 성질도 유지됐다.

## 4. v3 작업 우선순위

1. **D1**: 다중 제작 의도는 결과 문서로 분리하고, 공통 지식은 하나만
   유지하되 각 결과가 별도 참조 이유를 갖게 한다.
2. **D2**: 독립 지식과 하위 옵션/섹션의 구분 기준을 보강한다.
3. **D3**: 약한 제작 의도에서 결과 문서 생성을 막는 게이트를 강화한다.
4. **D4**: 명확한 제작 의도가 있는 경우 결과 문서와 지식 문서의 역할을
   분리한다.
5. **D5**: README를 메타 문서로 제한해 총 문서 수·본문 내용 확장 노이즈를
   줄인다.

## 5. v3 핵심 문장 후보

아래 문장들은 v3에 직접 반영할 수 있는 형태의 일반화된 규칙이다.

- "문서 수 최소화는 하위 지식 개념의 과분할을 막기 위한 규칙이며, 서로
  다른 결과 산출물을 하나로 합치라는 뜻이 아니다."
- "결과 문서는 제작 대상 단위로 나눈다. 서로 다른 기능·산출물·사용
  흐름이 있고 각 대상이 다른 고유 지식을 사용하면 결과 문서를 분리한다."
- "공통 지식은 한 번만 문서화하되, 여러 결과 문서가 각자의 이유로 같은
  지식 문서를 참조할 수 있다."
- "독립 구현 책임을 갖는 개념은 지식 문서로 남기고, 부모 도구의 옵션·제약·
  검증 규칙은 부모 문서의 섹션으로 둔다."
- "결과 문서 생성 전, 사용자의 제작 의도가 명확한지 확인한다. 제작 대상과
  적용 방식이 대화에서 확인되지 않으면 결과 문서를 만들지 않는다."
- "대화에서 직접 다루지 않은 설치 명령, 설정값, API 사용 흐름, 모범 사례는
  지식 문서에 추가하지 않는다."
