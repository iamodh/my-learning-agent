# v2 프롬프트 결함 피드백 (v3 작성용)

> 격리 원칙에 따라 **일반화된 결함 서술만** 담는다. 케이스별 정답지·특이검증
> 원문·기대 수치는 의도적으로 배제했다. v3는 특정 케이스에 맞추지 말고
> 아래 결함 메커니즘을 구조적으로 막도록 작성한다.

## 1. v1 ↔ v2 채점 요약 (Claude 채점)

| 케이스 | v1 점수 | v1 판정 | v2 점수 | v2 판정 | 변화 |
|--------|--------|---------|---------|---------|------|
| 01 | 1.00 | PASS | 0.65 | FAIL | **-0.35 회귀** |
| 02 | 0.60 | FAIL | 0.55 | FAIL | -0.05 |
| 03 | 0.62 | FAIL | 1.00 | PASS | +0.38 대폭 개선 |
| 04 | 0.45 | FAIL | 0.97 | PASS | +0.52 대폭 개선 |
| 05 | 0.40 | FAIL | 0.45 | FAIL | +0.05 |
| 06 | 0.68 | FAIL | 0.65 | FAIL | -0.03 |
| 07 | 0.55 | FAIL | 0.98 | PASS | +0.43 대폭 개선 |
| 08 | 1.00 | PASS | 0.92 | PASS | -0.08 |
| 09 | 0.92 | PASS | 0.55 | FAIL | **-0.37 회귀** |
| 10 | 0.85 | PASS | 0.97 | PASS | +0.12 |

평균: **0.71 → 0.77 (+0.06)**. PASS 비율: 4/10 → 5/10.

v2의 핵심 성과는 v1의 주된 실패 원인이던 **하위 개념 과분할**을
크게 줄인 것이다 (case 03·04·07이 대폭 개선). 반면 v2는 **"문서 수
최소화" 편향이 결과 문서 영역까지 번지면서**, 원래 PASS였던
case 01·09가 FAIL로 회귀했다. 회귀의 크기(-0.35, -0.37)는 개선 폭에
근접해, 평균은 약간만 올랐다.

## 2. 결함 (영향 큰 순)

### D1. 결과 문서 분리·생성의 회귀 — 최우선 (회귀 2건의 직접 원인)

**증상**: 두 형태로 나타난다.

- *통합형* (case 01): 명확한 제작 의도가 1개 있어도 결과 문서를 별도
  파일로 만들지 않고 지식 문서 안에 결과 구현 내용을 섞어 넣는다.
  결과적으로 결과 문서 자체가 누락되고, 결과↔지식 참조 매핑과 참조
  이유도 형성되지 않는다.
- *병합형* (case 09): 서로 다른 제작 대상이 여러 개일 때 결과 문서를
  각각 만들지 않고 하나의 통합 결과 문서로 합친다. 결과 문서 수가
  부족해지고, 결과별 참조 매핑이 흐려진다.

**원인**: v2의 "문서 수 최소화"와 "애매하면 합친다"가 지식 문서
분할에는 효과적이었지만, 결과 문서 차원에도 적용됐다. 결과 문서 생성
조건(§3.1)과 지식 문서 분할 편향(§2.1)이 충돌할 때 어디로 기우는지
명확하지 않아, 에이전트가 "합치는 쪽"으로 일관되게 기울었다.

**v3 처방**:
- 결과 문서 분할·생성 기준은 지식 문서의 통합 편향과 **독립**이다.
- 명확한 제작 의도가 1개라도 있으면 결과 문서를 지식 문서와 분리해
  생성한다. 결과 내용을 지식 문서 안에 통합하지 않는다.
- 서로 다른 제작 대상이 N개 식별되면 결과 문서도 N개로 만든다.
  (서로 다른 기능·산출물·사용 흐름 단위로 분리되고 고유 지식 구성이
  다르면 다른 대상으로 본다.)
- 공통 지식은 하나의 지식 문서로 유지하되, 각 결과 문서가 그 공통
  지식을 자기 맥락의 참조 이유로 별도 서술한다.
- "문서 수 최소화"는 지식 문서의 하위 개념 통합 한정 규칙이며, 결과
  문서 병합의 근거로 쓰지 않는다.

### D2. 지식 문서 통합/분리 판단의 양방향 실패

**증상**: 두 방향 모두에서 실패가 나온다.

- *과통합* (case 02): 독립 지식으로 남아야 할 개념을 부모 도구 문서의
  섹션으로 흡수한다. 결과 문서의 참조 매핑에서 지식 하나가 사라진다.
- *과분할* (case 06): 단일 상위 개념으로 묶여야 할 지식을 두 문서로
  쪼갠다. v1의 과분할 결함이 일부 회귀했다.

**원인**: v2의 "부모 개념을 떼면 단독으로 의미가 성립하지 않으면 섹션"
규칙은 통합 방향만 강하다. 어떤 개념이 독립 지식으로 남아야 하는지의
역방향 기준은 약하고, 같은 부모 영역 안에 묶일 개념이 잘못 쪼개지는
경우도 막지 못한다.

**v3 처방**:
- 분할/통합 기준을 **양방향으로** 명문화한다.
  - 독립 문서 조건 (다음 중 하나라도 강하면): 사용자가 그 개념을 별도
    학습 대상으로 반복 질문, 결과 문서에서 독립 선택·구성 요소·구현
    책임으로 사용, 부모 도구의 옵션이 아니라 저장/인증/배포 방식처럼
    독립 결정을 담당, 다른 결과에서도 재사용 가능.
  - 섹션 통합 조건: 부모 개념을 떼면 단독 의미가 없거나, 부모 도구의
    옵션·제약·검증 규칙·하위 동작 양식인 경우.
- 같은 부모 개념 영역에 속하는 서로 다른 측면(예: 비동기 처리 양식 +
  에러 처리 양식)은 별도 문서가 아닌 한 문서의 섹션으로 통합한다.
- 자가 점검에 양방향 질문을 둔다:
  1. "이 섹션은 독립 결정을 담당하는가?" → YES면 분리 후보.
  2. "이 두 문서는 같은 부모 영역의 다른 측면인가?" → YES면 통합 후보.

### D3. 약신호 입력에서 결과 문서 생성·환각 보강 지속

**증상**: 잡담 비중이 높고 제작 의도가 약한 대화에서도 결과 문서를
만든다(case 05). 동시에 대화에서 확립되지 않은 구현 디테일(설정값,
API 호출 흐름)을 보강해 지식 문서를 과하게 상세화한다.

**원인**: v2 §3.1의 게이트("명확한 제작 의도 + 대상 윤곽")가 있지만,
"명확함"의 기준이 모호해서 에이전트가 스치는 기술 키워드를 제작 의도로
과해석한다. 환각 보강 금지(§1)도 추상적이라 SMTP 호스트/포트 같은
관행적 디테일을 "당연한 기본값"으로 추가한다.

**v3 처방**:
- 게이트를 명시적 체크리스트로 바꾼다. 다음이 **모두** 충족되지 않으면
  결과 문서·참조 매핑·참조 이유를 생성하지 않는다.
  - 사용자가 만들 대상의 **이름이나 형태**를 명시했다(스치는 언급이
    아니라 한 번 이상 명확히 지목).
  - 그 대상에 적용할 지식의 **사용 방식**이 대화에서 확인됐다.
  - 단순 호기심·가정·비교만이 아니라 **실제 구현 방향**이 정해졌다.
- 환각 보강 금지를 구체화한다: 대화에서 직접 다루지 않은 설치 명령,
  설정값, API 호출 옵션, 인증 방식 디테일, 모범 사례는 지식 문서에
  쓰지 않는다. 일반 지식이라는 이유로도 끌어오지 않는다.
- 게이트를 통과하지 못한 경우에도 지식 문서는 만들되, 대화에서 명시된
  수준까지만 짧게 유지한다.

### D4. 결과 문서가 있을 때 지식 문서와의 역할 분리 (D1과 짝)

**증상**: 결과 문서 자체는 생성됐지만(case 02), 정답지가 기대하는
복수 지식 문서 중 일부를 결과 문서 또는 다른 지식 문서로 흡수해
참조 매핑이 불완전해진다.

**원인**: D1의 통합 편향이 약한 형태로 작동한다. 결과 문서가 만들어진
뒤에도 "이 지식은 multer 안에 두자"는 식의 흡수가 일어난다.

**v3 처방**:
- 결과 문서가 생성된 경우, 결과 문서의 참조 매핑에 **사용된 모든 지식
  개념**이 별도 지식 문서로 매핑되는지 자가 점검한다.
- 한 지식 문서가 다른 지식 문서의 섹션을 참조 매핑 항목으로 가리키는
  형태는 허용하지 않는다 — 독립 지식이라면 별도 파일로 분리한다.

## 3. v2가 잘한 것 — v3에서 유지할 것

- **하위 개념 과분할 억제**가 크게 개선됐다 (case 03·04·07이 PASS로
  전환된 핵심 이유). 부모 도구의 옵션·제약을 섹션으로 통합하는 능력.
- **탈락한 대안의 처리**: 결정 번복 시 탈락 대안을 결과 문서 참조 매핑
  에서 제외하고 한 줄 요약만 남기는 처리(case 07).
- **잡담 본문 유입 차단**: 거의 모든 케이스에서 ✓.
- **예시 요청 대상 개념에 코드 스니펫 포함**: 일관되게 충족.
- **지식 문서 첫 문단을 코드가 아닌 정의 텍스트로 시작**: 일관 충족.

D1~D4를 고치면서 위 5개를 약화시키지 않는지 확인한다.

## 4. v3 작업 우선순위

1. **D1** — 결과 문서 분할·생성 기준을 지식 문서의 통합 편향과
   분리해 명문화. 회귀 2건(case 01·09)의 직접 원인이며 가장 큰
   레버리지.
2. **D2** — 지식 문서 분할/통합의 양방향 기준 강화. 과통합과 과분할
   양쪽을 동시에 막는 자가 점검 도입.
3. **D3** — 약한 제작 의도 게이트를 명시적 체크리스트로 강화하고,
   환각 보강 금지를 구체화.
4. **D4** — 결과 문서 생성 시 지식 문서 분리의 완전성 자가 점검.

검증은 v3 작성 시 보지 않은 **홀드아웃 케이스**로 한다(같은 케이스로
튜닝+평가 시 과적합).

## 5. v3 핵심 문장 후보

아래 문장들은 v3에 직접 반영할 수 있는 형태의 일반화된 규칙이다.

- "결과 문서 분할·생성 기준은 지식 문서의 '문서 수 최소화' 편향과
  독립이다. 결과 문서를 합치는 근거로 통합 편향을 쓰지 마라."
- "명확한 제작 의도가 1개라도 있으면 결과 문서를 지식 문서와 별도
  파일로 분리해 만든다. 결과 구현 내용을 지식 문서 안에 섞지 마라."
- "서로 다른 제작 대상이 N개 식별되면 결과 문서도 N개로 만든다. 서로
  다른 기능·산출물·사용 흐름이고 고유 지식 구성이 다르면 다른 대상이다."
- "공통 지식은 하나의 지식 문서로 유지하되, 여러 결과 문서가 각자의
  맥락에서 별도 참조 이유와 함께 같은 지식 문서를 참조한다."
- "지식 문서를 분리하기 전에 자문하라: 이 개념이 독립 결정·독립
  구현 책임을 갖는가? 부모 도구의 옵션·제약이라면 섹션으로 통합한다."
- "지식 문서를 통합하기 전에 자문하라: 이 두 측면이 같은 부모 영역의
  부분인가? 다른 결과에서 독립 재사용된다면 별도 문서로 분리한다."
- "결과 문서 생성 게이트: ① 만들 대상의 이름·형태 명시, ② 적용 지식의
  사용 방식 확인, ③ 실제 구현 방향 결정. 셋 모두 충족하지 않으면 결과
  문서·참조 매핑·참조 이유를 만들지 않는다."
- "대화에서 직접 다루지 않은 설치 명령, 설정값, API 호출 옵션, 인증
  방식 디테일, 모범 사례는 어떤 문서에도 추가하지 않는다. 일반 지식
  이라는 이유로도 끌어오지 마라."
- "결과 문서가 만들어졌다면 그 참조 매핑의 각 지식이 별도 지식 문서로
  존재하는지 확인하라. 다른 지식 문서의 섹션을 참조 매핑 항목으로
  가리키는 형태는 허용하지 않는다."
